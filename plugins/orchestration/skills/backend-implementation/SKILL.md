# バックエンド実装スキル

1 つの openspec change の **バックエンド側スライス**（API / Service / Repository / ドメインロジック / データモデル / マイグレーション）を実装するための手法。あなたは 1 change のバックエンド担当として、**所有スコープ内**を縦に実装しきる。

## 前提として読むもの

| 何を | パス | 用途 |
|------|------|------|
| 技術方針 | `docs/ARCHITECTURE.md` | レイヤ構成・データフロー・技術スタック |
| 受け入れ基準 | `docs/e2e_case.md` | 満たすべき振る舞い（API が支える E2E ケース） |
| change 仕様 | 対象 change の `proposal.md` / `design.md` / `tasks.md` / spec delta | 実装すべきバックエンド要件 |
| 既存規約 | リポジトリの既存 Service / Repository / スキーマ | 命名・レイヤ分離・トランザクション作法 |

DB スキーマの設計判断（テーブル / カラム追加・リレーション・インデックス・マイグレーション・並行更新制御）が絡む場合は、`35-architect-db-design` スキルの原則に従う。新しいモジュール・レイヤー・抽象を導入する、あるいは既存構造を再設計する設計判断が絡む場合は、`35-architect-principle` スキルの原則（変更容易性・SOLID・KISS/YAGNI・過剰設計の見極め）に従う。

## 実装の原則

1. **レイヤを守る**: API（入出力・認証認可・検証）→ Service（ドメインロジック・トランザクション境界）→ Repository（永続化）の責務を混ぜない。既存のレイヤ規約に合わせる。
2. **org / テナントスコープを必須にする**: マルチテナント前提のプロジェクトでは、**すべての読み書きに org スコープを効かせる**。スコープ漏れは重大な認可欠陥。横断クエリでも所属境界を必ず適用する。
3. **入力は敵対的に検証する**: すべての外部入力（body / query / params / header）を型・必須・範囲・形式で検証し、不正はドメインに到達させない。認可チェック（誰がこのリソースを操作してよいか）を毎回明示的に行う。詳細は `35-architect-security` スキルに従う。
4. **サイレント失敗を作らない**: 失敗は握りつぶさず、適切なステータス / エラー型で呼び出し側に伝播する。`catch` して握りつぶす、握りつぶして既定値を返す、エラーをログだけして正常応答する、は禁止。
5. **トランザクションと整合性**: 複数テーブルにまたがる更新はトランザクション境界を明示する。並行更新のロストアップデートを想定し、必要なら楽観 / 悲観ロックを使う（`35-architect-db-design` 参照）。
6. **マイグレーションは実行まで**: スキーマ変更は ORM の正当な手段でマイグレーションを作成し、**実行（apply）まで完了**させる。破壊的変更（列削除・型変更）は既存データへの影響を明記する。archive 済み spec を壊す変更が必要になったら、続行せず親へ報告する。
7. **所有スコープを越えない / 並列前提**: 自分の change が縦に所有する API / Service / Repository / スキーマのみ編集する。共有スキーマの追加は土台へ寄せ、波の途中で他スライスを書き換えない。他サブエージェントの編集を巻き戻さず、衝突は自分側で吸収する。

## 手順

1. **整合確認**: proposal / design / tasks / spec delta と `docs/ARCHITECTURE.md` / `e2e_case.md` を突き合わせ、矛盾・欠落を実装前に洗い出す。
2. **データモデル → Repository → Service → API** の順に実装する。データ契約（API の型）はフロント担当と齟齬が出ないよう、spec / design を正とする。
3. **マイグレーション作成・実行**、シード / 既存データ影響の確認。
4. **型・静的解析・ユニット・DB マイグレーション** のうち完了できるものは実装・実行まで完了させる。
5. **自己検証**: 対応する `e2e_case.md` のケースが API 契約上成立するか自己点検する（実際の E2E 実行は E2E 担当。ここでは実行しない）。

## 完了報告に含めること（親エージェントへ返す）

- 変更した全ファイルパス（マイグレーションファイル含む）。
- 追加 / 変更した API エンドポイントと、そのリクエスト / レスポンス型（フロント担当が依存する契約）。
- スキーマ変更とマイグレーションの実行結果。既存データへの影響。
- org スコープ・認可・入力検証をどこで担保したか。
- 型 / lint / unit / migration の結果。未完了タスクとその理由（人間作業のみ残す）。

> あなたは実装に徹する。コードレビューは apply-review、E2E 実行は `40-run-and-report-e2e` skill が別途担う。自分でレビュー OK や E2E 緑を宣言しない。
