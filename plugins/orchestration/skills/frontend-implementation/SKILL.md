# フロントエンド実装スキル

1 つの openspec change の **フロントエンド側スライス**（画面・コンポーネント・フォーム・クライアント状態・クライアントからの API 呼び出し）を実装するための手法。あなたは 1 change のフロント担当として、**所有スコープ内**を縦に実装しきる。

## 前提として読むもの

着手前に必ず突き合わせる。**推測で UI を作らない**。

| 何を | パス | 用途 |
|------|------|------|
| デザインの正本 | `docs/DESIGN.MD` + `docs/design-sample.png` | 色・余白・タイポ・コンポーネント様式・レイアウト |
| 受け入れ基準 | `docs/e2e_case.md` | この画面が満たすべき振る舞い（E2E ケース） |
| change 仕様 | 対象 change の `proposal.md` / `design.md` / `tasks.md` / spec delta | 実装すべき UI 要件 |
| 既存 UI 規約 | リポジトリの既存コンポーネント / デザイントークン | 命名・再利用・スタイル体系 |

## 実装の原則

1. **デザイン忠実度を最優先**: `docs/DESIGN.MD` と `design-sample.png` に対して、色・間隔・角丸・影・タイポ・状態変化（hover / focus / disabled）まで合わせる。独自解釈で崩さない。既存のデザイントークン / コンポーネントを再利用し、車輪の再発明をしない。
2. **状態を全部作る**: 正常系だけでなく **ローディング / 空 / エラー / 権限なし / 部分失敗** を必ず実装する。空状態・エラー表示の欠落は未完了とみなす。
3. **サイレント失敗を作らない**: API エラー・バリデーションエラーは握りつぶさず、ユーザーに見える形（トースト / インラインエラー / 再試行導線）で必ず表出する。`catch` で握って握りつぶすのは禁止。
4. **フォームは境界値まで**: 必須・型・長さ・形式のクライアント検証、送信中の二重送信防止、サーバエラーのフィールド還元まで。バックエンドの検証を UI 側の唯一の防波堤にしない（サーバ側はバックエンド担当が持つ）。
5. **アクセシビリティの最低限**: セマンティックなマークアップ、フォーカス可視、ラベルと入力の関連付け、キーボード操作。
6. **所有スコープを越えない**: 自分の change が縦に所有する画面 / コンポーネント / クライアントモジュールのみ編集する。共有コンポーネントの破壊的変更や、他スライスのファイル書き換えはしない。共有基盤に手を入れる必要が出たら、勝手に変えず親エージェントへ報告する。
7. **並列作業前提**: 他の apply / frontend / backend サブエージェントが同時に走っている。他者の編集を巻き戻さず、衝突は自分側で吸収する。

## 手順

1. **整合確認**: proposal / design / tasks / spec delta と `docs/DESIGN.MD` / `e2e_case.md` を突き合わせ、矛盾・欠落があれば実装前に洗い出す。
2. **実装**: `tasks.md` のフロント該当タスクを順に実装する。データ契約（API のリクエスト / レスポンス型）はバックエンド担当の実装 / 仕様に合わせる。未確定なら仮置きせず親へ確認する。
3. **型・静的解析・ユニット**: 型チェック / lint / コンポーネントの unit テストのうち、完了できるものは実装まで完了させる。
4. **自己検証**: 対応する `e2e_case.md` のケースを読み、その画面遷移・表示がコード上成立しているか自己点検する（実際の E2E 実行は E2E 担当が行う。ここでは実行しない）。

## 完了報告に含めること（親エージェントへ返す）

- 変更した全ファイルパス。
- 実装した画面 / コンポーネントと、カバーした状態（正常 / ローディング / 空 / エラー / 権限なし）。
- デザイン忠実度: `docs/DESIGN.MD` / `design-sample.png` との一致状況と、やむを得ず外した箇所（理由付き）。
- バックエンドとのデータ契約（依存した API とその型）。未確定・要調整の点。
- 型 / lint / unit の結果。未完了タスクとその理由（人間作業のみ残す）。

> あなたは実装に徹する。コードレビューは apply-review、E2E 実行は `40-run-and-report-e2e` skill が別途担う。自分でレビュー OK や E2E 緑を宣言しない。
