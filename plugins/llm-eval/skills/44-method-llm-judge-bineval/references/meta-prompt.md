# 評価用プロンプト（meta-prompt）$M$ と質問スキーマ

Phase 1 の核。 **タスク非依存** で、変えるのは `task-prompt.md`($T$) だけ。以下はそのまま `meta-prompt.ts` に埋め込む本文（埋め込み方は `scaffold-typescript.md` を参照）。

## 設計不変条件（曲げない）

1. **2 ステップ**: Summarize($T \to R$) → Decompose($R \to Q$)。
2. 複数サブタスクを含む要件は **別質問に割る**。
3. 各質問に **違反例(violation example)** を必ず付与。
4. 「はい」=充足 / 「いいえ」=違反 に統一。
5. 質問は **評価次元 $D$** でタグ付け。
6. 出力は **structured outputs でパース可能な JSON**（後述スキーマ）。

## 評価用プロンプト本文（日本語運用・英語併記可）

> システム側に置く（`{{TASK_PROMPT}}` に $T$ を、`{{DIMENSIONS}}` に対象次元を差し込む。次元未指定なら「タスクに適した次元をあなたが定義せよ」とする）。

```
あなたは LLM-as-a-Judge の評価基準を設計する専門家です。あなたの仕事は「出力を採点すること」ではなく、
「出力を評価するための Yes/No 二値質問を設計すること」です。以下を厳密に 2 ステップで行ってください。

# 入力: 評価対象タスク T
{{TASK_PROMPT}}

# 対象の評価次元 D
{{DIMENSIONS}}
（未指定の場合は、このタスクに適した独立性の高い評価次元を 3〜5 個あなたが定義すること。
 典型例: coherence / consistency(事実的一貫性) / fluency / relevance）

## Step 1 — Summarize（要約）
タスク T を、個別に検証可能な「要件」の集合 R = {r_1, ..., r_K} に要約せよ。
- 各要件は 1 つの評価基準を捉える（例: 「重要情報を含む」「フォーマット制約に従う」「事実を捏造しない」）。
- ここではまだ質問化しない。タスク全体の一貫した表現を先に作ること。

## Step 2 — Decompose（分解）
各要件 r_k を 1 つ以上の Yes/No 二値質問に分解せよ。
- 「はい」= 要件を満たしている、「いいえ」= 違反。この向きを全質問で統一する。
- 複数のサブタスクを暗黙に含む要件は、必ず別々の質問に分割する
  （例:「エンティティと数値が正確」→「エンティティは正確か?」と「数値は正しいか?」の 2 問）。
- 各質問には必ず violation_example（その質問が「いいえ」になる具体的なケースの説明）を付ける。
- 各質問に、それが属する評価次元 dimension を付与する。
- 質問は互いにできるだけ独立(低相関)にし、異なる失敗モードを狙う。

# 出力形式（この JSON スキーマに厳密に従う）
{
  "requirements": [ { "id": "r1", "dimension": "...", "text": "..." }, ... ],
  "questions": [
    {
      "id": "q1",
      "dimension": "consistency",
      "requirement_id": "r1",
      "text": "すべてのクレームはソース文で裏付けられているか？",
      "violation_example": "要約が『ペンタゴンが識別のためと述べた』としているが、ソースではロシア国防省の発言である（誤帰属）。"
    }
  ]
}
```

## 質問スキーマ（`questions/<dimension>.json`）

生成された質問は **次元ごとにファイル分割** して保存する。1 質問 = 1 オブジェクト:

```json
{
  "dimension": "consistency",
  "questions": [
    {
      "id": "consistency_q1",
      "requirement_id": "r1",
      "text": "すべてのクレームはソース文で裏付けられているか？",
      "violation_example": "存在しない発言者への帰属、捏造 URL、数値の改変など。"
    }
  ]
}
```

`config` の `dimensions` と一致させる。`generate-questions` は Step2 の質問を dimension でグルーピングしてこの形で書き出す。

## 生成される質問の例（要約タスク・consistency 次元）

評価用プロンプトが出す質問の粒度・向きの参考。「はい=充足」で統一し、それぞれ違反例を付ける:

- すべての記述はソース記事に裏付けられているか？
- ソースと比べて事実誤りが無いか？
- ハルシネーション（ソースに無い捏造）が無いか？
- 固有名詞（人名/組織/地名）はソースどおり正確に表現されているか？
- 数値（日付/統計/数量）はソースと一致するか？
- 因果関係・出来事の順序はソースと矛盾しないか？
- ソース情報の意味を歪めて表現していないか？

> これを他タスク(対話/命令追従/RAG)に流用せず、 **必ず自分のタスク $T$ から評価用プロンプトで生成し直す** こと（質問セットは $T$ 依存）。ただし評価用プロンプト $M$ 自体は不変。日本語運用で構わないが「はい=充足」の向きと違反例を保つ。
