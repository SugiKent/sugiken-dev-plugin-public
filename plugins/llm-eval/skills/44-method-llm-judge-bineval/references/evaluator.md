# Phase 2: 二値評価とスコアリングの実装仕様

## 評価関数 $f_E(x, y, q_i)$

判定役の LLM（judge）は **検知した既存 LLM を `eval/llm` アダプタ経由** で呼ぶ（`detect-and-wire-llm.md`）。入力 $x$・出力 $y$・単一質問 $q_i$（違反例つき）を与え、 **独立に** Yes/No + 説明を返させる。

- **1 質問 = 1 呼び出し** を基本とする（独立性を担保。分散削減の前提）。一括処理（batching）する場合も各質問の判定は他質問に依存させない指示にする。
- **構造化出力で強制**（アダプタが JSON Schema を渡す。パースの揺れを排除）:

```json
{ "answer": "yes" | "no", "explanation": "この判定の根拠(日本語可)" }
```

- `answer` を $f_E \in \{0,1\}$ に写像: `yes → 1`, `no → 0`。
- `explanation` = $e_i$ を必ず保存（デバッグ・最適化の教訓抽出に使う）。

### 判定役へのシステムプロンプト（雛形）

```
あなたは厳密な二値評価者です。与えられた入力・出力・1 つの Yes/No 質問に対し、
その質問だけを独立に判定します。他の観点は考慮しません。

質問: {{QUESTION_TEXT}}
違反例（この場合 "no" になる）: {{VIOLATION_EXAMPLE}}

# 入力 x
{{INPUT}}

# 評価対象の出力 y
{{OUTPUT}}

判定: この質問に対し出力は要件を満たすか。満たすなら "yes"、違反があれば "no"。
必ず理由(explanation)を添えること。曖昧な場合は違反例と照合して判断する。
```

`{{VIOLATION_EXAMPLE}}` を必ず注入する（違反ケースを明確化するのが手法の要）。

## スコアリング

- 次元スコア: $S_d = \frac{1}{|Q_d|}\sum_{q_i \in Q_d} f_E$
- 全体スコア: $S = \frac{1}{N}\sum_{i=1}^{N} f_E$
- アフィン変換: $S' = S\cdot(b-a)+a$。`config` の `scale=[a,b]` で指定。1–5 なら `[1,5]` → $S'=S\cdot4+1$。
- 出力は **次元別スコア + 全体スコア + 各質問の {answer, explanation}** を必ず含む構造で返す（監査・デバッグのため）。

### 出力レポート形式（`evaluate` の返り値）

```json
{
  "overall": { "raw": 0.4286, "scaled": 2.71 },
  "dimensions": {
    "consistency": { "raw": 0.4286, "scaled": 2.71, "yes": 3, "total": 7 }
  },
  "answers": [
    { "id": "consistency_q1", "dimension": "consistency", "answer": "no",
      "explanation": "ペンタゴンの発言を誤帰属している。" }
  ]
}
```

## 決定性（temperature の扱い）

- ランダム性を抑えるため、対応モデルでは `temperature: 0`。撤廃されているモデルでは省略する。 **この分岐はアダプタ(`eval/llm`)に持たせる**（BINEVAL 本体は温度を意識しない）。
- 厳密比較が要る用途では `runs: 2` を config で指定し、$f_E$ を 2 回取り「はい」率を平均する。CI / プロダクトでは `runs: 1` で可。

## 実装上の注意

- 質問数 × ケース数の呼び出しになるためコストが増える。並列度(`concurrency`)と一括処理、次元/質問の絞り込みで調整する。
- 構造化出力が使えないモデルの場合のみ、アダプタ内で「JSON のみ返す」指示 + 厳密パース(`JSON.parse`) + 失敗時 1 回再試行にする（生文字列マッチはしない）。
