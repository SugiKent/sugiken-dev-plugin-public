# ゴールデンデータセット構築（Silver→Gold）— LLM 合成 + 人間レビュー

評価の「正解」を決められるのは人間だけである。このスキルは **人間検証済みの確定した正解データ（gold）
（唯一の正解集）** を、LLM の合成と人間のレビューで半自動的に固めていく手順を敷設する。
**どう採点するか（評価手法）には依存しない** — ここで作る確定正解データは コードによる照合（code-based）でも
判定役の LLM（judge）でも同じように「正解の物差し」として読まれる。

## このスキルの位置付け（先に読む）

- 前提: `40-eval-directory-setup` で `evals/datasets/` 構造・`splits.json`・`config.json` の
  刻印が用意されていること（確定正解データの住所と冪等性の器）。
- このスキルの本体は **人間との対話**。AI は候補ケースと論点を整えて意思決定を高速化する役に徹する。
  口頭合意は将来の実行（run）で再現できないので、決定は必ず `POLICY.md` と各ケース `expected` に**文書化**する。
- ここで確定した確定正解データと POLICY を、手法スキル（`43-method-code-based-scoring` /
  `44-method-llm-judge-bineval` / `45-method-gepa-optimization`）が共通の正として消費する。

## なぜ確定正解データが唯一の物差しか（中心思想）

- 「高いモデル A と安いモデル B をペアワイズ比較して B が負けなければ採用」は一見合理的だが、
  **A も間違える**。A を基準にすると (1) A の誤りを正解として模倣する、(2) 品質の天井が
  A に固定される。ユーザーがペアワイズを提案しても、この 2 点を説明して確定正解データ方式を薦める。
- 確定正解データ方式では他モデルは「判定基準」ではなく「確定正解データと同じ土俵で測られるもう 1 つの候補」に
  なる。判定役には `expected`（人間検証済み正解）を参照（reference）として渡す（正解参照方式の判定（reference-based judging））。

## Silver→Gold の手順

1. **ベースライン（現行モデルの実測）実走**: 現行モデルで全評価対象（target）を 1 回実行（eval が DB 等の共有状態（state）を使うなら
   **絶対に直列・1 プロセスのみ**）。実走にはモデルの差し替え口（seam, `42-eval-injection-seam`）を使うが、
   ベースライン計測は現行モデルのままでよい。
2. **失敗分類マップ（gap-map）作成**（`evals/runs/<ts>/gap-map.md`）: fail / 低 score ケースの理由と actual を読み、
   **失敗を原因で分類**する。分類が「どのケースを足すべきか」「後で何を最適化するか」を決める。
   分類語彙の例: 「submit tool 未呼び出し（正しい結果を text に吐いて終わる = tool 呼び出しの作法（tool-calling
   discipline））」「status 誤選択」「判別キー（discriminator）名不遵守」「存在しないデータの捏造」
   「引数総当たりの冗長ループ」。
3. **下書きケース（silver, レビュー前）の起案**: 弱点（gap）を突くケースを LLM が YAML で起案。既存ケースは変更しない。
   必ず `tags: ["silver", <normal|edge|adversarial>, ...]` を付ける（人間レビュー前の印）。
   - 配分: 各評価対象 10〜13 件、エッジケース + 敵対的（adversarial）を 3 割以上
   - eval seed（固定の家族構成・既存エンティティ（entity）等）に存在する前提だけで書く
   - 実行基盤（ハーネス）の制約（単一ターンのみ等）で原理的に到達不能なケースは**作らない**
     （失敗分類マップに「要ハーネス拡張」と記録するだけにする）
4. **SILVER_REVIEW.md**: 新規ケース全件を「case id / 種別 / 狙い / expected 要点 /
   **人間が確認すべき論点**」の表で提示。ベースラインでの実挙動が分かっていれば
   「expected 確定の材料」として添える。**推奨案を併記**し、人間が「コメントを書いていない
   箇所は問題なし」で済む形にする。
5. **人間レビュー → POLICY.md へ昇格（最重要）**: 人間の回答をケース個別の修正で終わらせず、
   **一般化した判断基準**として `evals/datasets/POLICY.md` に書く。以後のケース追加・
   評価手法の判定基準（criteria）・最適化はすべて POLICY が正。
   - 例:「出張は calendar に」→「ワンタイムのイベントごとは calendar。人に紐づく定常的・
     不変的な情報（好み・アレルギー・誕生日）は memory」
   - 例:「『午後』は聞き直す」→「開始時刻が特定できない指定は default で埋めず聞き直す。
     時刻明示時の終了時刻は reasonable default 可」
   - **既存の確定正解データも前提を疑う**: レビューで方針が変われば既存ケースの intent も追従させる
6. **確定正解データへの昇格**: 決定を expected に反映 → silver タグ除去 → `splits.json` に登録（学習用データ（train）/検証用データ（holdout）の
   層化固定は 40 参照）→ 整合の機械検証（load / 相互網羅 / silver 残存なし）。

## expected の書き方（昇格基準）

1. `expected` が「唯一の正解」として妥当か。別解が同等に正しいなら判定を緩める
   （コード照合側の許容集合を広げる、または判定役の判定基準に明記する — 具体語彙は各手法スキル）。
2. ユーザーが指定していない要素で減点しない（POLICY の default 規則に従う）。
3. エッジケース / 敵対的 は「やってはいけないこと」を明記する（例: `mustNotCall` / hardFail）。
4. 確定正解データは「あるべき姿」でよい — 現行モデルが達成できないケースも確定正解データにする
   （それが改善対象。ただし**実行基盤的に到達不能**なケースとは区別する）。

> `expected` に書く具体的なフィールド語彙（`acceptableFirstTools` / `mustNotCall` /
> `expectedType` など決定的な軸のもの、および判定役の判定基準の書き方）は、その軸を採点する
> **評価手法スキル側**が定義する。確定正解データ構築時は「何が正解か」を確定することに集中し、
> 「どう照合するか」は手法に委ねる。

## 実行前提の注入（前提データ（fixture）が状態（state）を必要とする場合）

「中身の入ったページ」「既存の予定」のような **case 固有の前提状態** は、cleanup が毎回
状態を消す設計なら **payload に宣言して runCase 冒頭で注入する**（例: `payload.seedPages`）。
グローバル seed に足すと他ケースの前提（「〜は存在しない」系）を壊すので、**case 単位で
個別に有効化（opt-in）** する。注入機構は「書いた後に読み戻して一致するか」の書き戻し検証（round-trip）を必ず付ける。

## ベースライン計測（確定正解データ確定後）

確定正解データが固まったら、候補モデルごとに同一の確定正解データで絶対評価し（モデル差し替えは `42-eval-injection-seam`）、
評価対象/case 別の pass 率・avg score を並べる。これが「どのモデルがどの機能でどれだけ確定正解データに
届いていないか」の地図になり、最適化（`45-method-gepa-optimization`）の投資先を確定する。

## やってはいけないこと

- 別モデルの出力を正解（ベースライン）として判定役に渡す（確定正解データ方式の放棄）
- 人間レビューの決定を口頭合意のまま進める（POLICY.md と expected に必ず文書化）
- 人間の回答をケース個別の修正で終わらせる（一般化して POLICY に昇格させる）
- 実行基盤的に到達不能なケースを確定正解データにする（「あるべき姿の未達」とは区別する）
- silver タグの残存（確定正解データへの昇格時に必ず除去し機械検証する）
