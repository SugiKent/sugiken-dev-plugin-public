# ゴールデンデータセット構築（Silver→Gold）と失敗分類マップ（gap-map）

## なぜ確定した正解データ（gold）が唯一の物差しか

- 「高いモデル A と安いモデル B をペアワイズ比較して B が負けなければ採用」は一見合理的だが、
  **A も間違える**。A を基準にすると (1) A の誤りを正解として模倣する、(2) 品質の天井が
  A に固定される。ユーザーがペアワイズを提案しても、この 2 点を説明して確定正解データ方式を薦める。
- 確定正解データ方式では他モデルは「判定基準」ではなく「確定正解データと同じ土俵で測られるもう 1 つの候補」に
  なる。判定役の LLM（judge）には `expected`（人間検証済み正解）を参照（reference）として渡す（正解参照方式の判定（reference-based judging））。

## 判定方式の分離（コードによる照合（code-based）か判定役 LLM か）

| 軸の性質 | 例 | 判定 |
|---|---|---|
| 正解が一意（決定的） | 呼ぶべき tool/agent 名、出力 type、status、ISO 日時 | **コード照合**（構造チェック。判定役より強く・安く・ブレない） |
| 曖昧（品質の程度） | 自然文の質、抽出内容の妥当性、draft の充実度 | 判定役 vs expected |

構造的な不合格のケースは判定役をスキップしてコストを節約する。
コード照合の expected 語彙（実績のあるパターン）:

- `acceptableFirstTools: [...]` — 最初の委譲先の許容集合（別解が同等に正しければ複数入れる）
- `mustNotCall: [...]` — 全期間で 1 度も呼ばれてはならない tool（敵対的（adversarial）の主武器）
- `expectedType` / `expectedStatus` — 出力の判別キー（discriminator）厳密一致
- `expectNoSuggestion: true` — 「出さないことが正解」ケース（捏造検知）

## Silver→Gold の手順

1. **ベースライン（現行モデルの実測）実走**: 現行モデルで全評価対象（target）を 1 回実行（eval が DB 等の共有状態（state）を使うなら
   **絶対に直列・1 プロセスのみ**）。
2. **失敗分類マップ（gap-map）作成**（`evals/runs/<ts>/gap-map.md`）: fail / 低 score ケースの判定役の判断理由と
   actual を読み、**失敗を原因で分類**する。分類が最適化の投資先を決める。
   実例の分類語彙: 「submit tool 未呼び出し（正しい結果を text に吐いて終わる = tool 呼び出しの作法（tool-calling
   discipline））」「status 誤選択」「判別キー（discriminator）名不遵守」「存在しないデータの捏造」
   「引数総当たりの冗長ループ」。
3. **下書きケース（silver, レビュー前）の起案**: 弱点（gap）を突くケースを LLM が YAML で起案。既存ケースは変更しない。
   必ず `tags: ["silver", <normal|edge|adversarial>, ...]` を付ける（人間レビュー前の印）。
   - 配分: 各評価対象 10〜13 件、エッジケース + 敵対的（adversarial）を 3 割以上
   - eval seed（固定の家族構成・既存エンティティ（entity）等）に存在する前提だけで書く
   - 実行基盤（ハーネス）の制約（単一ターンのみ等）で原理的に到達不能な status のケースは**作らない**
     （失敗分類マップに「要ハーネス拡張」と記録するだけにする）
4. **SILVER_REVIEW.md**: 新規ケース全件を「case id / 種別 / 狙い / expected 要点 /
   **人間が確認すべき論点**」の表で提示。ベースラインでの実挙動が分かっていれば
   「expected 確定の材料」として添える。**推奨案を併記**し、人間が「コメントを書いていない
   箇所は問題なし」で済む形にする。
5. **人間レビュー → POLICY.md へ昇格**: 人間の回答をケース個別の修正で終わらせず、
   **一般化した判断基準**として `POLICY.md` に書く。以後のケース追加・prompt 最適化・
   判定役の判定基準（criteria）はすべて POLICY が正。
   - 例: 「出張は calendar に」→「ワンタイムのイベントごとは calendar。人に紐づく定常的・
     不変的な情報（好み・アレルギー・誕生日）は memory」
   - 例: 「『午後』は聞き直す」→「開始時刻が特定できない指定は default で埋めず聞き直す。
     時刻明示時の終了時刻は reasonable default 可」
   - **既存の確定正解データも前提を疑う**: レビューで方針が変われば既存ケースの intent も追従させる
6. **確定正解データへの昇格**: 決定を expected に反映 → silver タグ除去 → splits.json に登録 →
   整合の機械検証（load / 相互網羅 / silver 残存なし）。

## expected の書き方（昇格基準）

1. `expected` が「唯一の正解」として妥当か。別解が同等に正しいならコード照合側を緩める
   （許容集合を広げる）か判定役の判定基準に明記する。
2. ユーザーが指定していない要素で減点しない（POLICY の default 規則に従う）。
3. エッジケース / 敵対的 は「やってはいけないこと」を `mustNotCall` / hardFail に明記する。
4. 確定正解データは「あるべき姿」でよい — 現行モデルが達成できないケースも確定正解データにする
   （それが改善対象。ただし**実行基盤的に到達不能**なケースとは区別する）。

## 実行前提の注入（前提データ（fixture）が状態（state）を必要とする場合）

「中身の入ったページ」「既存の予定」のような **case 固有の前提状態** は、cleanup が毎回
状態を消す設計なら **payload に宣言して runCase 冒頭で注入する**（例: `payload.seedPages`）。
グローバル seed に足すと他ケースの前提（「〜は存在しない」系）を壊すので、**case 単位で
個別に有効化（opt-in）** する。注入機構は「書いた後に読み戻して一致するか」の書き戻し検証（round-trip）を必ず付ける。

## Phase 2: ベースライン計測

確定正解データ確定後、候補モデルごとに同一の確定正解データで絶対評価し（`EVAL_TARGET_MODEL=<m>`）、
評価対象/case 別の pass 率・avg score を並べる。これが「どのモデルがどの機能でどれだけ確定正解データに
届いていないか」の地図になり、Phase 3 の最適化対象（コンポーネント × 失敗分類）を確定する。
