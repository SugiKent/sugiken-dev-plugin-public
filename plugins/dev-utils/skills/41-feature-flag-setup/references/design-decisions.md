# 設計判断（なぜこの 6 方針か）

このスキルの価値は「コード」ではなく「判断」にある。 各方針の根拠・代替案・トレードオフを残す。 スタックが違ってもここは流用できる。

---

## 決定1: DB が正本、 値は String 一本、 解釈は参照側

`FeatureFlag.value: String`。 型カラムを持たない。 boolean は `value === "true"`、 数値は `Number(value)` 等で**参照側がパース**する。

- **なぜ**: スキーマと管理画面が単純になる。 「とにかくデプロイなしで切り替えたい」という MVP 要件にはテキスト一本で十分。
- **トレードオフ**: 業界的には型付き（boolean/number/json 列）が推奨で、 キャストの安全性は参照側の規律に依存する。 → boolean は **`"true"` のときだけ true** と明示比較し、 **未知値・未定義は安全側（false）へ倒す**。 解釈を参照ヘルパ 1 箇所に集約して事故を防ぐ。
- **拡張**: 型付き値が要るほど育ったら `references/stack-variants.md` の「型付き値への拡張」を参照。 最初からはやらない（Non-Goal）。

## 決定2: CSV 冪等 seed を migrate deploy の後段に置く

`feature_flag/feature_flag.csv`（`name,value[,description]`）を読み、 `INSERT ... ON CONFLICT (name) DO NOTHING` 相当で投入。

- **なぜ**: Prisma migration は CSV を読めない（純 SQL のみ）。 「新規フラグだけ投入・既存値は不変更」は `ON CONFLICT DO NOTHING` でそのまま表現でき、 **何度流しても安全（冪等）**。 並行実行でも一意制約で破綻しない。
- **なぜ upsert ではないか**: `upsert`（既存も更新）にすると、 運用で管理画面から変えた値を seed が次のデプロイで踏み潰す。 seed の役割は「新フラグの初期値を入れる」であって「値を管理する」ではない。 値の正本はあくまで DB（の運用変更後の状態）。
- **代替案**: フラグ追加ごとに手書き SQL migration ＝ 追加のたびに migration 作成の手間。 CSV 一元管理の方が運用が軽い。
- **キーの扱い**: house 規約に合わせ `id`（cuid）+ `name`(unique) とし、 seed と更新は自然キー `name` で操作（挿入のみ / name 指定 update）。

## 決定3: サーバはシングルトン + 遅延（lazy）TTL 更新 ← 最重要

モジュールスコープにフラグマップと最終ロード時刻を保持。 参照時に `now - lastLoaded > TTL` なら DB から再取得して差し替える。

- **なぜ `setInterval` を使わないか**: サーバレス（Lambda 等）は**マルチインスタンス**で、 各プロセスの `globalThis` シングルトンはインスタンス間で共有されない。 さらに処理の合間に**フリーズ**されるため、 `setInterval` の定期ポーリングは発火が信頼できない。 → 「リクエストが来た契機で TTL を見て、 古ければ取り直す」遅延更新が唯一安定して動く形。 AWS AppConfig の Lambda 拡張も「warm インスタンスがメモリキャッシュ + アクセス時 TTL 判定」というまさにこの形。
- **本番の整合性**: 各インスタンスが独立キャッシュを持つので、 更新の反映は **TTL（約60s）範囲の結果整合**。 即時全インスタンス反映は本機構の対象外（必要なら共有キャッシュ＝決定の「代替案」へ）。
- **dev（単一プロセス）**: TTL=0（参照が常に最新）+ 書き込み成功時にシングルトンを直接 `reload()`。 「変更後すぐ反映」要件を単一プロセスの利点で素直に満たす。
- **代替案**: Redis 等の共有キャッシュで即時整合 ＝ 個人開発には過剰（Redis 不使用前提）。 毎リクエスト DB 読み ＝ コスト高・シングルトン要件に反する。
- **常駐サーバ（非サーバレス）の場合**: `setInterval` も使えるが、 遅延 TTL は常駐でもそのまま正しく動く（むしろ単純）。 どちらでも遅延 TTL でよい。 詳細 `references/stack-variants.md`。

## 決定4: フロント受け渡しは外部 bootstrap スクリプト `/api/feature-flags.js` ← 重要

サーバが `window.__FEATURE_FLAGS__ = {...}` を代入する JavaScript を生成し、 `index.html` の `<head>` から render-blocking で読む。 クライアントは起動時に同期で `window` を読み、 無ければ既定値へフォールバック。

- **なぜ**: `index.html` を **静的配信のまま**（S3 等）にでき、 `/api/*` は既存のサーバ経路を通るので **CDN/CloudFront の behavior 変更が不要**。 最小の構成変更でブリンクを回避できる。
- **なぜブリンクしないか**: render-blocking スクリプトは React バンドルより前に同期実行されるので、 React の初回描画時点で `window.__FEATURE_FLAGS__` が確定済み。 「フラグ取得を待つ間だけ別 UI が見える → 切り替わる」というちらつきが構造的に起きない。
- **代替案 (a)**: HTML に JSON をインライン埋め込み（サーバ経由で `/` を返す）＝ 静的配信なら HTML 生成経路が要り CDN behavior 変更で工事が大きい。 さらに **`</script>` ブレイクアウト XSS** のリスク。
- **代替案 (b)**: API 取得 + クライアントキャッシュ ＝ 初回ブリンクが残る（要件と逆）。
- **セキュリティ**: 独立した JS ファイルとして返すため `</script>` ブレイクアウトの懸念がない。 値は `JSON.stringify` で JS リテラルとして安全に出力する。
- **トレードオフ**: render-blocking で初期表示に 1 往復のネットワークが乗る（PageSpeed が一般に非推奨とする形）。 フラグは小さく単一リクエストなので内部ツールには許容。 任意で CDN に `/api/feature-flags.js` を短時間（約60s）キャッシュさせ、 Lambda 呼び出しと往復コストを抑制できる（決定3 の TTL と自然に一致）。

## 決定5: 編集は admin role 限定ゲート

更新手続きに `requireRole("system_admin")` 相当を適用。 一覧取得（管理画面表示）も同レベルに限定して始める。

- **なぜ**: 既存の認可ゲートで実現でき、 新規の権限機構が要らない。
- **広げ方**: 読み取りを admin 以上に開きたい等は後追いで緩める。 まずは最も狭く始める。

## 決定6: 手動 migration + migrate deploy

`migrate dev` は非対話環境（CI/エージェント）でデータ削除 warning 時に対話を要求して失敗する。 手動 SQL migration を作って `migrate deploy` で適用するのが定石。

- **なぜ低リスク**: 本機構は新規テーブル追加のみで rename を伴わない（rename はデータ消失リスク）。 適用前後に `migrate diff` で SQL 確認と drift 検証。
- プロジェクトに Prisma 運用規約（`.claude/rules/prisma-schema.md` 等）があれば必ず従う。

---

## リスク要約

| リスク | 緩和 |
|---|---|
| render-blocking で初期表示が 1 往復遅延 | フラグは小さく単一リクエスト。 任意で CDN 短時間キャッシュ |
| マルチインスタンスで更新が即時に全反映されない | TTL（約60s）の結果整合として仕様化。 即時性が要る要件は対象外 |
| テキスト値のキャスト事故 | boolean は `=== "true"` 明示比較、 未知・未定義は安全側（false）。 参照ヘルパに集約 |
| seed の二重/並行実行 | `ON CONFLICT DO NOTHING` で冪等。 一意制約で破綻しない |
| migration によるデータ消失 | 手動 migration + `migrate deploy`、 新規テーブルのみ、 diff で検証 |
| フラグ提供前のクライアント参照 | `window.__FEATURE_FLAGS__` 未定義時は既定値フォールバックで描画を止めない |
