# ユニットテスト観点チェックリスト

「振る舞いの記述」ではなく「意図・仕様」を検証する。 失敗時に何が壊れたか分かるように書く。

## 1. CSV 冪等 seed（決定2）
- [ ] 空テーブルに seed → CSV 行が全て投入される。
- [ ] **既存 name に異なる value がある状態で seed → 既存値が変わらない**（`DO NOTHING` の核心）。
- [ ] 同じ seed を 2 回流す → 2 回目で何も変わらない（冪等）。

## 2. シングルトン 遅延 TTL（決定3）
- [ ] 初回参照で DB ロードされる。
- [ ] TTL 内の再参照は DB を叩かない（キャッシュヒット）。 ← repository をモックして呼び出し回数で検証。
- [ ] **TTL 超過後の参照で再取得される**。 ← `Date.now` をモックして時間を進める。
- [ ] dev（TTL=0）は毎参照で最新を返す。
- [ ] `reloadFlags()` で即時に最新へ差し替わる。

## 3. 値解釈ヘルパ（決定1）
- [ ] `"true"` → `true`、 それ以外（`"false"`/`"1"`/`"yes"`/空）→ `false`。
- [ ] **未定義（フラグ無し）→ 渡した fallback**。
- [ ] number: 数値文字列 → 数、 不正・未定義 → fallback。

## 4. role ゲート（決定5）
- [ ] admin（system_admin）→ 更新できる。
- [ ] admin 未満（manager/user 等）→ 更新が拒否される。
- [ ] 一覧取得も同様にゲートされる（本 change の方針）。

## 5. bootstrap 出力（決定4）
- [ ] レスポンスが `window.__FEATURE_FLAGS__ = {...};` 形式。
- [ ] `Content-Type: application/javascript`。
- [ ] 値が `JSON.stringify` で安全に出力される（`</script>` を含む値でもブレイクしない／文字列としてエスケープされる）。
- [ ] シングルトンの現在値が反映される。

## 6. クライアント フォールバック（決定4/1）
- [ ] `window.__FEATURE_FLAGS__` 欠落時 → 既定値（安全側、 例 `false`）。
- [ ] 値あり時 → その値を解釈して返す。
