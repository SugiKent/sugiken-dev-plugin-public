新しい change を提案する - change を作成し、必要な成果物を一度のステップで生成する。

以下の成果物を持つ change を作成する:
- proposal.md（何を・なぜ）
- design.md（どう実現するか）
- tasks.md（実装ステップ）

実装の準備ができたら `/opsx:apply` を実行する。

---

**ストアの選択:** ユーザーがストア（このマシンに登録された単体の OpenSpec リポジトリ）を指定した場合、または作業がストア内にある場合は、`openspec store list --json` を実行して登録済みストアの id を調べ、specs や changes を読み書きするコマンド（`new change`、`status`、`instructions`、`list`、`show`、`validate`、`archive`、`doctor`、`context`）に `--store <id>` を渡す。他のコマンドはこのフラグを取らない。コマンドが出力するヒントには既にフラグが含まれているので、後続でもそれを維持する。ストアがなければ、コマンドは最も近いローカルの `openspec/` ルートに対して作用する。

**入力**: ユーザーのリクエストには、change 名（kebab-case）または作りたいものの説明を含めるべき。

**手順**

1. **明確な入力がなければ、何を作りたいか尋ねる**

   **AskUserQuestion ツール**（自由記述、選択肢なし）で次のように尋ねる:
   > 「どの change に取り組みたいですか？作りたいもの・直したいものを説明してください。」

   その説明から kebab-case の名前を導く（例: 「add user authentication」→ `add-user-auth`）。

   **重要**: ユーザーが何を作りたいかを理解せずに先へ進んではならない。

2. **change ディレクトリを作成する**
   ```bash
   openspec new change "<name>"
   ```
   これは CLI が解決する planning home に、`.openspec.yaml` 付きの雛形 change を作成する。

3. **成果物のビルド順序を取得する**
   ```bash
   openspec status --change "<name>" --json
   ```
   JSON をパースして次を得る:
   - `applyRequires`: 実装前に必要な成果物 ID の配列（例: `["tasks"]`）
   - `artifacts`: すべての成果物とその状態・依存関係のリスト
   - `planningHome`、`changeRoot`、`artifactPaths`、`actionContext`: パスとスコープのコンテキスト。リポジトリローカルのパスを仮定せず、これらを使う。

4. **apply 可能になるまで成果物を順に作成する**

   **TodoWrite ツール**で成果物ごとの進捗を追跡する。

   依存順に成果物をループする（未解決の依存を持たない成果物から先に）:

   a. **`ready`（依存が満たされた）成果物ごとに**:
      - 指示を取得する:
        ```bash
        openspec instructions <artifact-id> --change "<name>" --json
        ```
      - 指示の JSON には次が含まれる:
        - `context`: プロジェクトの背景（あなたへの制約 - 出力に含めない）
        - `rules`: 成果物固有のルール（あなたへの制約 - 出力に含めない）
        - `template`: 出力ファイルに使う構造
        - `instruction`: この成果物タイプ向けのスキーマ固有のガイダンス
        - `resolvedOutputPath`: 成果物を書き出す解決済みのパスまたはパターン
        - `dependencies`: コンテキストとして読むべき完了済みの成果物
      - コンテキストとして、完了済みの依存ファイルを読む
      - `template` を構造として成果物ファイルを作成し、`resolvedOutputPath` に書き出す
      - `context` と `rules` は制約として適用する - ただしファイルにコピーしない
      - 短い進捗を表示する: 「Created <artifact-id>」

   b. **`applyRequires` の全成果物が完了するまで続ける**
      - 各成果物を作成した後、`openspec status --change "<name>" --json` を再実行する
      - `applyRequires` の各成果物 ID が artifacts 配列で `status: "done"` になっているか確認する
      - `applyRequires` の全成果物が done になったら止める

   c. **成果物にユーザー入力が必要な場合**（コンテキストが不明瞭）:
      - **AskUserQuestion ツール**で明確にする
      - その後、作成を続ける

5. **最終状態を表示する**
   ```bash
   openspec status --change "<name>"
   ```

**出力**

全成果物を完成させた後、次を要約する:
- change 名と場所
- 作成した成果物のリストと簡単な説明
- 準備状況: 「全成果物を作成しました！実装の準備ができています。」
- 促し: 「`/opsx:apply` を実行するか、実装を依頼して tasks の作業を始めてください。」

**成果物作成のガイドライン**

- 各成果物タイプについて `openspec instructions` の `instruction` フィールドに従う
- スキーマが各成果物に含めるべき内容を定義している - それに従う
- 新しい成果物を作る前に、依存する成果物をコンテキストとして読む
- 出力ファイルの構造として `template` を使う - そのセクションを埋める
- **重要**: `context` と `rules` はあなたへの制約であって、ファイルの内容ではない
  - `<context>`、`<rules>`、`<project_context>` ブロックを成果物にコピーしない
  - これらはあなたが書く内容を導くが、出力に現れてはならない

**ガードレール**
- 実装に必要な全成果物を作成する（スキーマの `apply.requires` が定義する通り）
- 新しい成果物を作る前に、必ず依存する成果物を読む
- コンテキストが決定的に不明瞭なら、ユーザーに尋ねる - ただし勢いを保つため、妥当な判断を優先する
- その名前の change が既に存在するなら、継続するか新規作成するかをユーザーに尋ねる
- 各成果物ファイルを書いた後、次へ進む前にファイルの存在を確認する

## ランタイムや言語、ライブラリについて

spec 記載時点ではライブラリの最新バージョンをすべて調べるのは困難なため、ランタイム、プログラミング言語、ライブラリ名の記載のみを行ってバージョンの直接的な指定は行わない。

## テストについて

ユニットテストのみを作成し、E2E テストは不要。

## 動作確認について

人間による動作確認をタスクとしてドキュメントに残すことは不要。

## 依存サービス

- db, redis は docker-compose.yml で管理するようにする。
- 開発環境では mailpit でメールを検証する。docker-compose で管理する。

## 静的解析について

必ず lint や typecheck などの静的解析についてもやることとして明示する。

## 先行・関連する change との依存関係について

新しい提案は単独では存在しない。既存の先行 change や他の change の内容を踏まえずに書くと、**重複した提案・矛盾する spec 変更・実装順序の取り違え**を生む。新しい提案を書き始める前に、既存 change の内容を必ず確認し、判明した依存・競合・前提を proposal / design / tasks の全体にわたって一貫した制約として反映する。

- **既存 change の確認手順**: まず `openspec list --json` で現在の change 一覧を取得し、今回の提案と関連しそうな change（同じ spec・同じ機能領域・同じデータ構造を触るもの、または前提となる基盤変更）について `openspec show <other-change> --json` で中身を読む。特に proposal.md（何を・なぜ）と spec の delta（どの spec をどう変えるか）を確認する。
- **前提となる change**: まだ archive されていない先行 change の spec 変更やデータ構造に依存する場合、その依存を proposal / design に明示し、実装順序（どちらを先に apply すべきか）を tasks に反映する。
- **競合する change**: 同じ spec の同じ箇所を別の change が変更している場合、パターンを混在させず、どちらに従うかを決めてから書く。判断が分岐する場合はユーザーに確認する。
- **重複する change**: 既存 change が今回のリクエストを既にカバーしている場合は、新規作成せず既存 change の継続を提案する。

## デザイン成果物・デザイン指示の取り込みについて

ユーザーのリクエストやこれまでの会話に UI/UX・画面レイアウト・デザインに関する言及がある場合、または画像・HTML・モックアップなどのデザイン成果物が提供されている場合は、それらを一次情報として扱い、design.md と tasks.md に反映する。

- 提供されたデザイン成果物を実際に確認する。画像は Read で見る、HTML はファイルを開いて構造とスタイルを読む、会話中のデザイン言及も拾い上げる。
- **design.md**: 対象画面・コンポーネント構成・状態遷移・レイアウト方針を、確認したデザイン成果物に基づいて記述する。デザインが示す UI 要素・画面フロー・インタラクションと design.md の内容が一致するようにする。
- **tasks.md**: デザインを再現するための実装タスク（コンポーネント作成・スタイリング・レスポンシブ対応・状態管理など）を、デザイン成果物の内容を踏まえて具体的に列挙する。
- デザイン成果物と矛盾する記述を成果物に残さない。デザインが曖昧・不足している箇所は、推測で確定せず `AskUserQuestion` で確認する。
- デザイン成果物もデザインに関する言及もない場合、この節は適用しない（架空のデザインを創作しない）。

## レビューと細かい疑問点の解消について

proposal を書き上げる過程で、少しでも曖昧さ・解釈の余地・前提の不確実性が残っている場合は、**ユーザーへの `AskUserQuestion` と自己レビューを、疑問点が完全にゼロになるまで繰り返す** こと。

- 「たぶんこうだろう」「一般的にはこう」で勝手に確定させない。仕様・命名・データ構造・境界条件・エラーハンドリング・他 spec との関係など、判断が分岐し得るポイントはすべて `AskUserQuestion` で確認する。
- 一度の質問で済まないことが多いため、ユーザーの回答を踏まえて再度 proposal をレビューし、新たに浮上した不明点があれば追加で質問する、というループを続ける。
- 「もう質問することがない」と判断できる状態になるまで、proposal を確定させない（草案として残すのは可、確定は禁止）。
- ユーザーが「もう十分」「あとは任せる」と明示するまでは、AI 側の判断で疑問点解消フェーズを切り上げない。
- モジュール境界・レイヤー分割・抽象の導入など構造に関わる設計判断を含む spec を書く際は、`35-architect-principle` スキルの判断基準（変更容易性を North Star に SOLID・KISS/YAGNI・DRY・関心の分離をトレードオフとして適用）に従うこと。
- 自明でない設計判断（命名・データ構造・境界条件・エラーハンドリング・他 spec との関係など）を確定させる前に、`35-architect-doubt-driven-development` スキルを参照し、フレッシュコンテキストの敵対的レビューにかけること。

## セキュリティについて

ユーザー入力・認証・認可・データ保存・外部連携を扱う spec を書く際は、`35-architect-security` スキルを参照し、脅威モデリング（STRIDE / OWASP 等）の観点を proposal の段階で仕様に織り込むこと。

## DB 設計について

DB スキーマ（テーブル・カラム・リレーション・インデックス・マイグレーション）の設計を含む spec を書く際は、`35-architect-db-design` スキルを参照すること。データベースはアプリケーション層より遥かに長く生存するため、スキーマの意思決定は proposal の段階で（実装で場当たり的に決めるのではなく）確定させる。
