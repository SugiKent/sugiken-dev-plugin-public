# 意味のある単位でコミットするスキル

作業ツリーの差分を、後から読んで意図が分かる **atomic な単位** に分割してコミットする手法。1 つの論理的変更 = 1 コミットを原則とする。

## 原則

1. **1 コミット = 1 つの論理的変更**: 機能追加・リファクタ・バグ修正・フォーマットを **同じコミットに混ぜない**。openspec のオーケストレーションでは **1 change（または 1 Wave）= 1 コミット** を基本とする。
2. **外科的にステージングする**: `git add -A` で丸ごと入れない。関連ファイルだけを pathspec で、あるいは同一ファイル内の別変更は `git add -p`（hunk 単位）で選ぶ。無関係な変更を巻き込まない。
3. **誤コミットを防ぐ**: 生成物・ビルド成果物・`tmp/` の中間結果・`.env` やシークレット・巨大バイナリをコミットしない。ステージ後に `git diff --staged` で必ず目視確認する。
4. **メッセージで意図を伝える**: なぜこの変更をしたかが伝わる件名を書く。Conventional Commits 形式（`feat:` / `fix:` / `refactor:` / `chore:` 等）を既存リポジトリが採用していれば踏襲する。openspec のコミットには **対象 change ID を含める**。
5. **コミットは検証済みの状態で切る**: 壊れた中間状態をコミットしない。型 / lint / テストが通る、あるいは意図的に落ちる理由が説明できる状態でコミットする。

## 手順

1. **現状把握**: `git status` と `git diff`（未ステージ）、必要なら `git diff --staged` で全体像を掴む。既にステージされているものがあれば、意図と合うか確認する。
2. **論理単位に仕分ける**: 変更ファイル群を「どの論理的変更に属するか」で分類する。1 コミットにまとまるものと、分けるべきものを決める。
3. **外科的にステージ**:
   - ファイル単位で分けられるなら pathspec 指定: `git add path/to/a path/to/b`
   - 同一ファイルに複数の論理変更が混在するなら hunk 単位: `git add -p`（対話が使えない環境では、変更を分けて適用するか、ファイル単位に留める）
4. **ステージ内容を確認**: `git diff --staged` で、意図した変更だけが入っているか・シークレットや生成物が混じっていないかを目視する。
5. **コミット**: 意図が伝わる件名（＋必要なら本文）でコミットする。openspec なら change ID を件名か本文に含める。
6. **残差分を確認**: `git status` で、次のコミットに回すべき変更が残っていないか確認する。複数単位あれば 2〜5 を繰り返す。

## やってはいけないこと

- `git add -A` / `git add .` で無差別に全部ステージする。
- 無関係な変更を 1 コミットに同梱する。
- 指示なく `git push` / force push / `git reset --hard` / ブランチ切り替え / stash をする（破壊的・他者の作業を妨げる操作は指示があるまで行わない）。
- 壊れた中間状態やシークレット・生成物をコミットする。

## 報告に含めること

- 作成したコミットの一覧（SHA 短縮 + 件名）と、各コミットが含む論理単位。
- 意図的にコミットしなかった変更（残した理由）。
