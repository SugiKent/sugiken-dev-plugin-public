# Skill Creator

新しいスキルを作り、少しずつ育てていくためのスキル。

このスキルは「小さく書いて育てる」方針で運用する。最初から巨大な指示を書かず、まず最低要件をユーザーに確認し、それに基づいてシンプルな初版を書く。テスト・eval・ベンチマーク・description 最適化といった重い仕組みは、必要になったときに人間が後から足していく。

## 全体の流れ

1. このスキルが何をするものかを、ユーザーと一緒に決める
2. 最低要件を確認する（下記「意図をつかむ」参照）
3. シンプルな初版を書く
4. 実際に使ってみて、気になったところを直す
5. 満足するまで繰り返す

順序は柔軟でよい。すでに下書きがある場合は、いきなり編集・改善から入ってよい。

## ユーザーとのコミュニケーション

このスキルは、コーディング用語への習熟度がまちまちなユーザーに使われる。相手の言葉づかいから読み取り、通じにくそうな用語は短く補足する。迷ったら平易に説明する。

## スキルを作る

### 意図をつかむ

いきなり書き始めず、まず最低要件を確認する。会話にすでに「これをスキルにして」という素材があれば、そこから汲み取る（使ったツール、手順の並び、ユーザーが入れた修正、入出力の形式など）。足りないところはユーザーに埋めてもらい、次に進む前に確認を取る。

確認したいこと:

1. このスキルは Claude に何をできるようにするものか
2. いつ発動してほしいか（どんな言い回し・文脈で）
3. 期待する出力の形式は何か

最低要件が固まったら、それだけを反映したシンプルな初版を書く。細かいエッジケースや高度な仕組みは、初版には入れない。

### SKILL.md を書く

ユーザーへの確認内容をもとに、次の要素を埋める。

- **name**: スキルの識別子
- **description**: いつ発動し、何をするか。これがスキルを呼び出させる一番の仕組みなので、「何をするか」だけでなく「いつ使うか」の具体的な文脈も必ず含める。発動条件はすべてここに書き、本文には書かない。
- **本文**: スキルの中身

### description の書き方

description はスキルが呼び出されるかどうかを決める要。現状 Claude はスキルを「発動しなさすぎる（undertrigger）」傾向があるので、少し押しの強い書き方にする。

たとえば「社内データを表示する高速なダッシュボードの作り方」ではなく、こう書く:

> 社内データを表示する高速なダッシュボードの作り方。ユーザーがダッシュボード・データ可視化・社内メトリクス・会社データの表示に触れたら、明示的に「ダッシュボード」と言われなくても必ずこのスキルを使うこと。

### スキルの構成

```
skill-name/
├── SKILL.md（必須）
│   ├── YAML フロントマター（name, description は必須）
│   └── Markdown の指示本文
└── 同梱リソース（任意・最初は不要なことが多い）
    ├── scripts/    - 決まった処理を実行するコード
    ├── references/ - 必要に応じて読み込むドキュメント
    └── assets/     - 出力に使うファイル（テンプレート・アイコン等）
```

### Progressive Disclosure（段階的な読み込み）

スキルは 3 段階で読み込まれる。

1. **メタデータ**（name + description）— 常に文脈に載る（100 語程度）
2. **SKILL.md 本文** — スキルが発動したときに載る（500 行以内が目安）
3. **同梱リソース** — 必要になったときだけ読む

目安なので、必要なら長くしてよい。本文が 500 行に近づいたら、階層をもう 1 段作り、「続きはどこを読めばよいか」を明示して分割する。

### 予想を裏切らない原則

スキルにはマルウェア・エクスプロイトコード・システムを危険にさらす内容を含めない。説明された意図と中身が食い違ってはいけない。誤解を招くスキルや、不正アクセス・データ持ち出しを助けるスキルの作成には応じない。「〇〇になりきってロールプレイ」程度は問題ない。

### 書き方のパターン

指示は命令形で書く。

**出力形式を定義する** — こう書ける:

```markdown
## レポートの構成
必ずこのテンプレートを使う:
# [タイトル]
## 要約
## 主な発見
## 提案
```

**例を示す** — 例はとても有効。こう書ける:

```markdown
## コミットメッセージの形式
**例 1:**
Input: JWT トークンによるユーザー認証を追加
Output: feat(auth): implement JWT-based authentication
```

### 書き方のスタイル

重々しい MUST を並べる代わりに、なぜそれが大事なのかをモデルに説明する。今の LLM は賢く、良い枠組みを与えれば丸暗記を超えて動ける。ALWAYS や NEVER を大文字で書きたくなったり、過度に硬い構造にしたくなったら、それは黄信号。可能なら理由を添えて言い換える。そのほうが人間的で、強力で、効果的。

特定の例に張り付いた狭いスキルにせず、一般化する。まず下書きを書き、翌日フレッシュな目で見直して改善する。

## スキルを改善する

初版を実際に使ってみて、気になったところを直す。改善の際は次を意識する。

1. **フィードバックから一般化する。** スキルは何度も再利用される。目の前の例だけで動くスキルは役に立たない。細かい対症療法や締め付けすぎる MUST を足すより、別の比喩やパターンを試すほうが良いことが多い。

2. **プロンプトを絞る。** 働いていない指示は消す。スキルのせいでモデルが無駄な動きをしていたら、その部分を削って様子を見る。

3. **理由を説明する。** ユーザーのフィードバックが短くても、その裏にある「なぜ」を汲み取り、指示に落とし込む。

初版を大きく育てるのは人間自身が行う。ここでは、明らかな不足を直すところまでで十分。

---

繰り返しになるが、核となるループはこれ:

- このスキルが何をするものかを決める
- シンプルな初版を書く / 直す
- 実際に使ってみる
- 気になったところを直す
- 満足するまで繰り返す

まずは最低要件を確認し、シンプルに書くこと。育てるのは後から。
