# /retro — セッション振り返りスキル

セッション中の会話を分析し、Claude Code の設定資産（スキル・ルール・ドキュメント）への更新提案を生成する。

## 目的

セッション中に得た「気付き」を構造化し、以下の3カテゴリに対する具体的な更新提案としてまとめる:

1. **スキル** — 新規スキルの追加、既存スキルの改善（プロジェクト固有なら `.claude/skills/`、汎用なら独自 plugin に取り込むかを提案してほしい）
2. **ルール** (`.claude/rules/`) — 新規ルールの追加、既存ルールの修正
3. **ドキュメント** (`docs/howToDevelopment/`, `AGENTS.md`) — ドキュメントの追加・更新提案

**なぜレトロが重要か？** セッション中の知識は揮発性が高い。振り返りで構造化しないと、同じ落とし穴に何度もはまる。1回5分のレトロが、将来の数時間を節約する。

## 使い方

```bash
/retro                     # セッション全体を振り返り
/retro [トピック]           # 特定のトピックにフォーカス
```

## Workflow

### Phase 1: セッション分析（気付きの抽出）

現在のセッションの会話履歴をレビューし、以下の観点で「気付き」を抽出する:

| 検出カテゴリ | 具体例 |
|-------------|--------|
| **遭遇した落とし穴** | 「サービス名が実は別名だった」「環境変数の設定漏れで30分費やした」 |
| **新しい技術パターン** | 「Prismaのこの書き方が便利だった」「このテストパターンが有効」 |
| **ワークフローギャップ** | 「この手順がドキュメント化されていなかった」 |
| **ツール・コマンド発見** | 「このCLIコマンドが便利」「このデバッグ手法が有効」 |
| **時間浪費ポイント** | 「誤った前提で20分費やした」「ドキュメントが古くて混乱した」 |
| **成功パターン** | 「このアプローチがうまくいった」「このスキルの使い方が効果的だった」 |

### Phase 2: 更新対象の特定

抽出した気付きを、以下の3カテゴリに分類して更新対象を特定する:

#### A. スキル更新

- **新規スキル**: 繰り返し行った作業パターンがスキル化できる場合
- **既存スキル改善**: 使ったスキルに改善点がある場合
- 対象の判断:
  - そのプロジェクト固有の内容 → プロジェクトの `.claude/skills/`
  - プロジェクトを跨いで使える汎用的な内容 → plugin repo `SugiKent/ai-agents-skills-and-more-for-dev` の `plugins/*/skills/`（plugin 経由で配布されているスキルの改善はこちら。install 先のキャッシュを直接編集しない）

#### B. ルール更新 (`.claude/rules/`)

- **新規ルール**: 特定のファイルパス編集時に自動ロードすべき注意事項を発見した場合
- **既存ルール修正**: 現行ルールに不足や誤りがある場合
- 対象ディレクトリ: `.claude/rules/`

編集をしようとするファイルパスを指定できない場合は Skill またはドキュメントにする.
そのため提案時にはどの path をしているのかも提示する

#### C. ドキュメント更新

- **開発ガイドライン**: `docs/howToDevelopment/` 配下のドキュメント追加・修正
- **CLAUDE.md**: プロジェクト説明の更新
- **その他**: 手順書、トラブルシューティングガイド等

### Phase 3: 差分プレビューの生成

各更新提案について、以下のフォーマットでプレビューを生成する:

```
## 振り返り結果

### 検出した気付き（N件）

1. [気付き1の要約]
2. [気付き2の要約]
...

---

### 更新提案

#### 📁 スキル更新（N件）

##### [提案1] [新規/改善] — スキル名
- **ファイル**: `.claude/skills/[name]/SKILL.md`（汎用なら plugin repo の `plugins/[plugin]/skills/[name]/SKILL.md`）
- **理由**: [なぜこの更新が必要か]
- **概要**: [変更内容の要約]

#### 📏 ルール更新（N件）

##### [提案1] [新規/改善] — ルール名
- **ファイル**: `.claude/rules/[name].md`
- **理由**: [なぜこの更新が必要か]
- **概要**: [変更内容の要約]

#### 📝 ドキュメント更新（N件）

##### [提案1] [新規/改善] — ドキュメント名
- **ファイル**: `docs/howToDevelopment/[path]`
- **理由**: [なぜこの更新が必要か]
- **概要**: [変更内容の要約]

---

### 次のアクション

1. 提案を修正 → 修正内容を指示
2. 提案を却下 → 理由を共有（今後の改善に活用）
```

### Phase 4: 承認とDoc-ship連携

ユーザーが提案を承認した場合:

1. 承認された提案の内容を具体的なファイル変更として準備

**重要**: ユーザーの承認なしにファイルを変更しない。プレビューの提示 → 承認 → 実行の順序を厳守する。

## 気付きがない場合

セッション中に有意な気付きがなかった場合は、正直にその旨を報告する:

```
## 振り返り結果

今回のセッションでは、スキル・ルール・ドキュメントに反映すべき新しい気付きは検出されませんでした。
```

# 判断基準: 何を提案すべきか

| 提案すべき | 提案すべきでない |
|-----------|----------------|
| 繰り返し発生しうる問題の回避策 | 一度きりの特殊ケース |
| ドキュメント化されていない手順 | 既にドキュメント化済みの内容 |
| 既存ドキュメントの誤り・古い情報 | 些末な表現の改善 |
| 自動化できる繰り返し作業 | 自動化の恩恵が薄い作業 |
| チーム全体に役立つ知見 | 個人の好みレベルの内容 |
