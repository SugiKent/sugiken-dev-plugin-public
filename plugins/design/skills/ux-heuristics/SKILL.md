# UX ヒューリスティック フレームワーク

ユーザーインターフェースを評価・改善するための実践的なユーザビリティ原則。ユーザーは読まない、スキャンする。最適な選択はせず、満足化 (satisfice) する。仕組みを理解しようとはせず、なんとなく手探りで進める (muddle through)。

## 中核原則

**「Don't Make Me Think（考えさせるな）」** — すべてのページは自明であるべき。何かを考えさせる時点で、それはユーザビリティの問題である。ユーザーの忍耐力と認知帯域は限られているため、上記の「スキャン・満足化・手探り」という行動を前提に設計する。

## スコアリング

**目標: 10/10。** インターフェースを監査し、見つかった各問題を下記の重大度スケールで評価し、Quick Diagnostic（後述）の結果からスコアを付ける。10 点から始め、失敗した診断行ごとに、その行が引き起こす最悪の重大度で重み付けして減点する（catastrophic / major の行は約 2 点、minor / cosmetic は約 1 点）。バンド: **9-10** = 重大度 3 以上の問題がなく、失敗した診断行が 1 行以下。**6-8** = major な問題がいくつか、あるいは複数の失敗行。**3-5** = catastrophic な問題が 1 つ、あるいは多数の失敗行。**2 以下** = 中核タスクが遂行不能。常に、現在のスコア、最も重大度の高い問題、10/10 に到達するために必要な具体的な修正を提示すること。

## Krug のユーザビリティ原則

法則 1〜3 は Krug の「ユーザビリティ 3 原則」、4 番目の Trunk Test はナビゲーションの現在地確認テストである。

### 1. Don't Make Me Think（考えさせるな）

**中核概念:** ユーザーの頭に浮かぶ疑問符 1 つ 1 つが認知負荷を増やし、タスクから注意を逸らす。

**なぜ機能するか:** ユーザーは目的を持って行動している — ラベルに悩んだり、気の利いたマーケティング表現を解読したりしたくない。考える必要が少ないほど、タスク完了の可能性が上がる。

**重要な洞察:**

- 気の利いた名前は、明快な名前に必ず負ける
- マーケティング的言い回しは摩擦を生む。平易な言葉は摩擦を取り除く
- 馴染みのないカテゴリは、ユーザーの手を止めさせ解釈を強いる
- 曖昧なリンクやボタンは、ためらいを生む

**プロダクトへの適用:**


| 文脈              | 適用           | 例                                |
| --------------- | ------------ | -------------------------------- |
| **ナビゲーションのラベル** | 自明な名前        | 「経路案内」であって「目的地までのルートを計算」ではない     |
| **CTA**         | ユーザーが理解できる動詞 | 「サインイン」であって「アカウントポータルにアクセス」ではない  |
| **エラー状態**       | 次にすべきことを伝える  | 「メールの形式を確認してください」であって「検証エラー」ではない |


**コピーのパターン:**

- 行動を促すボタン: 動詞 + 名詞（「アカウントを作成」「レポートをダウンロード」）
- 専門用語を避ける: 「永続化する」ではなく「保存」、「関連付けを解除」ではなく「削除」
- ラベルに説明が必要なら、ラベルを簡潔にする

**倫理的境界:** 明快さはユーザーのためにある — 不利な条件を隠す上辺として平易な言葉を使ってはならない。

### 2. クリック数は問題ではない

**中核概念:** 「ユーザーは 3 クリックで離脱する」という俗説があるが、実際にはユーザーはクリック数を気にしない。各クリックが無痛で、自明で、確信を持てるものであれば問題ない。

**なぜ機能するか:** クリック数よりも 1 クリックあたりの認知的努力が重要。ユーザーはクリックの忍耐が尽きて離脱するのではなく、確信を失って離脱する。

**重要な洞察:**

- 各クリックは無痛・自明・確信を持てるものであるべき
- 熟慮を要する 1 クリックより、何も考えずに済む 3 クリックのほうが良い
- ラベルが曖昧な深い階層より、ラベルが明快な浅い階層のほうが良い

**プロダクトへの適用:**


| 文脈             | 適用               | 例                                               |
| -------------- | ---------------- | ----------------------------------------------- |
| **チェックアウトフロー** | 各ステップを自明にする      | 説明的なラベル付きの明確なステップインジケーター                        |
| **設定**         | フラットな一覧より明確なカテゴリ | 「アカウント &gt; セキュリティ &gt; パスワード変更」（確信を持てる 3 クリック） |
| **オンボーディング**   | 小さく明確なステップ       | 1 ステップ 1 アクションのウィザード                            |


**コピーのパターン:**

- 進捗インジケーター: 「ステップ 2 / 4: 配送先の詳細」
- 各ステップでの確認: 「メールを確認しました。次はプロフィールを設定しましょう」
- 明確なリンクテキスト: 「ランニングシューズをすべて見る」であって「こちらをクリック」ではない

**倫理的境界:** 解約フローを埋もれさせるために余計なステップを使ってはならない — すべてのクリックはユーザーをゴールに近づけるべきで、遠ざけてはならない。

### 3. 言葉を半分に削れ

**中核概念:** 各ページの言葉を半分に削り、残ったものをさらに半分に削る。簡潔さは有用なコンテンツを際立たせ、ユーザーの時間を尊重する。

**重要な洞察:**

- 「ようこそ！」のような社交辞令（happy-talk）はスペースの無駄
- 誰も読まない説明文は削除する
- 「どうぞ」「恐れ入りますが」といった丁寧な飾りはノイズを増やす
- ページが短いほどスクロールが減り、スキャンが速くなる

**プロダクトへの適用:**


| 文脈                    | 適用              | 例                       |
| --------------------- | --------------- | ----------------------- |
| **ランディングページ**         | ようこそ文を削り、価値から入る | 「ようこそ…」の段落を削除           |
| **エラーメッセージ**          | 問題と対処だけを述べる     | 「パスワードが短すぎます（最低 8 文字）」  |
| **空状態 (empty state)** | 行動を促し、最小限に      | 「結果がありません。別の検索を試してください」 |


**コピーのパターン:**

- Before:「次のステップに進むにはパスワードを入力していただく必要がございますのでご了承ください」→ After:「続けるにはパスワードを入力してください」
- Before:「メッセージを受け取りました。できるだけ早くご返信いたします」→ After:「送信しました。24 時間以内に返信します」

**倫理的境界:** 簡潔さは重要な情報の省略であってはならない — 価格・利用規約・データ利用に関する簡潔な開示はユーザーの権利である。

### 4. Trunk Test（トランクテスト）

**中核概念:** ユーザーを任意のページにいきなり放り込む（車のトランクから見知らぬ場所で放り出されるように）。それでもユーザーは 6 つの現在地確認の問いに即座に答えられるべきである: これは何のサイトか？ 何のページか？ 主要なセクションは何か？ ここでの選択肢は何か？ 階層のどこにいるか？ 検索はどこか？

**なぜ機能するか:** 良いナビゲーションは常に現在地を示す。自分がどこにいるか分からないユーザーは迷い、離脱する。

これはナビゲーションの確認テストとして適用する: ページタイトルはユーザーがクリックしたリンクと一致すること、「現在地」インジケーター（ハイライトされたナビ項目、太字のパンくず）があること、セクション見出しが現在地を示すこと（単なる「設定」ではなく「アカウント &gt; 請求」）。

**倫理的境界:** ナビゲーションのラベルはサイト構造を正直に表すべき — 誤解を招くラベルでユーザーをマーケティングページへ誘導してはならない。

## Nielsen の 10 ユーザビリティ ヒューリスティック

### 1. システム状態の可視性

適時のフィードバックでユーザーに状況を伝え続ける。すべての操作には確認が必要 — アップロードには進捗バー、送信には確認、読み込みにはスケルトン画面。無言の失敗は信頼を破壊する。コピー例:「保存中…」→「保存しました」。

### 2. システムと現実世界の一致

ユーザーの言葉で語る:「認証」ではなく「サインイン」、「クエリ」ではなく「検索」。現実世界のメタファー（ゴミ箱、ショッピングカート）と自然な順序（番地 → 市区町村 → 都道府県 → 郵便番号）に従う。

### 3. ユーザーのコントロールと自由

明確な「非常口」を用意する。「本当によろしいですか？」のダイアログより Undo が優れている — ユーザーは確認を読まずにクリックしてしまう。すべてのフローにキャンセル / 退出が必要で、戻るボタンは決して壊れてはならない。

### 4. 一貫性と標準

同じ言葉・スタイル・挙動は、全体を通じて同じ意味を持つ。内部的一貫性（自分のアプリ内）に加え、外部的一貫性（プラットフォームの慣習: ロゴは左上、検索は右上）。1 つの概念には 1 つの用語 —「プロジェクト」で統一し、「ワークスペース」と混在させない。

### 5. エラーの予防

問題が起きる前に防ぐ: 制約された入力（テキスト欄より日付ピッカー）、オートコンプリート、妥当なデフォルト、「未保存の変更」警告。スリップ（うっかり間違った操作）とミステイク（間違った意図）は別々の予防策が必要。

### 6. 記憶より認識

記憶の負荷を最小化する — 選択肢を見せ、覚えさせない。パンくず、最近の検索、事前入力されたフィールド、デコード済みの値を持つドロップダウン。作業記憶は約 7 項目しか保持できない。認識は再生（recall）より遥かに容易。

### 7. 柔軟性と効率性

初心者と熟練者の両方に応える: キーボードショートカット、一括操作、保存済み検索、コマンドパレット（Cmd+K）。段階的開示 (progressive disclosure) により、初心者にはシンプルなまま、熟練者はフル機能にアクセスできる。

### 8. 美的でミニマルなデザイン

すべての要素はその場所を勝ち取らねばならない — すべてが注意を求めて叫ぶと、何も際立たない。今重要なものを見せ、そうでないものは隠す。1 ページに主要 CTA は 1 つ。

### 9. エラーの認識・診断・回復を助ける

エラーメッセージには 3 要素が必要: 何が起きたか、なぜか、どう直すか。平易な言葉（「ECONNREFUSED」ではなく「接続に失敗しました」）、具体的に（「無効です」ではなく「パスワードは 8 文字以上必要です」）、ユーザーを責めず、入力内容を保持する。

### 10. ヘルプとドキュメント

ヘルプは検索可能で、タスク志向（技術リファレンスではなく「〜する方法」）で、文脈に沿ったもの（ツールチップ、インラインヒント、ガイドツアー）であるべき。

## 重大度評価スケール

監査で見つかった各問題を評価する:


| 重大度   | 評価                 | 説明                     | 優先度      |
| ----- | ------------------ | ---------------------- | -------- |
| **0** | 問題ではない             | 意見の相違であり、ユーザビリティ問題ではない | 無視       |
| **1** | 表面的 (Cosmetic)     | 些細な煩わしさ、影響は小さい         | 時間があれば修正 |
| **2** | 軽微 (Minor)         | 遅延や苛立ちを生む              | 修正を計画    |
| **3** | 重大 (Major)         | 深刻なタスク失敗を招く            | 早急に修正    |
| **4** | 致命的 (Catastrophic) | タスク完了を妨げる              | 直ちに修正    |


3 つの要因を勘案する: **頻度**（どれくらい起きるか）、**影響**（起きたときどれくらい深刻か）、**持続性**（一度きりか継続的か）。

## よくある間違い


| 間違い                                   | なぜ失敗するか                     | 修正                          |
| ------------------------------------- | --------------------------- | --------------------------- |
| **正体不明のナビ (mystery meat navigation)** | ラベルのないアイコンは推測を強いる           | アイコンにテキストラベルを併記             |
| **選択肢が多すぎる**                          | 決定麻痺でユーザーが鈍化                | 7 ± 2 項目に減らす                |
| **「現在地」インジケーターがない**                   | 階層の中で迷子になる                  | ナビとパンくずで現在のセクションをハイライト      |
| **インライン検証がない**                        | 送信 → エラー → スクロールの繰り返しが苛立たせる | blur 時に具体的なメッセージで検証         |
| **必須フィールドが不明確**                       | 何が必須か分からず混乱                 | 必須ではなく任意の項目に印を付ける           |
| **文字の壁**                              | 誰も密な段落を読まない                 | 見出し・箇条書き・余白で分割              |
| **ラベルの専門用語**                          | ユーザーは社内用語を話さない              | 全ラベルをユーザーテストし、平易な言葉を使う      |
| **読み込みインジケーターがない**                    | システムが壊れたと思われる               | スピナー・進捗バー・スケルトン画面を表示        |
| **タップ領域が小さい**                         | モバイルユーザーが誤タップを繰り返す          | 最低 44 × 44 px のタッチ領域        |
| **ホバー限定の情報**                          | モバイル・キーボードユーザーが見逃す          | 重要情報をホバー裏に隠さない              |
| **Undo がない**                          | ユーザーが操作を恐れる                 | 非破壊的操作すべてに Undo を用意         |
| **貧弱なエラーメッセージ**                       | 「無効な入力」では何も伝わらない            | 何が間違いでどう直すか説明               |
| **低コントラストの文字**                        | 多くの人が読めない                   | WCAG AA 最低基準（コントラスト比 4.5:1） |
| **ナビ位置が一貫しない**                        | ナビを見つけられない                  | 固定位置、全ページで同じ場所              |
| **戻るボタンが壊れている**                       | ブラウザの約束を破る                  | ブラウザ履歴を乗っ取ったり壊したりしない        |


## Quick Diagnostic（クイック診断）


| 問い                   | No の場合                 | アクション                          |
| -------------------- | ---------------------- | ------------------------------ |
| 何のサイト / ページか即座に分かるか？ | 迷子になる                  | 明確なロゴ・ページタイトル・パンくずを追加          |
| 主要な操作は自明か？           | 何をすべきか分からない            | 視覚的階層、単一の主要 CTA                |
| ナビゲーションは明快か？         | 進む道が分からない              | Trunk Test を適用し「現在地」インジケーターを追加 |
| 検索を見つけられるか？          | 目的志向のユーザーが行き詰まる        | ヘッダーに見える検索ボックス                 |
| システムは何が起きているか示すか？    | 信頼を失い再クリックする           | 読み込み状態・確認・進捗                   |
| エラーメッセージは役立つか？       | ユーザーが行き詰まる             | 平易な言葉で具体的な対処を示す                |
| Undo や戻るはできるか？       | 操作を恐れる                 | Undo・キャンセル・戻るをあらゆる場所に          |
| ホバーなしで機能するか？         | モバイル / キーボードユーザーが排除される | ホバー操作に見える代替を用意                 |
| すべての操作要素にラベルがあるか？    | アイコンを推測させる             | テキストラベルまたは説明的なツールチップ           |
| 「ん？」と立ち止まらせるものはないか？  | 認知負荷が高すぎる              | 単純化する — 説明が要るなら再設計             |


## ヒューリスティックの衝突

ヒューリスティックは互いに矛盾することがある。その場合:

- **シンプルさ vs 柔軟性**: 段階的開示を使う
- **一貫性 vs 文脈**: パターンは一貫させ、目立たせ方は文脈に応じる
- **効率性 vs エラー予防**: 確認ダイアログより Undo を優先する
- **発見性 vs ミニマリズム**: 主要な操作は見せ、副次的なものは隠す

## ダークパターンの認識

ダークパターンはユーザーを操作するために意図的にヒューリスティックを破る: 強制的継続（解約しにくい）、ローチモーテル（入るのは簡単、出るのは困難）、コンファームシェイミング（罪悪感を煽る選択肢）、隠れコスト（チェックアウト時の不意の追加料金）。設計がユーザーに奉仕するのではなく操作していると疑われるときは、それがユーザーの利益に反していないか点検する。

## 各手法を使う場面


| 手法          | いつ           | 所要時間   | 得られるもの  |
| ----------- | ------------ | ------ | ------- |
| ヒューリスティック評価 | ユーザーテストの前    | 1〜2 時間 | 主要な違反   |
| ユーザーテスト     | ヒューリスティック修正後 | 2〜4 時間 | 実際の行動   |
| A/B テスト     | 最適化するとき      | 数日〜数週間 | 統計的検証   |
| 分析レビュー      | 継続的          | 30 分   | パターンと問題 |
