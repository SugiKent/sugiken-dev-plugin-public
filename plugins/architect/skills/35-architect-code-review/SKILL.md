# コードレビューと品質

## 概要

品質ゲートを伴う多次元のコードレビュー。あらゆる変更はマージ前にレビューされる — 例外はない。レビューは 5 つの軸をカバーする: 正確性（correctness）、可読性（readability）、アーキテクチャ（architecture）、セキュリティ（security）、パフォーマンス（performance）。

**承認の基準:** 変更がコード全体の健全性を確実に改善するなら、完璧でなくても承認する。完璧なコードなど存在しない — 目的は継続的な改善である。自分ならこう書いた、という理由だけで変更をブロックしてはならない。コードベースを改善し、プロジェクトの規約に従っているなら承認する。

## いつ使うか

- あらゆる PR・変更をマージする前
- 機能実装を完了した後
- 別のエージェントやモデルが生成したコードを評価する必要があるとき
- 既存コードをリファクタリングするとき
- バグ修正の後（修正とリグレッションテストの両方をレビューする）

## 5 軸レビュー

あらゆるレビューは以下の次元でコードを評価する:

### 1. 正確性（Correctness）

コードは主張する通りに動作するか?

- 仕様やタスク要件に合致しているか?
- エッジケース（null・空・境界値）は処理されているか?
- エラー経路は処理されているか（ハッピーパスだけではないか）?
- すべてのテストをパスするか? テストは本当に正しいものを検証しているか?
- off-by-one エラー・競合状態（race condition）・状態の不整合はないか?

### 2. 可読性とシンプルさ（Readability & Simplicity）

他のエンジニア（やエージェント）が、作者の説明なしにこのコードを理解できるか?

- 名前は説明的で、プロジェクトの規約と一貫しているか?（文脈のない `temp`・`data`・`result` は不可）
- 制御フローは素直か（ネストした三項演算子や深いコールバックを避ける）?
- コードは論理的に整理されているか（関連コードがまとまり、モジュール境界が明確か）?
- シンプルにすべき「巧妙な」トリックはないか?
- **より少ない行数でできないか?**（100 行で済むところに 1000 行は失敗である）
- **抽象化はその複雑さに見合っているか?**（3 つ目のユースケースが来るまで一般化しない）
- 自明でない意図を明確にするコメントは役立つか?（ただし自明なコードにコメントしない）
- デッドコードの残骸はないか: no-op 変数（`_unused`）、後方互換のためのシム、`// removed` コメントなど?
- **無関係なフローに新しい条件分岐がボルト留めされていないか?** それは些細な指摘ではなく設計上の臭いである — 既存の経路に絡ませるのではなく、ロジックを専用のヘルパー・状態・ポリシーに押し出す。
- **同じ形状に対する条件分岐が繰り返し現れていないか?** それはモデルやディスパッチャの欠如を示している。「一時的な」分岐はたいてい恒久的な負債になる。

### 3. アーキテクチャ（Architecture）

この軸の判断基準（変更容易性・KISS/YAGNI・DRY・SOLID・関心の分離・過剰設計の見極め）は `35-architect-principle` スキルに一元化されている。構造の是非を深く問うときは同スキルを参照する。

変更はシステムの設計に適合しているか?

- 既存のパターンに従っているか、それとも新しいパターンを導入しているか? 新しいなら、それは正当化されるか?
- クリーンなモジュール境界を保っているか?
- 共有すべきコードの重複はないか?
- 依存関係は正しい方向に流れているか（循環依存がないか）?
- 抽象化のレベルは適切か（過剰設計でも密結合でもないか）?
- **このリファクタリングは複雑さを減らしているか、それともただ移動させているだけか?** 変更を追うために読み手が頭に保持すべき概念の数を数える。「よりクリーン」なバージョンでもその数が変わらないなら、それはクリーンではない — 同じロジックを再集約するものより、分岐・モード・レイヤーまるごとを消し去る再構成を優先する。抽象を磨くより、抽象を削除することを優先する。
- **機能固有のロジックが、共有・汎用のモジュールに漏れ出していないか?** ロジックはそれを所有するレイヤーに留め、近似の重複ではなく既存の正規のヘルパーを再利用し、アーキテクチャの逸脱を常態化させない。
- **型境界は明示的か?** 不必要な `any`/`unknown`/optional/キャストや、不明確な不変条件を覆い隠す暗黙のフォールバックを問いただす — 境界を明示することで、周囲の制御フローがしばしばシンプルになる。

### 4. セキュリティ（Security）

詳細なセキュリティガイダンスは `35-architect-security` スキルを参照。変更は脆弱性を導入していないか?

- ユーザー入力は検証・サニタイズされているか?
- シークレットはコード・ログ・バージョン管理の外に保たれているか?
- 必要な箇所で認証・認可がチェックされているか?
- SQL クエリはパラメータ化されているか（文字列連結ではないか）?
- 出力は XSS を防ぐためにエンコードされているか?
- 依存関係は信頼できるソースからで、既知の脆弱性がないか?
- 外部ソース（API・ログ・ユーザーコンテンツ・設定ファイル）からのデータを信頼できないものとして扱っているか?
- 外部データフローは、ロジックやレンダリングで使用する前にシステム境界で検証されているか?

### 5. パフォーマンス（Performance）

詳細なプロファイリングと最適化のチェックは `references/performance-checklist.md` を参照。変更はパフォーマンス問題を導入していないか?

- N+1 クエリのパターンはないか?
- 上限のないループや制約のないデータ取得はないか?
- 非同期にすべき同期処理はないか?
- UI コンポーネントに不要な再レンダリングはないか?
- リストエンドポイントにページネーションの欠如はないか?
- ホットパスで大きなオブジェクトを生成していないか?

## 構造的な是正策

構造的な問題を指摘するときは、問題だけでなく「動き」を提案する。「これは複雑だ」とだけ言うレビューは、作者を推測させたままにする。名前のついた再構成に手を伸ばす:

- **条件分岐の連鎖を置き換える** — 型付きのモデルや明示的なディスパッチャで。
- **重複した分岐を畳む** — より明確な単一のフローに。
- **オーケストレーションとビジネスロジックを分離する** — それぞれが単独で読めるように。
- **機能固有のロジックを移動する** — 共有モジュールから、その概念を所有するパッケージへ。
- **正規のヘルパーを再利用する** — 独自仕様の近似重複ではなく。
- **型境界を明示する** — 下流の分岐が消えるように。
- **パススルーのラッパーを削除する** — API を明確にせず間接層を増やすだけのもの。
- **ヘルパーを抽出する、または大きなファイルを分割する** — 焦点の絞られたモジュールへ。

動く部品を増やすものより、同じ複雑さをあちこちにばらまくものより、動く部品を取り除く是正策を優先する。

## 変更の説明（Change Descriptions）

あらゆる変更には、バージョン管理の履歴の中で単独で成立する説明が必要である。

**最初の行:** 短く、命令形で、単独で成立する。"Deleting the FizzBuzz RPC" ではなく "Delete the FizzBuzz RPC"。履歴を検索した誰かが、diff を読まずに変更を理解できる程度に有益でなければならない。

**本文:** 何が、なぜ変わるのか。コード自体には見えない文脈・決定・理由を含める。関連するならバグ番号・ベンチマーク結果・設計ドキュメントへのリンクを張る。アプローチに欠点があるなら認める。

**アンチパターン:** "Fix bug"、"Fix build"、"Add patch"、"Moving code from A to B"、"Phase 1"、"Add convenience functions"。

## レビュープロセス

### ステップ 1: 文脈を理解する

コードを見る前に、意図を理解する:

```
- この変更は何を達成しようとしているか?
- どの仕様やタスクを実装するものか?
- 期待される振る舞いの変化は何か?
```

### ステップ 2: テストを先にレビューする

テストは意図とカバレッジを明らかにする:

```
- 変更に対するテストは存在するか?
- それらは（実装の詳細ではなく）振る舞いをテストしているか?
- エッジケースはカバーされているか?
- テストには説明的な名前が付いているか?
- コードが変わったとき、テストはリグレッションを捕まえられるか?
```

### ステップ 3: 実装をレビューする

5 軸を念頭にコードを追う:

```
変更された各ファイルについて:
1. 正確性: このコードはテストが言う通りに動作するか?
2. 可読性: 助けなしに理解できるか?
3. アーキテクチャ: システムに適合するか?
4. セキュリティ: 脆弱性はないか?
5. パフォーマンス: ボトルネックはないか?
```

### ステップ 4: 指摘を分類する

すべてのコメントに重大度のラベルを付け、作者が必須と任意を区別できるようにする:

| 接頭辞 | 意味 | 作者のアクション |
|--------|------|------------------|
| *(接頭辞なし)* | 必須の変更 | マージ前に対応必須 |
| **Critical:** | マージをブロック | セキュリティ脆弱性・データ損失・機能の破壊 |
| **Nit:** | 些細・任意 | 作者は無視してよい — フォーマット・スタイルの好み |
| **Optional:** / **Consider:** | 提案 | 検討の価値はあるが必須ではない |
| **FYI** | 情報のみ | アクション不要 — 将来の参考のための文脈 |

これにより、作者がすべてのフィードバックを必須として扱い、任意の提案に時間を浪費するのを防ぐ。

**重要なものを先頭に。** 指摘はレバレッジ順に並べる: まず正確性とセキュリティ、次に構造的な後退と見逃された簡素化、その後にそれ以外すべて。実在する問題を見栄えだけの些細な指摘の下に埋もれさせない — 高確信のコメント数件は、長いリストに勝る。構造的な問題が 1 つと些細な指摘が 10 個あるなら、その構造的な問題こそ*が*レビューである。

### ステップ 5: 検証を検証する

作者の検証ストーリーをチェックする:

```
- どんなテストが実行されたか?
- ビルドはパスしたか?
- 変更は手動でテストされたか?
- UI 変更のスクリーンショットはあるか?
- before/after の比較はあるか?
```

## マルチモデルレビューのパターン

異なる観点のために異なるモデルを使う:

```
モデル A がコードを書く
    │
    ▼
モデル B が正確性とアーキテクチャをレビューする
    │
    ▼
モデル A がフィードバックに対応する
    │
    ▼
人間が最終判断を下す
```

これは単一のモデルが見逃しうる問題を捕まえる — モデルごとに盲点が異なる。

**レビューエージェント向けのプロンプト例:**
```
このコード変更を、正確性・セキュリティ・我々のプロジェクト規約への準拠の
観点からレビューせよ。仕様は [X] と述べている。変更は [Y] であるべきだ。
問題は Critical、Required、Optional、Nit として指摘せよ。
```

## デッドコードの衛生（Dead Code Hygiene）

リファクタリングや実装変更の後、孤立したコードをチェックする:

1. 到達不能・未使用になったコードを特定する
2. それを明示的にリスト化する
3. **削除する前に尋ねる:**「これらの未使用要素を今削除すべきか: [リスト]?」

デッドコードを放置しない — 将来の読み手やエージェントを混乱させる。ただし、確信のないものを黙って削除しない。迷ったら尋ねる。

```
特定されたデッドコード:
- src/utils/date.ts の formatLegacyDate() — formatDate() に置き換え済み
- src/components/ の OldTaskCard コンポーネント — TaskCard に置き換え済み
- src/config.ts の LEGACY_API_URL 定数 — 参照が残っていない
→ これらを安全に削除してよいか?
```

## レビュー速度（Review Speed）

遅いレビューはチーム全体をブロックする。レビューのためのコンテキストスイッチのコストは、他者に課す待機コストより小さい。

- **1 営業日以内に応答する** — これは最大値であり、目標ではない
- **理想的なペース:** 集中したコーディングの最中でない限り、レビュー依頼が届いたら速やかに応答する。典型的な変更は 1 日のうちに複数回のレビューラウンドを完了すべきである
- **速い最終承認より、速い個別応答を優先する。** 速いフィードバックは、複数ラウンドが必要でもフラストレーションを減らす
- **大きな変更:** 一つの巨大な変更セットをレビューするより、作者に分割を依頼する

## 意見の不一致への対処（Handling Disagreements）

レビューの論争を解決するとき、この階層を適用する:

1. **技術的事実とデータ**は、意見や好みに優先する
2. **スタイルガイド**は、スタイル事項についての絶対的な権威である
3. **ソフトウェア設計**は、個人の好みではなくエンジニアリング原則で評価しなければならない
4. **コードベースの一貫性**は、全体の健全性を損なわない限り受け入れられる

**「あとで掃除します」を受け入れない。** 経験上、先送りされた掃除はめったに行われない。本物の緊急事態でない限り、提出前に掃除を要求する。周辺の問題をこの変更で対応できないなら、自己アサイン付きのバグ起票を要求する。

## レビューにおける誠実さ（Honesty in Review）

コードをレビューするとき — 自分・別のエージェント・人間のいずれが書いたものであっても:

- **ゴム印を押さない。** レビューの証跡なしの「LGTM」は誰の役にも立たない。
- **実在する問題を和らげない。** 本番を直撃するバグなのに「これは些細な懸念かもしれません」と言うのは不誠実である。
- **可能なら問題を定量化する。**「この N+1 クエリはリスト内のアイテムごとに ~50ms 追加する」は「これは遅いかもしれない」より良い。
- **明確な問題のあるアプローチには反論する。** おべっかはレビューにおける失敗モードである。実装に問題があるなら、率直にそう言い、代替案を提案する。
- **覆されることを潔く受け入れる。** 作者が十分な文脈を持ち、それでも反対するなら、その判断に従う。人ではなくコードについてコメントする — 個人攻撃をコード自体への焦点に再構成する。

## 依存関係の規律（Dependency Discipline）

コードレビューの一部は依存関係のレビューである:

**いかなる依存関係も追加する前に:**
1. 既存のスタックでこれを解決できないか?（しばしば解決できる）
2. その依存関係はどのくらい大きいか?（バンドルへの影響をチェック）
3. 活発にメンテナンスされているか?（最終コミット・オープン issue をチェック）
4. 既知の脆弱性はないか?（`npm audit`）
5. ライセンスは何か?（プロジェクトと互換でなければならない）

**ルール:** 新しい依存関係より、標準ライブラリと既存のユーティリティを優先する。あらゆる依存関係は負債である。

## レビューチェックリスト

```markdown
## レビュー: [PR/変更のタイトル]

### 文脈
- [ ] この変更が何をし、なぜするのか理解している

### 正確性
- [ ] 変更は仕様/タスク要件に合致している
- [ ] エッジケースが処理されている
- [ ] エラー経路が処理されている
- [ ] テストが変更を十分にカバーしている

### 可読性
- [ ] 名前が明確で一貫している
- [ ] ロジックが素直である
- [ ] 不要な複雑さがない

### アーキテクチャ
- [ ] 既存のパターンに従っている
- [ ] 不要な結合や依存がない
- [ ] 抽象化のレベルが適切である
- [ ] リファクタリングは複雑さを移動させるのではなく減らしている
- [ ] 共有モジュールに機能ロジックがない

### セキュリティ
- [ ] コードにシークレットがない
- [ ] 入力が境界で検証されている
- [ ] インジェクション脆弱性がない
- [ ] 認可チェックが置かれている
- [ ] 外部データソースを信頼できないものとして扱っている

### パフォーマンス
- [ ] N+1 パターンがない
- [ ] 上限のない処理がない
- [ ] リストエンドポイントにページネーションがある

### 検証
- [ ] テストがパスする
- [ ] ビルドが成功する
- [ ] 手動検証が完了している（該当する場合）

### 判定
- [ ] **承認** — マージ準備完了
- [ ] **変更を要求** — 問題に対応が必要
```

## よくある自己正当化（Common Rationalizations）

| 自己正当化 | 現実 |
|---|---|
| 「動くんだから十分だ」 | 動くが読めない・安全でない・アーキテクチャ的に誤ったコードは、複利で膨らむ負債を生む。 |
| 「自分が書いたから正しいと分かっている」 | 作者は自分自身の前提に盲目である。あらゆる変更は別の目によって利益を得る。 |
| 「あとで掃除する」 | あとは来ない。レビューが品質ゲートである — それを使え。掃除はマージ後ではなく前に要求する。 |
| 「AI 生成コードはたぶん問題ない」 | AI のコードはより少ない精査ではなく、より多くの精査を必要とする。誤っていても自信に満ち、もっともらしい。 |
| 「テストがパスするから良い」 | テストは必要だが十分ではない。アーキテクチャの問題・セキュリティ問題・可読性の懸念を捕まえない。 |
| 「リファクタリングでよりクリーンになる」 | 複雑さの移動は削減ではない。読み手が保持する概念の数が同じなら、構造は改善していない — 分岐が消えるバージョンを探せ。 |
| 「このファイルへのほんの小さな追加だ」 | 小さな diff でもファイルを健全なサイズを超えて押し上げ、無関係なフローに分岐をボルト留めする。diff のサイズではなく、結果として生じる構造を判断せよ。 |

## レッドフラグ（Red Flags）

- レビューなしでマージされる PR
- テストがパスするかだけをチェックするレビュー（他の軸を無視している）
- 実際のレビューの証跡なしの「LGTM」
- セキュリティに敏感な変更を、セキュリティに焦点を当てたレビューなしで通すこと
- 「大きすぎて適切にレビューできない」大規模 PR（分割せよ）
- バグ修正 PR にリグレッションテストがない
- 重大度ラベルのないレビューコメント — 必須と任意が不明になる
- 「あとで直す」を受け入れること — それは決して起こらない
- 読み手が保持すべき概念の数を減らさずにコードを移動させるだけのリファクタリング
- 既に大きいファイルを、分解せずにさらに肥大させる変更
- 無関係なコード経路にばらまかれた新しい条件分岐（抽象の欠如）
- 既存の正規ヘルパーを重複させる独自仕様のヘルパー、または共有モジュールに置かれた機能ロジック

## 検証（Verification）

レビュー完了後:

- [ ] すべての Critical 問題が解決されている
- [ ] すべての Required（接頭辞なし）の変更が解決されているか、正当な理由とともに明示的に先送りされている
- [ ] テストがパスする
- [ ] ビルドが成功する
- [ ] 検証ストーリーが文書化されている（何が変わり、どう検証されたか）

**推定的なブロッカー:** 以下それぞれについて、より単純な設計を表面化させ提案する; 変更が積極的に構造を悪化させる場合にのみ Required に格上げする: 複雑さを減らすのではなく移動させるリファクタリング; 分解なしにファイルをサイズ境界を超えて押し上げる変更; 共有モジュールに追加された機能ロジック; 既存の正規ヘルパーの近似重複; 不明確な不変条件を隠す暗黙のフォールバック。
