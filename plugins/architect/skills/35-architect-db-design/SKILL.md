# データベース設計

## 概要

リレーショナルデータベース設計の普遍的な原則。特定の ORM やフレームワークには依存しない。

### 最重要原則: データベースはアプリケーション層より遥かに長く生存する

これがこのスキル全体を貫く前提である。アプリケーションは書き換えられ、フレームワークは流行が変わり、言語も ORM もサービス分割も入れ替わる — その間、データベースは残り続け、すべての世代のアプリケーションを支える。10 年もののシステムで、初代のコードが残っていることは稀だが、初代のテーブルはほぼ確実に残っている。

この非対称性から、設計の時間軸と慎重さもアプリケーション層とは別物になる:

- **アプリケーションコードの設計ミスは diff で直る。スキーマの設計ミスは本番データのマイグレーションを伴い、10 倍の値段で返ってくる。** よってスキーマの意思決定には、コードの 10 倍の慎重さをかける価値がある。
- **スキーマは「今のアプリの都合」ではなく「業務の事実の構造」に従って設計する。** 今の画面・今の API のレスポンス形に合わせたテーブルは、次のアプリの世代で負債になる。事実を正しく写したテーブルは世代を越えて使える。
- **整合性はアプリケーションコードではなくデータベースの制約で守る。** アプリのバグ・並行リクエスト・手動 SQL・将来の別サービス — DB より短命なものに整合性の番を任せてはいけない。最後の砦は常に DB 自身である。

「アプリを書くついでにテーブルも作る」という姿勢を捨て、「長期に生きるデータ資産を設計し、そこに今のアプリを接続する」という姿勢で臨む。

例は標準的な SQL で示す。特定の ORM を使っている場合も、ここで示す制約・構造が最終的に DDL として発行されているかを確認する — ORM のデフォルトが正しい設計を保証してくれることはない。

## いつ使うか

- 新しいテーブルやカラムを追加するとき
- テーブル間のリレーションを設計するとき
- インデックスを追加・変更するとき
- マイグレーションを作成・実行するとき
- 既存スキーマの変更（型変更、リネーム、削除）を行うとき
- 並行更新（カウンタ・残高・在庫・予約）を扱うとき
- パフォーマンス問題（遅いクエリ、N+1）を調査するとき

## プロセス: 概念 → 論理 → 物理

いきなり CREATE TABLE を書くのは、間取りを決めずに壁を建てるのと同じ。3 層を意識して順に下りる:

### 1. 概念モデル — 業務の事実を写し取る

- **エンティティと関係を列挙する。** 何が「モノ」で、何が「モノとモノの関係」か。1:1 / 1:N / M:N を明示する。M:N は必ず中間テーブルになる — 「この関係自体に属性が生えるか?」（参加日時、役割、数量…）を先に問う。
- **業務ルールを不変条件として書き出す。**「注文は必ず 1 人の顧客に属する」「アクティブな契約はユーザーあたり 1 つ」「残高は負にならない」— これらが後で制約になる。
- **「消える」時の挙動を業務の言葉で決める。** 親が消えたら子はどうなるか。顧客が退会したら請求書は? これは技術判断ではなくプロダクト・法務要件 — 不明なら確認する。

### 2. 論理モデル — 正規化と鍵

- 各エンティティの識別子（キー）を決め、属性を配置する。
- **事実は 1 箇所にだけ書く**（正規化、後述）。同じ事実が 2 箇所にあると、片方だけ更新されて食い違う日が必ず来る。
- NULL になりうる属性それぞれについて「NULL の業務的意味」を言語化する。言えないなら NOT NULL。

### 3. 物理モデル — アクセスパターンに合わせる

- **主要な読み取りクエリを文で書き出す。**「ユーザーのタスク一覧を更新日降順で 20 件」「未処理のジョブを古い順に 1 件」— この文がそのままインデックス設計になる。
- **書き込みの頻度と並行性を見積もる。** 同じ行を複数リクエストが同時に更新するか? カウンタ・残高・在庫・座席予約は競合の火薬庫（後述の並行性制御で守る）。
- **カーディナリティ（行数の桁）を見積もる。** 100 行のマスタと 1 億行のイベントログでは、インデックス・パーティション・保持期間の判断がすべて変わる。

主要クエリと不変条件を言葉にできないテーブルは、まだ設計する準備ができていない。

## 三段階の境界システム（Three-Tier Boundary System）

### 常に行う（例外なし）

- **命名はプロジェクトの既存スキーマ・コードを読んで倣う** — 単複・snake/camel・`〜_at` などの規約を新たに発明せず、既存テーブルと同じ流儀に揃える（新規プロジェクトで前例がないときだけ規約を決め、以後それに従う）
- **NOT NULL をデフォルトにする** — nullable は「NULL が正当な業務的意味を持つ」場合にのみ選ぶ
- **外部キー制約を張る** — 「アプリで整合性を守るから不要」は幻想。孤児レコードは必ず生まれる
- **すべての外部キーに ON DELETE の挙動を明示する**（CASCADE / RESTRICT / SET NULL を意図的に選ぶ）
- **一意であるべきものに UNIQUE 制約を付ける** — アプリ側の「存在チェックして INSERT」は並行リクエストで必ず破れる
- **`created_at` / `updated_at` を全テーブルに付ける**
- **タイムスタンプはタイムゾーン付き（PostgreSQL なら `timestamptz`）または UTC 固定で保存する** — 「サーバのローカル時刻」は移設・多地域化で必ず壊れる
- **金額は浮動小数点にしない** — 整数の最小単位（円・セント）か固定小数点（DECIMAL / NUMERIC）を使う
- **すべてのスキーマ変更をマイグレーションファイルとして残し、SQL をレビューしてからコミットする**

### まず確認する（人間の承認が必要）

- 既存カラムの型変更・リネーム・削除（データ損失の可能性）
- 本番テーブルへのインデックス追加（大テーブルではロックに注意）
- 非正規化（重複データの導入）— 更新時に同期が必要になるコストを説明してから
- スキーマレスカラム（JSON 型など）の導入 — 「スキーマを決めたくない」は理由にならない（後述）
- 論理削除（soft delete）の導入 — 全クエリに削除フィルタが漏れなく付く保証が必要
- パーティショニング・シャーディング・リードレプリカなどの大規模構成
- ユーザーデータの削除方針（物理削除 / 匿名化 / 保持期間）

### 決して行わない

- **本番データベースへの手動 ALTER** — すべてマイグレーションとして残す
- **適用済みマイグレーションの書き換え** — 修正は常に新しいマイグレーションを積む
- **浮動小数点（FLOAT / DOUBLE）で金額を扱う**
- **アプリケーションコードだけで一意性・参照整合性を守る**
- **「とりあえず自由入力の文字列」で状態・種別・ロールを保存する**（typo が本番データに混入する）
- **1 つのカラムに複数の意味を持たせる**（カンマ区切りリスト、状態によって意味が変わる汎用カラム、`value1`/`value2` のような番号付きカラム）
- **EAV（Entity-Attribute-Value）でドメインの中核をモデリングする** — 型も制約も JOIN も失う

## 設計原則

### 正規化 — 事実は 1 箇所に

| 形 | 破れている状態 | 直し方 |
|---|---|---|
| 1NF | 1 カラムに複数値（カンマ区切りタグ、配列的な `phone1, phone2, phone3`） | 行に展開する（子テーブル） |
| 2NF | 複合キーの一部にだけ依存する属性（注文明細に顧客住所） | 依存先のテーブルへ移す |
| 3NF | キー以外の属性に依存する属性（社員テーブルに部署名と部署所在地） | 参照テーブルに切り出し FK で参照 |

デフォルトは 3NF。ここから崩す（非正規化する）のは、**計測して遅かった読み取り** を救うときだけで、その際は「どの書き込みが複製をどう同期するか」を必ず文書化する。

なお **履歴・スナップショットとしての複製は非正規化ではない**。注文行に「注文時点の単価・商品名」を持つのは、商品マスタの現在値とは別の事実（過去の合意）を記録する正しいモデリングである。「現在の事実」と「時点の事実」を区別せよ。

### キー設計

- **サロゲートキー（連番 / UUID / ULID 等の無意味な識別子）を主キーのデフォルトにする。** 自然キー（メールアドレス、社員番号、ISBN）は「変わらない」と信じられていても変わる — 主キーにすると変更が全 FK に波及する。
- **ただし自然キーの一意性は UNIQUE 制約として必ず残す。** サロゲートキーを導入した瞬間に自然キーの一意性を忘れるのが最頻の事故（同じメールのユーザーが 2 人できる）。
- **外部に露出する ID は推測不能にする。** 連番を URL / API に出すと、総件数と存在有無が漏れる（enumeration）。UUID / ULID / ランダム文字列を使うか、内部は連番 + 外部用の公開 ID を別カラムで持つ。
- 中間テーブルの主キーは複合キー `(a_id, b_id)` で十分なことが多い。関係自体が他から参照される・属性が育つならサロゲートキーを足す。

### NULL の規律

NULL は「値がない」ではなく「不明・未定・非該当のどれか分からない」である。3 値論理（`NULL = NULL` は真にならない、`NOT IN` にしれっと混ざると全滅する）が全クエリに漏れ出す。

- NOT NULL がデフォルト。nullable にするのは「NULL の業務的意味」を一言で言えるカラムだけ（`shipped_at IS NULL` = 未発送、は良い例）
- 「不明」と「非該当」を同じ NULL で表現しない — 区別が要るなら状態カラムを分ける
- 文字列に `''`（空文字）と NULL の両方を許すと地獄になる — どちらか一方に規約で倒す

### 制約はアプリではなく DB に置く

```sql
CREATE TABLE order_items (
  id          BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  order_id    BIGINT      NOT NULL REFERENCES orders (id) ON DELETE CASCADE,
  product_id  BIGINT      NOT NULL REFERENCES products (id) ON DELETE RESTRICT,
  quantity    INTEGER     NOT NULL CHECK (quantity > 0),
  -- 注文時点のスナップショット（商品マスタの現在値とは別の事実）
  unit_price  NUMERIC(12, 2) NOT NULL CHECK (unit_price >= 0),
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),

  UNIQUE (order_id, product_id)  -- 同一注文に同一商品は 1 行
);

-- 「アクティブな契約はユーザーあたり 1 つ」— アプリの存在チェックでは並行 INSERT で破れる。
-- 部分 unique index（対応 RDBMS の場合）や条件付き制約で DB に強制させる
CREATE UNIQUE INDEX one_active_subscription
  ON subscriptions (user_id) WHERE status = 'active';
```

「アプリでバリデーションしているから DB 制約は不要」は逆である。アプリのバリデーションは UX のため（早く・親切に断る）、DB 制約は整合性のため（最後の砦）。両方置く。

### 参照整合性と削除挙動

ON DELETE はデフォルトに任せず、テーブルごとに「親が消えたらどうあるべきか」を業務の言葉で決めてから書く:

| 子の性質 | 選択 | 例 |
|---|---|---|
| 親の所有物（親なしで存在意味がない） | `CASCADE` | ユーザーの投稿・投稿のコメント |
| 会計・監査記録（消してはいけない） | `RESTRICT` | 顧客の請求書、取引履歴 |
| 任意参照（消えても行は残るべき） | `SET NULL`（カラムは nullable に） | タスクの担当者 |

迷ったら `RESTRICT`。「消せなくて困る」は運用で直せるが、「消えて困る」は取り返せない。

**ポリモーフィック関連（`target_type` + `target_id` で複数テーブルを指す）は避ける** — FK が張れず、参照整合性が誰にも守れなくなる。テーブルを分けるか、対象ごとの nullable FK を並べて CHECK で排他にする。

### 型の選択

- **金額**: 整数の最小単位、または DECIMAL / NUMERIC。FLOAT は丸め誤差で会計が合わなくなる
- **日時**: タイムゾーン付き型か UTC 固定。「日付だけ」（誕生日・締め日）は DATE — 時刻を持たせると TZ 変換でずれる
- **状態・種別・ロール**: enum 型か参照テーブル。自由入力文字列は禁止
  - 値の集合がコードと共に変わる（デプロイで増える）→ enum
  - 値が実行時に増える・値自体に属性がある → 参照テーブル + FK
  - enum は多くの RDBMS で追加は容易だが削除・リネームが面倒 — 揮発しそうな集合は最初から参照テーブル
- **識別コード類**（電話番号・郵便番号・商品コード）: 数値に見えても文字列。先頭ゼロ・ハイフン・国際表記で数値型は破綻する
- **文字列長**: 業務上の上限があるなら型か CHECK で制約する。無制限 TEXT は巨大入力の DoS 面にもなる

### スキーマレスカラム（JSON 型）の使い所

JSON カラムが正解なのは: **スキーマが本質的に外部依存**（webhook のペイロード保存、外部 API のレスポンスキャッシュ）、または **キーが不定な設定値の袋**。

JSON カラムが逃げなのは: 自分のドメインのエンティティを「カラム設計を後回しにしたい」から押し込むとき。WHERE で使う・JOIN する・NOT NULL / 型を保証したいフィールドは、必ず通常カラムに昇格させる。

### 履歴と監査

「今の値」しか持たないテーブルは「いつ・誰が・何から何へ変えたか」に答えられない。要件に応じて:

- **スナップショット**: 取引時点の値を取引行に複製する（注文の単価）— 最も単純で十分なことが多い
- **履歴テーブル**: 変更のたびに旧行を `xxx_history` へ書く（有効期間 `valid_from` / `valid_to` 付き）
- **イベント / 監査ログ**: 変更を追記専用（append-only）のログとして残す。UPDATE / DELETE しない

「あとから履歴が欲しい」は最頻の後悔。金・権限・契約に関わるテーブルは、最初に履歴要件を確認する。

### インデックス設計

```sql
-- 主要クエリ「オーナーのタスクを状態で絞って新しい順」がそのまま複合インデックスになる
-- カラム順は「等値条件 → 範囲 / ソート」
CREATE INDEX idx_tasks_owner_status_created
  ON tasks (owner_id, status, created_at DESC);
```

- **外部キーには必ずインデックスを張る。** 多くの RDBMS（PostgreSQL 等）は FK に自動でインデックスを張らない — JOIN と親削除（CASCADE の子検索）が遅くなる（MySQL/InnoDB は自動で張る。使用中の DB の挙動を確認する）
- 複合インデックス `(a, b, c)` は `(a)` と `(a, b)` の検索もカバーする — 冗長な単一インデックスを重複させない
- インデックスは書き込みコストとストレージを食う。**対応する主要クエリを言えないインデックスは張らない**
- 遅いクエリは推測せず **実行計画（EXPLAIN）で確認する**。大テーブルのフルスキャンが典型的な赤信号

## 並行性とトランザクション

スキーマが正しくても、並行更新の設計を誤ると整合性は壊れる。

### lost update — 最頻の事故

```sql
-- 悪い例: 読み → アプリで計算 → 書き。2 リクエストが同時に走ると片方の加算が消える
SELECT balance FROM wallets WHERE user_id = $1;  -- 両方が 1000 を読む
UPDATE wallets SET balance = 1100 WHERE user_id = $1;  -- 両方が 1100 を書く（+100 が 1 回分消えた）

-- 良い例 1: 計算を DB に寄せる（アトミック更新）
UPDATE wallets SET balance = balance + 100 WHERE user_id = $1;

-- 良い例 2: 条件を WHERE に含め、更新行数で成否を判定する（残高不足なら 0 行）
UPDATE wallets SET balance = balance - 500
 WHERE user_id = $1 AND balance >= 500;

-- 良い例 3: 行ロックで直列化する（複数テーブルにまたがる更新のとき）
BEGIN;
SELECT * FROM wallets WHERE user_id = $1 FOR UPDATE;
-- ... 検証と複数の更新 ...
COMMIT;

-- 良い例 4: 楽観ロック（version カラム）。競合が稀で、失敗をユーザーに再試行させられるとき
UPDATE documents SET body = $2, version = version + 1
 WHERE id = $1 AND version = $3;  -- 0 行なら他者が先に更新した
```

### トランザクションの規律

- **複数テーブルにまたがる不変条件は 1 トランザクションで更新する**（注文と在庫、送金の両口座）
- **トランザクションは短く。** トランザクション内で外部 API を呼ばない — ロックを握ったまま数秒待つと全体が詰まる
- デフォルト分離レベル（多くは READ COMMITTED）では「読んだ値が commit までに変わらない」保証は **ない**。それが必要な箇所は `FOR UPDATE` か楽観ロックを使う
- **一意制約は最強の並行性ガードでもある。**「なければ作る」は存在チェックではなく、UNIQUE 制約 + 重複エラーの捕捉（または UPSERT）で書く
- 外部からの多重送信（決済 webhook の重複配送、二度押し）には **冪等キー**: リクエスト ID に UNIQUE を張り、2 回目を無害化する

## マイグレーション安全性

### 基本ルール

- マイグレーションは **生成 / 記述 → SQL を読む → コミット** の 3 手。`DROP` が含まれていたら、それが意図通りか必ず確認する
- 適用済みマイグレーションは編集しない。修正は常に **新しいマイグレーションを積む**
- ロールバック計画のない破壊的変更を本番に流さない

### 破壊的変更は expand-contract で

カラムのリネームや型変更を 1 ステップでやると、デプロイの瞬間に旧コードが新スキーマ（またはその逆）を踏んで落ちる。3 段階に分ける:

```
1. Expand   : 新カラムを追加（nullable / デフォルト付き）。コードは両方に書き、新を読む
2. Migrate  : バックフィルで旧 → 新へデータをコピー。バッチ化して検証する
3. Contract : 旧カラムへの参照をコードから消し、次のリリースで旧カラムを DROP
```

即時 1 ステップでよいのは: テーブルが空・機能が未リリース・開発環境のみ、のいずれかのときだけ。

### 大テーブルでのロック回避

- インデックス作成が書き込みをブロックする RDBMS では、オンライン作成手段を使う（PostgreSQL: `CREATE INDEX CONCURRENTLY`、MySQL: オンライン DDL）
- NOT NULL や CHECK の後付けは「検証を伴う追加」— 全行スキャンのロック時間を短縮する手段（PostgreSQL: `NOT VALID` → `VALIDATE CONSTRAINT`）を検討する
- バックフィルは 1 本の巨大 UPDATE にしない — 数千行ずつのバッチで、レプリケーション遅延を見ながら流す

## クエリとアプリケーション境界

- **一覧取得には必ず上限（LIMIT）とページネーションを付ける。** 無制限の全件取得は、行数が育った日に必ず事故る。深いページングは OFFSET ではなくカーソル方式（`WHERE (created_at, id) < (最後に見た値)`）— OFFSET は捨てる行も全部読む
- **ループ内クエリ（N+1）を書かない。** 一覧 + 関連は JOIN か `WHERE id IN (...)` の一括取得にまとめる。ORM 使用時は発行される SQL をログで確認する — N+1 の大半は ORM の遅延ロードが黙って生む
- **SELECT する列を絞る。** 巨大な TEXT / JSON / BLOB を含むテーブルの `SELECT *` は、不要な I/O と転送を毎回払う
- **論理削除（soft delete）は覚悟して選ぶ。** 全読み取りに削除フィルタが漏れなく付く仕組み、UNIQUE 制約との衝突（削除済みと同じ値を再登録できない → 部分 unique index）、PII 削除要求（GDPR 等）との矛盾。復元・監査要件が本当にあるときだけ。デフォルトは物理削除 + ON DELETE の設計 + 必要なら監査ログ

## DB 設計レビューチェックリスト

```markdown
### モデリング
- [ ] 主要な読み取りクエリと業務不変条件を言葉にした上で設計した
- [ ] 1:N / M:N の関係が中間テーブル含め正しく表現されている
- [ ] 事実の重複がない（あるなら「時点スナップショット」として意図的で、文書化済み）
- [ ] 履歴・監査の要件を確認した（金・権限・契約に関わるテーブルは特に）

### キーと制約
- [ ] 主キーはサロゲートキー、かつ自然キーの一意性を UNIQUE で残した
- [ ] 外部に露出する ID が推測不能
- [ ] nullable なカラムはすべて「NULL の業務的意味」を説明できる
- [ ] すべての FK に ON DELETE が明示され、業務要件と一致している
- [ ] CHECK で表現できる不変条件（正の数量、開始 < 終了）を DB に置いた

### 型
- [ ] 金額は整数最小単位 or DECIMAL（浮動小数点ではない）
- [ ] 日時はタイムゾーン付き or UTC 固定、日付のみは DATE
- [ ] 状態・種別は enum または参照テーブル（自由文字列ではない）

### インデックス
- [ ] すべての外部キーにインデックス（自動で張られない RDBMS の場合）
- [ ] 主要クエリに対応する複合インデックス（等値 → 範囲/ソートの順）
- [ ] 対応するクエリを言えないインデックスがない

### 並行性
- [ ] カウンタ・残高・在庫の更新がアトミック（read-modify-write していない）
- [ ] 「なければ作る」が UNIQUE 制約 + エラー捕捉 / UPSERT で書かれている
- [ ] 複数テーブルにまたがる不変条件がトランザクションで守られている
- [ ] トランザクション内で外部 API を呼んでいない

### マイグレーション
- [ ] SQL をレビューした（意図しない DROP がない）
- [ ] 破壊的変更は expand-contract に分割されている
- [ ] 大テーブルへの変更はロック時間を検討した

### クエリ
- [ ] 一覧取得に LIMIT + ページネーションがある
- [ ] ループ内クエリ（N+1）がない
```

## よくある言い訳（Common Rationalizations）

| 言い訳 | 現実 |
|---|---|
| 「アプリでバリデーションしてるから DB 制約は不要」 | 並行リクエスト・手動 SQL・将来の別サービスはアプリのバリデーションを通らない。最後の砦は DB。 |
| 「あとでスキーマは直せる」 | コードは diff で直るが、スキーマは本番データのマイグレーションを伴う。設計ミスは 10 倍の値段で返ってくる。 |
| 「とりあえず JSON に入れておけば柔軟」 | 柔軟なのではなく未決定なだけ。WHERE・JOIN・NOT NULL が必要になった瞬間に負債化する。 |
| 「まだデータが少ないからインデックスは不要」 | データは黙って育つ。遅くなった時にはユーザーが先に気づいている。FK と主要クエリの分は初日から張る。 |
| 「NULL 許可にしておけば安全」 | 逆。NULL は全コードに 3 値論理の分岐を強いる値。NOT NULL がデフォルト、NULL は業務的意味があるときだけ。 |
| 「メールアドレスを主キーにすれば自然」 | 自然キーは変わる。変わった日に全 FK への波及が待っている。主キーはサロゲート、自然キーは UNIQUE。 |
| 「同時に更新されることなんて滅多にない」 | 滅多に起きない競合は、テストをすり抜けて本番で起きる。lost update はコードレビューで防げず、設計でしか防げない。 |
| 「リネームくらい 1 マイグレーションでいい」 | デプロイ中は旧コードと新スキーマが同時に生きる。expand-contract に割れ。 |
| 「履歴はあとで足せばいい」 | 記録しなかった過去は復元できない。金・権限・契約のテーブルは最初に履歴要件を確認せよ。 |

## 危険信号（Red Flags）

- 外部キー制約のない `〜_id` カラム
- ほぼすべてのカラムが nullable なテーブル
- 金額・数量が FLOAT / DOUBLE
- 自由入力文字列に status / type / role が保存されている
- カンマ区切りの ID リストが 1 カラムに入っている
- `target_type` + `target_id` のポリモーフィック参照
- 自分のドメインの中核エンティティが JSON カラムに入っている
- SELECT した値をアプリで計算して UPDATE する残高・カウンタ（lost update）
- ループの中で発行されるクエリ（N+1）
- LIMIT のない一覧取得
- マイグレーション SQL の中の、誰も説明できない `DROP COLUMN`
- トランザクション内の外部 API 呼び出し

## 検証（Verification）

スキーマ変更・マイグレーションを実装した後:

- [ ] マイグレーションがクリーンに通り、発行される SQL をレビューした
- [ ] ゼロから再構築できる（全マイグレーション + seed の再現性）
- [ ] UNIQUE / CHECK / FK 制約に **違反するデータの挿入が実際に失敗する** ことをテストで確認した
- [ ] ON DELETE の挙動（親削除 → 子の CASCADE / RESTRICT）を実データで確認した
- [ ] 並行更新の箇所は、同時実行のテスト（または少なくとも机上の 2 リクエスト同時トレース）で lost update がないことを確認した
- [ ] 主要クエリの実行計画を確認し、意図したインデックスが使われている（大テーブル想定がある場合）
- [ ] 既存データがある環境なら、バックフィル後の新旧カラムの値が一致している
