# DB 設計チェックリスト

リレーショナルデータベース設計のクイックリファレンス。`35-architect-db-design` スキルと併せて使用する。

大前提: **データベースはアプリケーション層より遥かに長く生存する。** 以下のすべての項目は「今のアプリに都合がよいか」ではなく「10 年後のデータ資産として正しいか」で判定する。

## モデリング（まずここから）

テーブルを書く前に言葉にする:

- [ ] エンティティと関係（1:1 / 1:N / M:N）を列挙した
- [ ] 業務の不変条件を書き出した（「アクティブな契約はユーザーあたり 1 つ」「残高は負にならない」）
- [ ] 主要な読み取りクエリを文で書き出した（「〜を〜順で N 件」）
- [ ] 同一行への並行書き込みの有無を確認した（カウンタ・残高・在庫・予約）
- [ ] 各テーブルの行数の桁（100 / 100万 / 1億）を見積もった
- [ ] 親削除時の子の挙動を業務の言葉で決めた（技術判断ではなくプロダクト・法務要件）
- [ ] M:N の中間テーブルに将来属性が生えるかを検討した
- [ ] 「現在の事実」と「時点の事実」（スナップショット）を区別した
- [ ] 今の画面・API のレスポンス形にテーブルを合わせていない（事実の構造に合わせた）

## 命名

- [ ] **プロジェクトの既存スキーマ・コードを読み、その規約に倣った**（単複、snake/camel、`〜_at` / `〜_flag` などの流儀を新たに発明しない）
- [ ] 新規プロジェクトで前例がない場合のみ規約を決め、最初のテーブルから一貫させた
- [ ] 使用中の RDBMS の予約語と衝突していない
- [ ] 同じ概念に別名を与えていない（`customer` と `client` の混在など）
- [ ] 曖昧な省略形・番号付きカラム（`value1`, `value2`）がない

## キー設計

- [ ] 主キーはサロゲートキー（自然キーは変わる — 変わった日に全 FK へ波及する）
- [ ] サロゲートキー導入後も、自然キー（メール、社員番号等）の一意性を UNIQUE で残した
- [ ] 外部に露出する ID は推測不能（連番の露出は総件数と存在有無を漏らす）
- [ ] 中間テーブルの主キーは複合キーで足りるか、他から参照される・属性が育つ見込みがあるかを判断した

## 型の選択

- [ ] 金額: 整数の最小単位（円 / セント）or DECIMAL / NUMERIC — 浮動小数点禁止
- [ ] 日時: タイムゾーン付き型 or UTC 固定。「サーバのローカル時刻」に依存しない
- [ ] 日付のみ（誕生日・締め日）: DATE — 時刻を持たせると TZ 変換でずれる
- [ ] 状態・種別・ロール: enum または参照テーブル — 自由入力文字列禁止
- [ ] enum は削除・リネームが面倒（RDBMS による）— 揮発しそうな値の集合は参照テーブル
- [ ] 識別コード類（電話番号・郵便番号・商品コード）は文字列（先頭ゼロ・ハイフン・国際表記）
- [ ] 業務上の長さ上限がある文字列は型か CHECK で制約（無制限 TEXT は巨大入力の面にもなる）
- [ ] 真に連続な値（割合・係数・座標）のみ浮動小数点を許可

## 制約（Constraints）

- [ ] NOT NULL がデフォルト。nullable は「NULL の業務的意味」を一言で言えるカラムのみ
- [ ] 「不明」と「非該当」を同じ NULL で表現していない
- [ ] 文字列に空文字と NULL の両方を許していない（どちらかに規約で倒す）
- [ ] 一意であるべきものすべてに UNIQUE 制約（アプリの存在チェックは並行で破れる）
- [ ] 「アクティブな行だけ一意」等の条件付き一意性を DB で強制した（部分 unique index 等）
- [ ] 不変条件を CHECK に（`quantity > 0`, `starts_at < ends_at`）
- [ ] デフォルト値は DB 側に置き、アプリ生成と二重管理していない
- [ ] ORM 使用時: 上記が最終的な DDL に実際に含まれていることを確認した

## リレーション設計

- [ ] すべての `〜_id` カラムに外部キー制約がある
- [ ] すべての FK に ON DELETE を明示した:
  - 所有物（ユーザーの投稿・コメント）→ `CASCADE`
  - 会計・監査記録（請求書・取引履歴）→ `RESTRICT`
  - 任意参照（担当者など）→ `SET NULL`（カラムは nullable に）
  - 迷ったら `RESTRICT`（消せなくて困る < 消えて困る）
- [ ] ポリモーフィック関連（`target_type` + `target_id`）を避けた — FK が張れない。テーブル分割か、対象ごとの nullable FK + CHECK で排他に
- [ ] 自己参照（ツリー構造）は深さの上限と取得方法（再帰 CTE 等）を先に検討した

## 履歴と監査

- [ ] 金・権限・契約に関わるテーブルは、履歴要件を最初に確認した（記録しなかった過去は復元できない）
- [ ] 取引時点の値（単価・商品名等）は取引行にスナップショットした — マスタの現在値への参照と混同していない
- [ ] 変更履歴が必要なテーブルは方式を選んだ: 履歴テーブル（有効期間付き）or 追記専用の監査ログ
- [ ] 追記専用テーブルに UPDATE / DELETE の経路がない

## インデックス設計

- [ ] すべての外部キーにインデックス（FK に自動でインデックスを張らない RDBMS がある — 使用中の DB の挙動を確認）
- [ ] 主要クエリごとに複合インデックス — カラム順は「等値 → 範囲 / ソート」
- [ ] 複合 `(a, b)` があるとき、冗長な `(a)` 単独インデックスを重複させていない
- [ ] 部分インデックス（対応 RDBMS の場合）でホットな絞り込みを軽くできないか検討した
- [ ] 対応する主要クエリを言えないインデックスがない（書き込みコスト）
- [ ] 大文字小文字を区別しない一意性（email 等）の実現方法を確認した（照合順序、式インデックス等 — RDBMS で異なる）
- [ ] 大テーブル想定のクエリは実行計画（EXPLAIN）でフルスキャンになっていないことを確認した

## 並行性とトランザクション

- [ ] カウンタ・残高・在庫を read-modify-write していない（`SET x = x + ?` のアトミック更新）
- [ ] 条件付き更新（残高不足なら失敗）は条件を WHERE に含め、更新行数で成否を判定
- [ ] 「なければ作る」は UNIQUE 制約 + 重複エラー捕捉 / UPSERT（SELECT してから INSERT しない）
- [ ] 複数テーブルにまたがる不変条件は 1 トランザクションで更新
- [ ] 複数行・複数テーブルの読み → 判断 → 更新は行ロック（`SELECT ... FOR UPDATE`）か楽観ロック（version カラム）
- [ ] トランザクション内で外部 API を呼んでいない（ロックを握ったまま待たない）
- [ ] 外部からの多重送信（webhook 重複配送、二度押し）に冪等キー（リクエスト ID に UNIQUE）
- [ ] デフォルト分離レベルで「読んだ値が commit まで変わらない」保証はないことを前提にした

## マイグレーション安全性

- [ ] すべてのスキーマ変更がマイグレーションファイルになっている（本番への手動 ALTER なし）
- [ ] 発行される SQL を読んだ（意図しない `DROP` / `ALTER` がない）
- [ ] 適用済みマイグレーションを編集していない（修正は新しいマイグレーションで）
- [ ] リネーム・型変更は expand-contract（追加 → バックフィル → 削除）に分割
- [ ] デプロイ中に旧コードと新スキーマが同時に生きても壊れない
- [ ] 大テーブルへのインデックス作成・制約追加はロック時間を検討した（オンライン作成手段は RDBMS で異なる）
- [ ] バックフィルはバッチ化（1 本の巨大 UPDATE で全行ロックしない）
- [ ] ゼロから再構築できる（全マイグレーション + seed の再現性）
- [ ] 破壊的変更にロールバック計画がある

## クエリ設計

- [ ] 一覧取得に LIMIT + ページネーション（深いページは OFFSET ではなくカーソル方式）
- [ ] ループ内クエリ（N+1）がない — JOIN か `IN (...)` の一括取得。ORM 使用時は発行 SQL をログで確認
- [ ] 巨大な TEXT / JSON / BLOB を含むテーブルで `SELECT *` していない
- [ ] 集計の重いクエリはオンラインの画面から都度実行していない（集計テーブル・キャッシュ・非同期化を検討）

## データライフサイクル

- [ ] ユーザー退会時の扱い（物理削除 / 匿名化 / 保持）を決めた — PII は `35-architect-security` を併用
- [ ] 論理削除を使うなら: 全読み取りに削除フィルタを一律適用する仕組み + UNIQUE 制約との衝突対策（部分 unique index）+ PII 削除要求との整合
- [ ] 無限に育つテーブル（ログ・イベント・通知）に保持期間 / アーカイブ方針がある
- [ ] スキーマレスカラム（JSON）は外部由来データ・不定キーの設定値に限定 — WHERE / JOIN / NOT NULL が要るフィールドは通常カラムに昇格
