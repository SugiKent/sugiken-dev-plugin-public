# オーケストレーションパターン

このリポジトリが推奨するエージェントオーケストレーションパターンと、避けるべきアンチパターンのリファレンスカタログ。複数の persona を協調させる新しいスラッシュコマンドを追加する前、あるいは既存の persona を「ラップ」する新しい persona を導入する前に、これを読むこと。

統べるルール: **ユーザー（またはスラッシュコマンド）がオーケストレーターである。persona は他の persona を呼び出さない。** スキルは persona のワークフロー内の必須のホップ（経由地）である。

---

## 推奨パターン

### 1. 直接呼び出し（オーケストレーションなし）

単一 persona、単一の観点、単一の成果物。デフォルトかつ最も安価な選択肢。

```
user → code-reviewer → report → user
```

**使うとき:** 作業が 1 つの成果物に対する 1 つの観点であり、1 文で記述できるとき。

**例:**
- 「この PR をレビューして」 → `code-reviewer`
- 「`auth.ts` のセキュリティ問題を見つけて」 → `security-auditor`
- 「checkout フローに不足しているテストは?」 → `test-engineer`

**コスト:** 1 往復。オーケストレーション付きパターンと常に比較すべきベースライン。

---

### 2. 単一 persona のスラッシュコマンド

1 つの persona をプロジェクトのスキルでラップするスラッシュコマンド。毎回ワークフローを説明し直す手間をユーザーから省く。

```
/review → code-reviewer（code-review-and-quality スキル付き） → report
```

**使うとき:** 同じ単一 persona の呼び出しが、同じセットアップで繰り返し起きるとき。

**このリポジトリの例:** `/review`, `/test`, `/code-simplify`。

**コスト:** 直接呼び出しと同じ。スラッシュコマンドは単に保存されたプロンプトにすぎない。

**アンチシグナル:** スラッシュコマンドの本体が大部分「どの persona を呼ぶか決める」だけなら、それは削除してユーザーに persona を直接呼ばせよ。

---

### 3. 並列ファンアウト + マージ

複数の persona が同じ入力に対して並行して動作し、それぞれが独立したレポートを生む。マージステップ（メインエージェントのコンテキスト内）がそれらを単一の意思決定へ統合する。

```
                    ┌─→ code-reviewer    ─┐
/ship → fan out  ───┼─→ security-auditor ─┤→ merge → go/no-go + rollback
                    └─→ test-engineer    ─┘
```

**使うとき:**
- サブタスクが真に独立している（共有可変状態なし、順序依存なし）
- 各サブエージェントが自前のコンテキストウィンドウから恩恵を受ける
- マージステップがメインコンテキストに収まるほど小さい
- 実時間（wall-clock）のレイテンシが重要

**このリポジトリの例:** `/ship`。

**コスト:** N 個の並列サブエージェントコンテキスト + 1 マージターン。直接呼び出しより高いが、実時間は速く、各サブエージェントが単一の観点に集中し続けるためレポートはより良くなる。

**このパターンを採用する前の検証チェックリスト:**
- [ ] すべてのサブエージェントを順序問題なしに同時実行できるか?
- [ ] 各 persona は、同じ指摘を別角度から出すのではなく、異なる *種類* の指摘を生むか?
- [ ] マージステップはメインエージェントの残りコンテキストに収まるか?
- [ ] ユーザーの待ち時間は、並列化が実際に体感できるほど長いか?

いずれかが「No」なら、直接呼び出しか単一 persona コマンドにフォールバックせよ。

---

### 4. ユーザー駆動のスラッシュコマンドによる逐次パイプライン

ユーザーが定義された順序でスラッシュコマンドを実行し、ステップ間でコンテキスト（またはコミット履歴）を引き継ぐ。オーケストレーターエージェントは存在しない — ユーザーがオーケストレーターである。

```
user runs:  /spec  →  /plan  →  /build  →  /test  →  /review  →  /ship
```

**使うとき:** ワークフローに依存関係があり（各ステップが前ステップの出力を必要とする）、ステップ間の人間の判断が価値を加えるとき。

**このリポジトリの例:** DEFINE → PLAN → BUILD → VERIFY → REVIEW → SHIP のライフサイクル全体。

**コスト:** ステップごとに 1 サブエージェントコンテキスト。オーケストレーターエージェントが存在しないため、オーケストレーション層は無料。

**なぜ自動化しないか:** LLM「ライフサイクルオーケストレーター」は (a) ハンドオフのために要約せざるを得ずステップ間のニュアンスを失い、(b) 誤った方向の作業を早期に捕まえる人間のチェックポイントをスキップし、(c) 言い換えターンによってトークンコストを倍増させる。

---

### 5. リサーチの隔離（コンテキスト保全）

タスクが、メインコンテキストを汚染すべきでない大量の資料の読み込みを要するとき、ダイジェストだけを返すリサーチサブエージェントを起動する。

```
main agent → research sub-agent（50 ファイルを読む） → digest → main agent が続行
```

**使うとき:**
- メインセッションが下流のタスクに集中し続ける必要がある
- 調査結果が、それが消費する入力よりずっと小さい
- 後でメインエージェントが考える余地を持つことで意思決定の質が高まる

**例:** 「monorepo 全体でこの deprecated API のすべての呼び出し箇所を見つけて」「これら 30 個の ADR がキャッシュについて何を言っているか要約して」。

**コスト:** 1 つの隔離されたサブエージェントコンテキスト。代替案がメインコンテキストに何百ものファイルを読み込むことであるなら、常に見合う。

**Claude Code では、カスタムのリサーチ persona を定義するのではなく組み込みの `Explore` subagent を使え。** `Explore` は Haiku で動き、write/edit ツールを拒否され、このパターンのために専用設計されている。`Explore` が合わない場合（例: モデルが推測しないドメイン固有のシステムプロンプトが必要）に限ってカスタムのリサーチ subagent を定義する。

## Claude Code 互換性

このカタログはハーネス非依存だが、ほとんどの読者は Claude Code 上で実行するだろう。各パターンが Claude Code のプリミティブにどう対応するか、そしてプラットフォームがどこで我々のルールを代わりに強制してくれるかを示す。

### persona はどこに置くか

プラグイン subagent はプラグインルートの `agents/` に置く。このリポジトリはプラグイン（`.claude-plugin/plugin.json`）なので、`agents/code-reviewer.md`・`agents/security-auditor.md`・`agents/test-engineer.md` はプラグイン有効時に自動検出される。パス設定は不要。

### Subagent と Agent Team

Claude Code には 2 つの並列化プリミティブがある。パターン 3（並列ファンアウト + マージ）は **subagent** に対応する。互いに会話するチームメイトが必要なら、代わりに **Agent Team** を使え。

| | Subagent | Agent Team |
|--|-----------|-------------|
| 協調 | メインエージェントがファンアウトし、サブエージェントは報告するだけ | チームメイトが互いにメッセージし、タスクリストを共有 |
| コンテキスト | subagent ごとに自前のコンテキストウィンドウ | チームメイトごとに自前のコンテキストウィンドウ |
| いつ使うか | レポートを生む独立タスク | 議論を要する協働作業 |
| ステータス | 安定 | 実験的 — `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` が必要 |
| コスト | 低い | 高い — 各チームメイトが別個の Claude インスタンス |

**このリポジトリの persona は両モードで動作する。** subagent として起動されたとき（例: `/ship` による）は、メインセッションに指摘を報告する。チームメイトとして起動されたとき（`security-auditor agent type を使ってチームメイトを起動…`）は、互いの指摘を直接ぶつけ合える。persona の定義は同じで、起動コンテキストだけが変わる。

1 つ微妙な点: persona の `skills` および `mcpServers` frontmatter フィールドは、subagent として動くときは尊重されるが **チームメイトとして動くときは無視される** — チームメイトは通常のセッションと同様、プロジェクトとユーザー設定からスキルと MCP サーバをロードする。persona が特定のスキルや MCP サーバのロードに依存するなら、両モードで利用可能になるようセッションレベルで設定せよ。

### プラットフォームが強制するルール

このカタログの 2 つのルールは単なる慣習ではない — Claude Code が強制する:

- **「subagent は他の subagent を起動できない」**（ドキュメントそのまま）。アンチパターン B（persona が persona を呼ぶ）とアンチパターン D（深い persona ツリー）は Claude Code 上では構造的に存在しえない。
- **「ネストしたチームはない」** — チームメイトは自分のチームを起動できない。同じアンチパターンがチームレベルでもブロックされる。

つまり、貢献者が誤ってアンチパターンを構築する心配なしにこのカタログのパターンを採用できる。それらは単にロードに失敗するだけだ。

### 知っておくべき組み込み subagent

カスタム subagent を定義する前に、以下のいずれかがその役割をカバーしないか確認せよ:

| 組み込み | 目的 |
|----------|---------|
| `Explore` | 読み取り専用のコードベース検索・分析。パターン 5（リサーチ隔離）にはこれを使う。 |
| `Plan` | plan モード中の読み取り専用リサーチ。 |
| `general-purpose` | 探索と変更の両方を要する多段タスク。 |

これらを再定義するな。専門 persona（code-reviewer, security-auditor, test-engineer）をこれらの上に重ねよ。

### プラグインエージェントの frontmatter 制約

プラグイン subagent は `hooks`・`mcpServers`・`permissionMode` frontmatter フィールドを **サポートしない** — これらは黙って無視される。将来の persona がそれらを必要とするなら、ユーザーはファイルを `.claude/agents/` または `~/.claude/agents/` にコピーする必要がある。

プラグインエージェントで動作する フィールドは: `name`, `description`, `tools`, `disallowedTools`, `model`, `maxTurns`, `skills`, `memory`, `background`, `effort`, `isolation`, `color`, `initialPrompt`。コスト最適化したいなら persona ごとに `model` を使え（例: `test-engineer` のカバレッジスキャンに Haiku、`code-reviewer` に Sonnet、`security-auditor` に Opus）。

### 複数の subagent を並列で起動する

Claude Code では、並列ファンアウト（パターン 3）には **1 つのアシスタントターンで複数の Agent ツール呼び出しを発行する** ことが必要。逐次のターンは実行を直列化する。`/ship` はこれを明示的に呼び出している。新しいオーケストレーターコマンドも同様にすべき。

## 実例: 競合仮説デバッグのための Agent Team

この例は、`/ship` の subagent ファンアウトの代わりに **Agent Team** に手を伸ばすときを示す。2 つのパターンは遠目には似ている — どちらも同じ 3 つの persona を起動する — が、価値の源が異なる。

### シナリオ

> *checkout が時々、完了前に約 30 秒ハングする。およそ 50 セッションに 1 回起きる。ログにエラーはない。先週のリリース後に始まった。*

もっともらしい根本原因（互いに排他的で、すべて症状に合致する）:

1. 新しい payment-confirmation フローの race condition
2. 時々遅い同期ネットワーク呼び出しに落ちる auth チェック
3. cart サイズに応じてスケールするクエリの index 欠落
4. SDK が黙ってリトライしてからタイムアウトする、不安定なサードパーティ API

単一エージェントは最初のもっともらしい説を選んで調査を止める。`/ship` 風の subagent ファンアウトは各 persona が独立に報告する — だがそれらのレポートは決して出会わないので、誤った説を排除できない。

これはまさに Agent Teams のドキュメントが述べるケースだ: *「互いを反証しようと能動的に試みる複数の独立した調査者がいれば、生き残った説が実際の根本原因である可能性ははるかに高い。」*

### なぜこれが `/ship` の仕事 *ではない* のか

| | `/ship`（subagent） | Agent Team |
|--|--------------------|-------------|
| サブエージェントが見るもの | 同じ diff、異なるレンズ | 共有タスクリスト、互いのメッセージ |
| 出力 | 3 つの独立レポート → 1 マージ | 敵対的議論 → 合意された根本原因 |
| 適するとき | 既知の成果物への判定が欲しい | 仮説の中から成果物を *見つけ* たい |

`/ship` は判定。Agent Team は調査。

### セットアップ（環境ごとに一度）

Agent Teams は実験的。`~/.claude/settings.json` に:

```json
{
  "env": {
    "CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS": "1"
  }
}
```

Claude Code v2.1.32 以降が必要。このリポジトリの persona は自動的に拾われる — 手で書くチーム設定ファイルはない。

### トリガープロンプト

リードセッションに自然言語で打ち込む:

```
ユーザーから、先週のリリース後に checkout が断続的に約 30 秒ハングすると
報告がある。ログにエラーはない。

これを競合仮説でデバッグするための agent team を作成して。既存の
agent type を使って 3 つのチームメイトを起動して:

  - code-reviewer  — checkout コードパス内の race condition と
                     ブロッキング呼び出しを調査
  - security-auditor — auth チェック・セッション処理・最近追加された
                       同期ネットワーク呼び出しを調査
  - test-engineer  — 仮説を区別できるテストを提案し、checkout の
                     カバレッジギャップをチェック

互いに直接メッセージし合って、各自の説に異議を唱えさせて。合意が
形成されるにつれ findings を更新して。2 人のチームメイトが他を反証
できると合意したときだけ収束して。
```

リードは既存の persona 名を参照して 3 つのチームメイトを起動する。persona 本体は各チームメイトのシステムプロンプトに追加指示として **付加** され（リードがインストールするチーム協調指示の上に）、上のトリガープロンプトが彼らのタスクになる。

### 何が起きるか

1. 各チームメイトは自前のコンテキストウィンドウで動き、自分のレンズからコードベースを探索する。
2. チームメイトは `message` を使って findings を互いに直接送る。リードが中継する必要はない。
3. 共有タスクリストが誰が何を調査中かを示す — `Ctrl+T`（インプロセスモード）または tmux ペイン（split モード）でいつでも見える。
4. `code-reviewer` が逐次であるべき `Promise.all` を見つけると、`security-auditor` にメッセージして auth 呼び出しが race の一部でないか確認する。`security-auditor` はチェックして返答する — race が本当の問題だと確認するか、反証を生むか。
5. `test-engineer` は勝ちつつある説に対して焦点を絞った統合テストを提案し、チームは合意を宣言する前にそれで検証する。
6. リードは収束した findings を統合してあなたに提示する。

`Shift+Down` で巡回して打ち込めば、どのチームメイトでもいつでも割り込める — 誤った道に進んだ調査者をリダイレクトするのに便利。

### いつ片付けるか

調査が根本原因に着地したら、リードに伝える:

```
Clean up the team
```

片付けは常にリード経由で、チームメイト経由ではない（ドキュメント曰く: チームメイトは片付けに必要な完全なチームコンテキストを欠く）。

### コストの目安

3 つの Sonnet チームメイトを約 10〜15 分の調査で走らせると、`/ship` が subagent として起動する同じ 3 persona よりも目立ってコストがかかる。その正当化は *結論の質* である — 誤った修正が高くつく本番デバッグでは、追加トークンは安い買い物だ。ルーチンの PR レビューなら `/ship` を使え。

### このシナリオでのアンチパターン

これを subagent をファンアウトする `/debug` スラッシュコマンドとして **作り直すな**。subagent は互いにメッセージできない — このパターンを機能させる敵対的議論を失う。あるワークフローが繰り返し出てくるなら、上のトリガープロンプトを、subagent を誤用するスラッシュコマンドでラップする代わりにスニペットとして文書化せよ。

### Agent Team を使わない *とき*

- 既知の diff への本番向け判定 → `/ship`（subagent）を使う。
- 1 つの成果物への 1 つの専門観点 → persona の直接呼び出し。
- 逐次ライフサイクル（spec → plan → build） → ユーザー駆動のスラッシュコマンド（パターン 4）。
- 小さなダイジェストを伴う読み込み中心のリサーチ → 組み込みの `Explore` subagent。

正しい答えを出すためにチームメイトが互いに異議を唱える **必要がある** ときだけ Agent Team に手を伸ばせ。

## アンチパターン

### A. ルーター persona（「メタオーケストレーター」）

どの他 persona を呼ぶか決めるのが仕事の persona。

```
/work → router-persona → 「これはレビューが必要」 → code-reviewer → router（言い換え） → user
```

**なぜ失敗するか:**
- ドメイン価値のない純粋なルーティング層
- 2 つの言い換えホップを加える → 情報損失 + 約 2 倍のトークンコスト
- ユーザーは既にレビューが欲しいと分かっていた。`/review` を直接呼べたはず
- スラッシュコマンドと `AGENTS.md` の意図マッピングが既にやっている作業を複製する

**代わりにすること:** スラッシュコマンドを追加・改善する。意図 → コマンドのマッピングを `AGENTS.md` に文書化する。

---

### B. 別の persona を呼ぶ persona

auth コードを見ると内部で `security-auditor` を呼び出す `code-reviewer`。

**なぜ失敗するか:**
- persona は単一の観点を生むよう設計されている。連鎖はそれを台無しにする
- 呼び出し側 persona が渡す要約は、呼ばれた persona が必要とするコンテキストを失う
- 失敗モードが増える（どちらの persona の出力フォーマットが勝つ? 誰のルールが適用される?）
- ユーザーからコストを隠す

**代わりにすること:** 呼び出し側 persona がレポートでフォローアップの監査を *推奨* する。2 回目のパスはユーザーかスラッシュコマンドが走らせる。

---

### C. 言い換える逐次オーケストレーター

ユーザーの代わりに `/spec`、次に `/plan`、次に `/build` …を呼ぶエージェント。

**なぜ失敗するか:**
- 誤った方向の作業を捕まえる人間のチェックポイントを失う
- 各ハンドオフがコンテキストを要約する — 長いパイプラインで累積するドリフト
- トークンコストを倍増させる: ステップごとにオーケストレーターターン + サブエージェントターン
- 判断が最も重要な、まさにその地点でユーザーの主体性を奪う

**代わりにすること:** ユーザーをオーケストレーターのまま保つ。推奨シーケンスを `README.md` に文書化し、ユーザーに起動させる。

---

### D. 深い persona ツリー

`/ship` が `pre-ship-coordinator` を呼び、それが `quality-coordinator` を呼び、それが `code-reviewer` を呼ぶ。

**なぜ失敗するか:**
- 各層が決定価値なしにレイテンシとトークンを加える
- デバッグが多層の調査になる
- 末端の persona が複数の要約ステップでコンテキストを失う

**代わりにすること:** オーケストレーションの深さを高々 1 に保つ（スラッシュコマンド → persona）。マージはメインエージェントで起きる。

---

## 意思決定フロー

新しいオーケストレーション付きワークフローを検討するとき、このフローを歩め:

```
作業は 1 つの成果物への 1 つの観点か?
├── Yes → 直接呼び出し。終わり。
└── No  → 同じ構成が繰り返されるか?
         ├── No  → 直接呼び出し、その場限り。終わり。
         └── Yes → サブタスクは独立か?
                  ├── No  → ユーザーが走らせる逐次スラッシュコマンド（パターン 4）。
                  └── Yes → 並列ファンアウト + マージ（パターン 3）。
                           上のチェックリストで検証する。
                           いずれかが不合格 → 単一 persona コマンド（パターン 2）にフォールバック。
```

## いつこのカタログに新パターンを追加するか

新エントリの追加は、以下を満たした後に限る:

1. そのパターンを実作業で少なくとも 2 回使った
2. それを実証するこのリポジトリ内の具体的な成果物を名指しできる
3. なぜ既存パターンでは機能しなかったかを説明できる
4. そのアンチパターンの影（人々が代わりに誤って作るもの）を記述できる

時期尚早なカタログエントリは、誰も従わない理想論のドキュメントになる。
