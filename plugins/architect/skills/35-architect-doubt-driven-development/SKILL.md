# 疑い駆動開発（Doubt-Driven Development）

## 概要

自信ありげな答えは、正しい答えではない。長いセッションはコンテキストを蓄積し、誰も気づかないうちに「仮定」を静かに「事実」へと変えてしまう。疑い駆動開発とは、自明でない出力が確定する前に、**承認ではなく反証** に偏らせたフレッシュコンテキストのレビュアーを必ず立ち上げる、という規律である。

これは `/review` ではない。`/review` は完成した成果物に対する判定だ。本スキルは進行中の姿勢である — 自明でない意思決定は、軌道修正がまだ安いうちに反対尋問にかけられる。

## いつ使うか

意思決定が **自明でない（non-trivial）** のは、以下のうち少なくとも 1 つが当てはまるときである:

- 分岐ロジックを導入または変更する
- モジュールまたはサービスの境界をまたぐ
- 型システムやコンパイラが検証できない性質を主張する（スレッドセーフ性、冪等性、順序保証、不変条件）
- その正しさが、将来の読み手には見えないコンテキストに依存する
- ブラストradius（影響範囲）が取り返しのつかないものである（本番デプロイ、データマイグレーション、公開 API の変更）

以下の場面でこのスキルを適用する:

- 不確実性のもとでアーキテクチャ上の意思決定をしようとしているとき
- 自明でないコードをコミットしようとしているとき
- 自明でない事実を主張しようとしているとき（「これは安全だ」「これはスケールする」「これは仕様に合っている」）
- 自分が完全には理解していないコードを扱っているとき

**使わない場合:**

- 機械的な操作（リネーム、フォーマット、ファイル移動）
- 明快で曖昧さのないユーザー指示に従うだけのとき
- 既存コードの読解や要約
- 正しさが自明な 1 行の変更
- 純粋なツーリング操作（テスト実行、ファイル一覧表示）
- ユーザーが明示的に検証より速度を求めたとき

すべてのキーストロークを疑えば、何も出荷できない。本スキルは上で定義した自明でない意思決定にのみ適用される。

## ロードに関する制約

このスキルは **メインセッションのオーケストレーター** 向けに設計されている。そこでは Step 3（後述の DOUBT）がフレッシュコンテキストのレビュアーを起動できる。

- **このスキルを persona の `skills:` frontmatter に追加してはならない。** Step 3 に従う persona は別の persona を起動することになり、これは `references/orchestration-patterns.md` が明示的に禁じるオーケストレーションのアンチパターン（「persona は他の persona を呼び出さない」）に該当する。
- **subagent コンテキストの中からこのスキルを適用しようとしている自分に気づいた場合**（Claude Code が subagent のネストした起動を禁じている場面）: 推奨される対応は、疑い駆動はネストして実行できないことをユーザーに表面化し、メインセッションに処理を委ねることである。**最後の手段としてのみ**、劣化した自己問答フォールバックが存在する — ARTIFACT + CONTRACT を、それまでの自分の推論から心理的に強く切り離した新しいセルフプロンプトとして書き直し、Step 1〜5 を歩む。これは **フレッシュコンテキストレビューではない**（自分のコンテキストを自分で持ち込んでいる）ため、結果を「劣化版」として明示し、ユーザーに到達できる限りはエスカレーションを優先する。

## プロセス

スキルを適用するときは、このチェックリストをコピーする:

```
Doubt cycle:
- [ ] Step 1: CLAIM — クレームと「なぜ重要か」を書いた
- [ ] Step 2: EXTRACT — artifact + contract を切り出し、推論を剥がした
- [ ] Step 3: DOUBT — 敵対的プロンプトでフレッシュコンテキストレビュアーを起動した
- [ ] Step 4: RECONCILE — すべての指摘を artifact 本文と突き合わせて分類した
- [ ] Step 5: STOP — 停止条件を満たした（些末な指摘のみ・3 サイクル・ユーザーの上書き）
```

### Step 1: CLAIM — 何が確定しようとしているかを表面化する

意思決定を 2〜3 行で命名する:

```
CLAIM: 「新しいキャッシュ層は、仕様に記述された read-heavy な
        ワークロードのもとでスレッドセーフである。」
WHY THIS MATTERS: ここでの race はユーザーデータを破壊し、
                  QA で検出するのが難しい。
```

これをそれだけコンパクトに書けないなら、それは意思決定ではなく「雰囲気（vibe）」である。精査する前に表面化せよ。

### Step 2: EXTRACT — レビュー可能な最小単位

フレッシュコンテキストのレビュアーが必要とするのは **artifact** と **contract** であって、そこに至る旅路ではない。

- コード: diff または関数 — ファイル全体ではない
- 意思決定: 提案を 3〜5 文 + それが満たすべき制約
- 主張: クレーム + それを支えるとされる証拠（Step 1 の CLAIM ブロック — 精査対象であるオーケストレーターの仮説 — とは別物として区別する）

自分の推論を剥がせ。結論を渡せば、返ってくるのは自分の結論への追認だ。単位は、レビュアーが 1 回読んで頭の中に保持できるほど小さくなければならない — 500 行の PR なら、まず分解せよ。

### Step 3: DOUBT — フレッシュコンテキストのレビュアーを起動する

レビュアーのプロンプトは **敵対的でなければならない**。フレーミングが答えを決める。

```
敵対的レビュー。この artifact の何が間違っているかを見つけよ。
著者は自信過剰だと仮定せよ。以下を探せ:
- 明示されていない仮定
- ハンドリングされていないエッジケース
- 隠れた結合や共有状態
- contract が破られうる経路
- これが破壊しうる既存の慣習
- 想定外の入力に対する失敗モード

検証（validate）するな。要約するな。問題を見つけよ。
あるいは、徹底的に調べても見つからないと明示的に述べよ。

ARTIFACT: <artifact を貼る>
CONTRACT: <contract を貼る>
```

**ARTIFACT + CONTRACT のみ渡せ。CLAIM は渡すな。** レビュアーに自分の結論を渡すと、同意へとバイアスがかかる。レビュアーは、artifact が contract を満たすかどうかを独立に判断しなければならない。

Claude Code では、組み込みの汎用 subagent（`general-purpose` / `Explore` など）が設計上、隔離されたコンテキストで始まるため、フレッシュコンテキストのレビュアーとしてここで利用できる。

**上記の敵対的プロンプトは、レビュアーのデフォルトの応答フォーマットに優先する。** バランスのとれた判定（強みと弱みの両方）を返すよう構成されたレビュアーもあるが、疑い駆動が必要とするのは「問題のみ」の出力である。デフォルトを上書きするため、敵対的プロンプトを呼び出しにそのまま貼り付けよ。応答フォーマットをきれいに上書きできない場合は、敵対的プロンプトを与えた汎用 subagent にフォールバックする。

#### クロスモデルエスカレーション

単一モデルのレビュアーは、元の著者と盲点を共有する — より冷ややかで、異なるアーキテクチャのモデルがそれを捕まえる。疑い駆動はすでに自明でない意思決定に対してオプトインなので、そのスコープ内ではクロスモデルを提供することはスキルの価値の一部であり、不要な摩擦ではない。

**対話セッション: 常に提供せよ。黙ってスキップするな。**

**Step 1: ユーザーに尋ねる**

上記 Step 3 の単一モデルレビューの後、ただし RECONCILE の前に、いったん止めて尋ねる:

> *「単一モデルレビューが完了しました。クロスモデルのセカンドオピニオンが欲しいですか? 選択肢: Gemini CLI / Codex CLI / 手動の外部レビュー（あなたが別の場所に貼り付ける）/ スキップ。」*

この質問は、あらゆる対話的な doubt cycle で必須である — 低ステークスに感じる artifact でも。コストに見合うかを決めるのはエージェントではなくユーザーだ。エージェントの仕事は選択肢を表面化することである。

**Step 2: ユーザーが CLI を選んだら — 検証してから起動する**

1. ツールが PATH にあるか確認する（`which gemini`, `which codex`）。
2. フルプロンプトを渡す前に動作確認する（`gemini --version` など） — 古い／壊れたバイナリは `which` を通過しても実入力で失敗しうる。
3. 必要なフラグ・認証・環境変数（API キーなど）を含め、正確な呼び出しコマンドをユーザーと確認する。実装によって異なる。決して仮定するな。
4. ARTIFACT + CONTRACT + 敵対的プロンプト **のみ** を渡す。セッションコンテキストも CLAIM も渡さない。
5. シェルのエスケープに注意せよ。artifact にクォート・`$(...)`・バッククォートが含まれる場合、インラインの `-p "…"` より stdin（`echo … | gemini`）やヒアドキュメントを優先する。迷ったら、実行前にユーザーに呼び出しコマンドを確認してもらう。
6. 出力を Step 4（RECONCILE）に持っていく。

**artifact を絶対にシェルでクォートされた引数へ展開（interpolate）するな。** コード・markdown・レビュープロンプトには、バッククォート・`$(...)`・クォート文字が日常的に含まれ、これらはプロンプトを切り詰めるか、埋め込まれたシェルを実行してしまう。フルプロンプトをファイルに書き出し、stdin 経由でパイプせよ。

例の形（フラグはインストール済みツールに対して検証せよ — 構文は実装やバージョンによって異なる）:

```bash
# まず敵対的プロンプト + ARTIFACT + CONTRACT を一時ファイルに書き出す。
# その後 stdin 経由でパイプし、artifact 内のシェルメタ文字を不活性に保つ。

# Codex（read-only sandbox で CLI がワークスペースへ書き込むのを防ぐ）:
codex exec --sandbox read-only -C <repo-path> - < /tmp/doubt-prompt.md

# Gemini（'--approval-mode plan' は read-only。'-p ""' で非対話モードになり、
# プロンプトは stdin から読まれる）:
gemini --approval-mode plan -p "" < /tmp/doubt-prompt.md
```

read-only sandbox は要となる細部である: doubt artifact 自体に、クロスモデル CLI がワークスペースに対して実行してしまう命令（意図的または偶発的なプロンプトインジェクション）が含まれている可能性がある。

**Step 3: CLI が利用不可または失敗したら**

失敗を明示的に表面化する。選択肢を提示する: 手動で実行する / 別のツールを試す / スキップする。黙って単一モデルにフォールバックするな — クロスモデルが行われなかったことをユーザーは知るべきだ。

**Step 4: ユーザーがスキップしたら**

出力にスキップを明記し（*「単一モデルの指摘のみで進めます」*）、RECONCILE へ進む。スキップは構わない。黙ってスキップするのは構う。

**非対話コンテキスト**（CI, `/loop`, autonomous-loop, スケジュール実行）:

- クロスモデルは **スキップ** し、そのスキップを出力に **明記** しなければならない: *「クロスモデルはスキップしました: 非対話コンテキスト。」*
- **明示的なユーザー認可なしに外部 CLI を起動するな** — これは要となる安全性の性質である。

クロスモデルはコスト・レイテンシ・ツールの脆さを加える。エージェントは毎サイクル選択肢を表面化し、この artifact にそれが見合うかをユーザーが決める。

### Step 4: RECONCILE — 指摘を畳み込む

レビュアーの出力は判定ではなくデータである。**あなたは依然としてオーケストレーターだ。** 分類する前に、各指摘に対して artifact 本文を読み直せ — レビュアーをラバースタンプ（無批判承認）するのは、レビュアーを無視するのと同じ失敗モードである。

各指摘について、この **優先順位（precedence）** で分類する（最初に一致したクラスが勝つ）:

1. **Contract misread（契約の読み違い）** — レビュアーが指摘したのは、あなたが提供した CONTRACT が不明瞭または不完全だったからである。まず contract を直し、次のサイクルで分類し直す。
2. **Valid + actionable（有効かつ対処すべき）** — artifact への変更を要する本物の問題。変更してループし直す。
3. **Valid trade-off（有効なトレードオフ）** — 問題は本物だが、修正コストが受容コストを上回る。トレードオフをユーザーに見えるよう明示的に記録する。
4. **Noise（ノイズ）** — レビュアーが、彼の持たなかったコンテキストのもとでは実は正しいものを指摘した。記録して先へ進み、こう問う: そのコンテキストを contract に加えていれば、この誤検知は防げたか?

フレッシュなレビュアーは、コンテキストを欠くがゆえに間違えることがある。「フレッシュだから」というだけで従うな。

### Step 5: STOP — 再帰ではなく、境界のあるループ

以下のときに停止する:

- 次の反復が些末な、または既に検討済みの指摘しか返さない、**または**
- 3 サイクル完了した（ユーザーにエスカレーションせよ。1 人で 4 サイクル目を回すな）、**または**
- ユーザーが明示的に「ship it（出荷しろ）」と言った

3 サイクル後もレビュアーが実質的な問題を表面化し続けるなら、その artifact はまだ準備できていないのかもしれない。これをユーザーに表面化せよ — 解決されない 3 サイクルは、ループを続ける理由ではなく、artifact についての情報である。

artifact が大きいために 3 サイクルが「明らかに不十分」なら、それは artifact が大きすぎるのだ — Step 2 に戻って分解せよ。境界を引き上げるな。

## よくある自己正当化

| 自己正当化 | 現実 |
|---|---|
| 「自信があるから doubt ステップは飛ばす」 | 新規問題では自信と正しさの相関は低い。確信の瞬間こそ、盲点が潜む場所だ。 |
| 「レビュアーを起動するのは高くつく」 | 本番で誤ったコミットをデバッグするほうが高くつく。チェックには境界があるが、バグにはない。 |
| 「レビュアーはどうせ重箱の隅をつつくだけ」 | スコープを切っていない場合だけだ。プロンプトを「contract のもとで失敗させる問題」に制約せよ。 |
| 「doubt は最後に `/review` でやる」 | `/review` は最終ゲート。疑い駆動は、軌道修正が安い早期に誤った方向を捕まえる。PR の時点では遅すぎる。 |
| 「すべてのステップを疑ったら永遠に出荷できない」 | 本スキルは自明でない意思決定に適用され、すべてのキーストロークにではない。「使わない場合」を読み直せ。 |
| 「2 つの意見は常に 1 つよりよい」 | 2 つ目がコンテキストを欠きノイズを生むなら、そうではない。従う（defer）のではなく reconcile せよ。 |
| 「レビュアーが反対したから自分が間違っていた」 | レビュアーはあなたのコンテキストを欠く — 反対は判定ではなく情報だ。artifact を読み直し、分類してから決めよ。 |
| 「クロスモデルは常によりよい」 | クロスモデルは単一モデルが自分自身と共有する盲点を捕まえるが、コストとツールの脆さを加える。対話的な doubt cycle ごとに提供し、artifact にそれが見合うかはユーザーが決める。エージェントの仕事は選択肢を表面化することであって、ゲートをかけることではない。 |
| 「ユーザーが一度 yes と言ったから CLI を呼び続けてよい」 | 各呼び出しはそれ自体が認可である。artifact・プロンプト・フラグは呼び出しごとに変わる — 毎回、正確なコマンドをユーザーと再確認せよ。 |

## レッドフラグ

- 1 行のリネームやフォーマット変更のためにフレッシュコンテキストレビュアーを起動する
- artifact 本文を読み直さずにレビュアー出力を権威として扱う
- ユーザーにエスカレーションせず 3 サイクルを超えてループする
- レビュアーに「これは良いか?」と尋ねる（「問題を見つけよ」ではなく）
- 高ステークスの意思決定で、時間的プレッシャーのもとで doubt を飛ばす
- 変更されていない artifact に対してフレッシュコンテキストを再起動する（同じ指摘が返ってくる。あなたは足踏みしている）
- **Doubt theater（チェック可能なシグナル）**: レビュアーが実質的な指摘を表面化した 2 サイクル以上にわたって、actionable に分類された指摘がゼロ。あなたは疑っているのではなく検証している。止めてエスカレーションせよ。
- コミットしてからしか疑わない — それは疑い駆動開発ではなく `/review` だ
- ツールが存在し・設定され・その正確な構文を受け付けることをユーザーと確認せずに、外部 CLI 呼び出しをハードコードする
- **対話的な doubt cycle でクロスモデルを黙ってスキップする。** 推奨しない場合でも、提供は可視でなければならない。スキップは構わない。黙ってスキップするのは構う。
- 外部 CLI がエラーになるか見つからないとき、黙ってフォールバックする — 失敗を表面化し、ユーザーにリダイレクトさせよ
- レビュアーの入力から contract を剥がす
- CLAIM をレビュアーに渡す（同意へバイアスがかかる）

## 他スキルとの相互作用

- **`35-architect-code-review` / `/review`**: 補完関係。`/review` は事後の PR 判定。疑い駆動は進行中の意思決定単位。両方使え。
- **`35-architect-principle`**: 設計判断の **判断基準** を供給する。principle スキルで下した設計上の結論が自明でないとき、本スキルの敵対的レビューにかける。principle ＝ 判断基準、doubt ＝ その判断の反証。
- **リポジトリのオーケストレーションルール**（`references/orchestration-patterns.md`）: このスキルはメインセッションからオーケストレーションする。persona が別の persona を呼ぶのはアンチパターン B — 上記「ロードに関する制約」を参照。

## 検証

疑い駆動開発を適用した後:

- [ ] 自明でないすべての意思決定（上の定義による）が、確定する前に CLAIM として明示的に命名された
- [ ] 自明でない artifact ごとに少なくとも 1 回のフレッシュコンテキストレビューが行われた
- [ ] レビュアーが受け取ったのは ARTIFACT + CONTRACT — CLAIM ではなく、あなたの推論でもない
- [ ] レビュアーのプロンプトは敵対的（「問題を見つけよ」）だった。検証的（「良いか?」）ではない
- [ ] 指摘は artifact 本文と突き合わせて分類された（ラバースタンプではなく）。優先順位は contract misread / actionable / trade-off / noise
- [ ] 停止条件が満たされた（些末な指摘・3 サイクル・ユーザーの上書き）
- [ ] 対話モードでは、クロスモデルが（artifact のステークスに関わらず）ユーザーに **明示的に提供** され、その応答が出力に明記された
- [ ] 非対話モードでは、クロスモデルがスキップされ、そのスキップが明記された
- [ ] 外部 CLI の呼び出しは、PATH チェック・動作するバイナリのテスト・ユーザーとの構文確認・実行への明示的認可が先行した
