## 1. 判断の起点: 変更容易性

すべての原則は「変更が来たときの影響を局所化する」ための手段である。設計判断に迷ったら、原則名を思い出す前に次を問う:

- **どんな変更が実際に来そうか**（過去の変更履歴・ロードマップから推定する。想像ではなく根拠を持つ）
- その変更が来たとき、**何ファイル・何箇所を触ることになるか**
- 影響範囲を小さくする構造は何か

「来るかもしれない変更」に先回りするのは変更容易性の向上ではない。**実際に来る変更** の影響を小さくするのが向上である。この区別が KISS/YAGNI と拡張性のトレードオフを解く鍵になる。

## 2. シンプルさ

### KISS

| 観点 | 内容 |
|---|---|
| 定義 | 設計・実装をできるだけ単純に保つ |
| 適用する | 共通化・汎用化・パターン適用を検討する **すべての瞬間** |
| やりすぎのサイン | — |
| 判断の問い | この抽象が読み手に強いる「新しい概念」は、削減する重複より安いか |

KISS はやりすぎになりにくい唯一の原則に近いが、「単純にしようとして機能を削る」形で要件を侵すことはある。単純さは手段であって、要件充足が目的。

### YAGNI

| 観点 | 内容 |
|---|---|
| 定義 | 将来必要かもしれない機能を今は作らない |
| 適用する | 「将来のため」という理由が出た抽象・設定・拡張点 |
| やりすぎのサイン | 実際に繰り返し来ている変更にまで抽象化を拒否し、コピペ地獄になる |
| 判断の問い | この要素を要求しているのは今の仕様か、将来の予測か |

YAGNI の正しい適用は「予測駆動の抽象を拒否する」ことであって、「一切抽象化しない」ことではない。実在する変更軸には抽象を入れる。

## 3. 凝集と重複

### DRY の正しい理解

DRY は "Don't Repeat Yourself" だが、対象は **コードの見た目** ではなく **知識（意図・ルール・決定）** である。

- **正しい DRY**: 消費税率の計算ロジックが 3 箇所にコピーされている → 税率変更時に 3 箇所直す羽目になる → 1 箇所に集約する
- **誤った DRY（過剰な結合）**: 「請求書の合計計算」と「見積書の合計計算」が今はたまたま同じ式 → 共通関数化 → 後日、見積だけ端数処理が変わる → 共通関数に `if 見積` が生え始める → 2 つの無関係な知識が 1 関数に癒着

**判定**: 「この 2 つは、同じ 1 つの決定に由来するか？ それとも独立に変わりうる別の決定か？」独立に変わりうるなら、重複を許す方が疎結合で安い。

### 関心の分離（Separation of Concerns）

異なる関心（UI 表示 / 業務ルール / 永続化 / 外部通信）を別のモジュールに分ける。各関心が独立に変更できることが目的。

- 認知面: 1 モジュールの理解に必要な文脈を、ワーキングメモリに収まる量へチャンク化する
- 失敗のサイン: 1 つの関数の中で、SQL を組み立て・業務判定し・HTML を整形している

### 高凝集・疎結合

- **高凝集**: 1 つのモジュール内の要素が、同じ目的のために強く関連している
- **疎結合**: モジュール間の依存が少なく・細く・抽象を介している

SOLID はこの 2 つを実現する具体的手段の集合と見なせる。凝集は「何を 1 箇所に集めるか」、結合は「モジュール間をどう繋ぐか」。

## 4. SOLID 詳解

### SRP — 単一責任

- **定義**: モジュールが変更される理由は 1 つだけであるべき
- **判定軸**: 行数でもメソッド数でもなく「**変更を要求するアクター／理由の数**」
- **例（違反）**: `Report` クラスが「集計ロジック」と「PDF 整形」を両方持つ → 経理都合でも表示都合でも変わる
- **例（改善）**: 集計は `ReportCalculator`、整形は `ReportRenderer` に分離
- **やりすぎ**: 1 メソッド 1 クラスに割って、追うファイルが爆発する。責任 = 変更理由であって、操作単位ではない

### OCP — 開放/閉鎖

- **定義**: 拡張に開き、修正に閉じる
- **手段**: 分岐追加ではなく、抽象への実装追加（Strategy / Template Method / ポリモーフィズム）
- **例（違反）**: 決済手段が増えるたびに `switch(paymentType)` に `case` を足す
- **例（改善）**: `PaymentMethod` インターフェースを実装するクラスを追加するだけで済む
- **やりすぎ**: まだ 1 種類しかない軸に対して抽象を用意する（YAGNI 違反）。**2 回目の変更が来てから** 抽象化する（Rule of Three）

### LSP — リスコフの置換

- **定義**: 派生型は基底型と置換可能でなければならない
- **契約規則**: 事前条件を強めない・事後条件を弱めない・不変条件を保つ
- **例（違反）**: `Rectangle` を継承した `Square` が `setWidth` で height も変え、`Rectangle` 前提のコードを壊す
- **例（違反）**: `Bird.fly()` を `Penguin` が例外で潰す
- **改善方針**: 継承より委譲（composition over inheritance）。振る舞いが違うなら is-a ではない

### ISP — インターフェース分離

- **定義**: クライアントに、使わないメソッドへの依存を強いない
- **例（違反）**: `Worker` インターフェースに `work()` と `eat()` があり、`RobotWorker` が `eat()` を実装させられる
- **例（改善）**: `Workable` と `Eatable` に分割
- **やりすぎ**: 全メソッドを 1 つずつのインターフェースに割り、利用側が大量の型を組み合わせる羽目になる

### DIP — 依存性逆転

- **定義**: 上位モジュールも下位モジュールも、抽象に依存する。抽象は詳細に依存しない
- **例（違反）**: `OrderService`（ドメイン）が `PostgresClient`（詳細）を直接 new / import
- **例（改善）**: `OrderRepository` インターフェースにドメインが依存し、Postgres 実装を注入する
- **効果**: テストでモックに差し替え可能・DB 変更がドメインに波及しない
- **やりすぎ**: 差し替え・テストの現実的必要がないのに、全クラスにインターフェースを対で用意する。抽象は「実在する差し替え軸」が正当化する

## 5. 安全性

### Design by Contract（契約による設計）

- **事前条件（precondition）**: 呼び出し側が満たすべき条件。関数入口で検証し、破れば即拒否
- **事後条件（postcondition）**: 関数が保証する結果
- **不変条件（invariant）**: オブジェクトが常に満たす条件。生成時から破壊時まで維持
- **実践**: ドメインオブジェクトは「不正な状態では生成できない」コンストラクタにする（常に有効な状態のみ存在）

### Fail Fast（フェイルファスト）

- **定義**: 異常を検出したら即座に失敗させ、障害の潜伏・拡大を防ぐ
- **アンチパターン**: `try { ... } catch { return null }` で握りつぶし、呼び出し側が `null` を知らずに使い続ける → 遠く離れた場所で NPE → デバッグ困難
- **正しい形**: 想定外はその場で例外にし、スタックトレースが原因地点を指すようにする
- **レビュー観点**: 「この構造はバグを早期に顕在化させるか、静かに飲み込むか」

## 6. Rule of Three と抽象化のタイミング

KISS/YAGNI（今は作るな）と OCP/DIP（拡張に開け）の衝突を実務で解く経験則:

- **1 回目**: そのまま書く。抽象化しない
- **2 回目**: 重複に気づく。まだコピーで様子を見てよい（2 つの差異が本質か偶然か見極める）
- **3 回目**: パターンが確定。ここで初めて抽象化する

「3 回目で抽象化」は絶対ではないが、「予測で 0 回目に抽象化する」よりはるかに外れにくい。抽象は情報が増えてから入れる方が正確になる。

## 7. アンチパターン早見表

| アンチパターン | 違反する原則 | 症状 | 対処 |
|---|---|---|---|
| 投機的汎用化 | YAGNI | 使われない拡張点・設定 | 実際の要件が来るまで作らない |
| 誤った DRY 化 | DRY の誤用 | 共通関数に `if 呼び出し元` が増殖 | 独立変更軸なら重複に戻す |
| 神クラス | SRP | 1 クラスが何でも知っている | 変更理由ごとに分割 |
| switch 地獄 | OCP | 種類が増えるたび分岐追加 | 抽象への実装追加に置換 |
| 壊れた継承 | LSP | 子が親の契約を破る | 委譲に切り替える |
| 太いインターフェース | ISP | 使わないメソッドの空実装 | 関心ごとに分割 |
| 詳細への直接依存 | DIP | ドメインが DB/HTTP を直 import | 抽象を挟んで注入 |
| ラザニアコード | KISS / 認知負荷 | 薄い機能に過剰なレイヤー | 層を畳んで単純化 |
| 例外握りつぶし | Fail Fast | `catch { return null }` | 想定外は即例外に |
| 原則の目的化 | 全体 | 原則を守るための複雑化 | 変更容易性で再評価 |
