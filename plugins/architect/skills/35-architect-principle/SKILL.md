# 設計原則（Design Principles）

## 概要

設計原則とは、**変更に強く・理解しやすく・壊れにくいソフトウェアを作るための判断基準** の集合である。目的は保守性・拡張性・安全性を高め、将来の変更コストを下げることにある。

原則は「守れば必ず正解になるルール」ではなく、**トレードオフの指針** である。プロダクトの目的・組織・期間によって適用の度合いは変わる。したがって本スキルは、原則をチェックリストとして機械的に適用するためのものではなく、**設計上の対話の軸** として使うためのものである。

### 最重要原則: よい設計の中心は「変更容易性」

**よい設計のもっとも重要な品質は変更容易性（発展性）である。** その中心にあるのがモジュール化 — 変更を安全かつ局所的に行える構造をつくること。

以下の全ての原則は、突き詰めれば「変更が来たときに、影響範囲を小さく・安全に・理解可能に保つ」ための道具にすぎない。原則名を暗記することに意味はない。**目の前の設計判断が変更容易性を上げているか下げているか** だけを問え。

## いつ使うか

以下の場面でこのスキルを適用する:

- 新しいクラス・モジュール・レイヤー・境界を設計しようとしているとき
- 既存の構造をリファクタしようとしているとき
- 抽象（インターフェース・基底クラス・共通化）を導入するか迷っているとき
- ロジックを共通化すべきか、あえて重複を許すか判断するとき
- 設計案・PR に対して「この構造は将来の変更に耐えるか」をレビューするとき
- 「なんとなく複雑になってきた」「どこに置けばいいか分からない」と感じたとき

**使わない場合:**

- 明快で曖昧さのない 1 行の変更・機械的な操作（リネーム、フォーマット）
- 設計の余地がない、既存パターンへの単純な追従
- 原則を持ち出すより先に、まず動くものを作るべき探索段階（ただし「シンプルに保つ」判断自体は KISS/YAGNI として本スキルの範囲）

原則を過剰適用すれば、単純な機能に無用な抽象とレイヤーを積み上げてしまう。**原則同士もトレードオフを持つ** ことを常に忘れないこと。

## 原則を適用するプロセス

設計判断を下すとき、このチェックリストで考える:

```
Design decision:
- [ ] 1. 変更容易性 — この判断は「将来ありうる変更」の影響範囲を小さくするか
- [ ] 2. シンプルさ — KISS / YAGNI に照らして、過剰ではないか
- [ ] 3. 重複と凝集 — DRY と関心の分離に照らして、知識は 1 箇所に集まっているか
- [ ] 4. 結合と拡張 — SOLID に照らして、疎結合で拡張に開かれているか
- [ ] 5. 安全性 — 契約とフェイルファストに照らして、壊れたとき早く気づけるか
- [ ] 6. トレードオフ — 上の原則同士が衝突していないか。事業・期間に照らしてどちらを取るか
```

**順序が大事**: まず 1（変更容易性）を North Star に置く。次に 2（シンプルさ）で「そもそも作るべきか / 抽象化すべきか」をゲートする。原則を適用して複雑になるなら、たいてい 2 に負ける。

## シンプルさ — 最初にかけるゲート

### KISS（Keep It Simple, Stupid）— できるだけシンプルに

設計・実装はできるだけシンプルに保つ。過度な抽象化・汎用化を避け、理解・レビュー・変更を容易にする。

**問い**: 「この抽象化は本当に、いま価値があるか？」「もっと単純な形で同じ要件を満たせないか？」

共通化・汎用化したくなった瞬間こそ立ち止まる。抽象は「読み手が保持すべき概念」を増やす。その概念が、削減する重複より高くつくなら、抽象は損である。

### YAGNI（You Aren't Gonna Need It）— 今いらないものは作らない

「将来必要になるかもしれない」機能は、今は実装しない。予測ベースの実装は外れることが多く、複雑性だけを残す。

**問い**: 「これは今の要件が要求しているか、それとも将来の予測に基づくだけか？」

- 「たぶん将来こういう拡張が必要」という理由だけの抽象インターフェース・プラグイン機構・設定項目を作る前に止まる
- MVP フェーズでは、拡張用の仕組みを後回しにする判断基準にする

KISS と YAGNI は「拡張性」と衝突する。この衝突は本スキルの後半「トレードオフ」で扱う。

## 再利用性・凝集 — 知識を 1 箇所に

### DRY（Don't Repeat Yourself）— 同じ知識を繰り返さない

同じ **知識・ロジック** を複数箇所に重複させない。重複は変更漏れ・不整合・バグの温床になる。単一の「ソース・オブ・トゥルース」に集約する。

**重要な誤解の訂正**: DRY は「同じ見た目のコードを消す」原則ではなく、「同じ **知識** を 1 箇所に」する原則である。たまたまコードが似ているだけの、由来の異なる 2 つのロジックを共通化すると、片方の変更がもう片方を壊す **誤った結合（不当な DRY 化）** を生む。「今は同じだが独立に変わりうるか？」を問い、Yes なら重複を許す。

### モジュール化・関心の分離（Separation of Concerns）

複雑なソフトウェアを、分かりやすく扱いやすいモジュールへ分解する。UI・ドメイン・インフラなど **関心ごと** を分け、それぞれが独立に変更できるようにする。

これは認知の観点でも重要だ — システム全体を一度に頭に入れなくても、**部分的にワーキングメモリへロード可能な構造** をつくる（チャンク化）。1 つのモジュールを理解するのに他の 10 個を同時に頭に入れねばならないなら、分割に失敗している。

**問い**: 「このモジュールが変わる理由は 1 つか？」「別々の理由で変わるものが 1 箇所に混ざっていないか？」

## 拡張性・柔軟性 — SOLID

オブジェクト指向設計の 5 原則。いずれも「変更が来たとき影響を局所化する」ための道具である。

### SRP — 単一責任の原則（Single Responsibility Principle）

クラス・モジュールは **単一の責任** を持つ。言い換えると「変更理由を 1 つに絞る」。影響範囲を局所化し、理解しやすくする。

- 判定基準は「行数」ではなく「変更理由の数」。1 つのクラスが、UI の都合でも DB の都合でも業務ルールの都合でも変わるなら、責任が混ざっている。

### OCP — 開放/閉鎖原則（Open-Closed Principle）

拡張には開かれ、修正には閉じている。既存コードを **修正せずに** 新機能を追加できる構造を目指し、回帰バグを減らす。

- 典型手段は、条件分岐の追加ではなく、抽象への実装追加（ポリモーフィズム / ストラテジ）。
- ただし OCP は YAGNI と衝突する。「まだ来ていない拡張軸」に対して先回りで開くのは過剰設計。**実際に 2 回目の変更が来た時点** で抽象を入れるのが穏当（Rule of Three）。

### LSP — リスコフの置換原則（Liskov Substitution Principle）

親クラスのインスタンスは、子クラスで置き換えても振る舞いが破綻してはならない。継承した型を差し替えても契約（事前条件を強めない・事後条件を弱めない）が保たれること。

- 「is-a に見えるが振る舞いが違う」継承は LSP 違反のサイン。継承より委譲（composition over inheritance）を検討する。

### ISP — インターフェース分離の原則（Interface Segregation Principle）

クライアントは、自分が使わないメソッドに依存させられてはならない。肥大化したインターフェースを、利用側の関心ごとに小さく分ける。

- 利用側が「実装しないといけないが使わないメソッド」を持つなら、インターフェースが太すぎる。

### DIP — 依存性逆転の原則（Dependency Inversion Principle）

上位（ビジネスロジック）は下位（具体実装）に依存せず、**両者が抽象に依存** する。実装詳細と高レベルロジックを疎結合にし、差し替えとテストを容易にする。

- 「ドメイン層が DB ライブラリを直接 import している」のは典型的な DIP 違反。抽象（リポジトリインターフェース）を挟む。
- これも YAGNI と衝突しうる。差し替え・テストの必要が実在しないのに全てを抽象化するのは過剰。**テスト容易性・差し替えの現実的必要** が正当化する。

## 安全性・安定性 — 壊れたら早く気づく

### デザイン・バイ・コントラクト（Design by Contract）

事前条件・事後条件・不変条件を明示し、コードが満たすべき **契約** として扱う。契約を破る入力・状態は、その場で拒否する。

- 公開関数の入口で引数を検証（事前条件）、ドメインオブジェクトは常に不変条件を満たす状態でのみ存在させる。

### フェイルファスト（Fail Fast）

異常状態を早期に検出して即座に失敗させ、障害の拡大・潜伏を防ぐ。握りつぶして「なんとなく動く」状態を作らない。

- `null` やデフォルト値で誤魔化して処理を続けるより、想定外はその場で例外にする方が、後のデバッグコストを下げる。
- これらは設計だけでなく **レビューの観点** にもなる: 「この構造はバグを早期に顕在化させるか、それとも静かに握りつぶすか」。

## トレードオフ — 原則同士は衝突する

原則は万能ルールではない。以下は代表的な衝突であり、**事業・期間・チームに照らして** 意図的に選ぶ。

| 衝突 | 一方 | 他方 | 判断の目安 |
|---|---|---|---|
| シンプルさ vs 拡張性 | KISS / YAGNI | OCP / DIP | 変更軸が実在し繰り返し来るなら抽象化。予測だけなら今はシンプルに。 |
| DRY vs 疎結合 | DRY（共通化） | 独立変更可能性 | 「今は同じだが独立に変わりうる」なら重複を許す。誤った DRY 化は結合を生む。 |
| 抽象化 vs 認知負荷 | DIP / 多層化 | 全体把握のしやすさ | レイヤーは変更容易性を上げるが、追う対象を増やす。薄い機能に厚い層を積まない。 |
| 完全性 vs 出荷速度 | 全原則の適用 | MVP の速度 | 探索段階・短命コードでは原則適用を緩める。長命・多人数・高ステークスでは締める。 |

**判断の North Star は常に変更容易性**。「原則を守ること」ではなく「この選択が、実際に来る変更に対して影響範囲を小さく保つか」で決める。予測が外れる変更軸に先回りするのは、変更容易性を上げていない — 複雑性を上げているだけだ。

### 認知スタイルへの配慮

「関心の分離」「単一責任」「レイヤードアーキテクチャ」は、**問題を分割して段階的に理解する** 認知スタイルを前提としている。全体を一度に把握したい認知スタイルとは衝突しうる。設計原則は「認知負荷のかけ方」にも関わるため、チームの思考スタイルに応じて分割の粒度を調整する。過剰な分割は、ある人には整理でも、別の人には「あちこち飛ばないと読めない」負債になる。

## 設計原則チェックリスト（レビュー観点）

詳細な適用ガイドとアンチパターン集は `references/principles-catalog.md` を参照。

**変更容易性（最優先）**
- [ ] この設計は「実際に来そうな変更」の影響範囲を小さくしているか
- [ ] 1 つのモジュールを理解するのに、他の多数を同時に頭へ入れる必要がないか

**シンプルさ**
- [ ] KISS: もっと単純な形で同じ要件を満たせないか。抽象は削減する複雑さに見合うか
- [ ] YAGNI: これは今の要件が要求するものか、将来の予測に過ぎないか

**凝集・重複**
- [ ] DRY: 同じ **知識** が複数箇所に散っていないか（見た目の重複ではなく知識の重複か）
- [ ] 関心の分離: 別々の理由で変わるものが 1 箇所に混ざっていないか

**結合・拡張**
- [ ] SRP: 各クラス／モジュールの変更理由は 1 つに絞れているか
- [ ] OCP: 拡張が「既存コードの修正」ではなく「追加」で済む構造か（過剰でない範囲で）
- [ ] LSP: 継承した型は、親の契約を破らず置換可能か
- [ ] ISP: 利用側が使わないメソッドへの依存を強いられていないか
- [ ] DIP: 上位ロジックが具体実装を直接 import していないか（抽象に依存しているか）

**安全性**
- [ ] 契約（事前・事後・不変条件）が明示され、破る入力はその場で拒否されるか
- [ ] フェイルファスト: 想定外を握りつぶさず、早期に失敗して顕在化させるか

**トレードオフ**
- [ ] 原則同士の衝突を認識し、事業・期間に照らして意図的に選んだか
- [ ] 適用した抽象・分割が、変更容易性を実際に上げているか（複雑さを増やしただけでないか）

## よくある言い訳（Common Rationalizations）

| 言い訳 | 現実 |
|---|---|
| 「将来使いそうだから抽象化しておく」 | YAGNI 違反。予測は外れる。抽象は 2 回目の変更が実際に来た時点で入れる。 |
| 「コードが同じだから共通化した」 | 見た目の重複と知識の重複は別。独立に変わりうるなら共通化は誤った結合を生む。 |
| 「レイヤーを増やせば綺麗になる」 | 薄い機能に厚い層を積むと、変更容易性は上がらず認知負荷だけ増える。 |
| 「SOLID を全部守ったから良い設計だ」 | 原則は目的ではなく手段。North Star は変更容易性。守った結果複雑化したなら本末転倒。 |
| 「とりあえず動くから設計は後でいい」 | 長命・多人数コードでは後回しにした設計負債が 10 倍で返る。ただし短命コードなら妥当。 |
| 「原則同士が矛盾するから無視する」 | 矛盾は無視の理由ではなく、意図的に選ぶべき分岐点。トレードオフ表で選べ。 |
| 「例外を握りつぶしておけば落ちない」 | 落ちないだけで壊れている。フェイルファストで早く気づく方が安い。 |

## 危険信号（Red Flags）

- 要件が要求していない拡張点（抽象インターフェース・設定・プラグイン）を「将来のため」に追加している
- 由来の異なる 2 つのロジックを、見た目が似ているだけで共通化しようとしている
- 1 つのクラスが、UI・DB・業務ルールの複数の理由で変更される
- ドメイン層が具体的なフレームワーク／ライブラリを直接 import している
- 「is-a」に見える継承だが、子が親の振る舞い契約を破っている
- 薄い機能のために、追うべきレイヤー・ファイルが不釣り合いに多い
- 想定外の入力・状態を `null` やデフォルト値で握りつぶし、処理を続けている
- 「原則を守ること」自体が目的化し、変更容易性が上がっているか検証していない
- 原則同士の衝突を認識せず、どちらか一方を無自覚に優先している

## 他スキルとの相互作用

- **`35-architect-code-review` / `/review`**: 本スキルはコードレビューの「アーキテクチャ／可読性」軸の判断基準を供給する。レビューで構造の是非を問うとき本スキルを使う。
- **`35-architect-db-design`**: DB 設計にも同じ思想（変更容易性・正規化=DRY・制約=契約）が流れる。データモデルは本スキルより長命なので、より保守的に原則を適用する。
- **`35-architect-doubt-driven-development`**: 設計判断が自明でないとき、本スキルで下した結論を敵対的レビューにかける。本スキル＝判断基準、doubt＝その判断の反証。

## 検証（Verification）

設計原則を適用した後:

- [ ] 下した設計判断ごとに、それが「変更容易性」をどう上げるかを一言で説明できる
- [ ] 導入した抽象・分割それぞれについて、KISS/YAGNI に照らして正当化できる（削減した複雑さ ≥ 増やした概念）
- [ ] 共通化した箇所が「同じ知識」であり、たまたま似た別知識ではないことを確認した
- [ ] 各モジュール／クラスの「変更理由」を 1 つに言語化できる（SRP）
- [ ] ドメインロジックが具体実装ではなく抽象に依存している（DIP）ことを確認した（過剰でない範囲で）
- [ ] 想定外の入力・状態が握りつぶされず、早期に顕在化する構造になっている
- [ ] 原則同士の衝突があった箇所で、事業・期間に照らしてどちらを取ったかを明示した
