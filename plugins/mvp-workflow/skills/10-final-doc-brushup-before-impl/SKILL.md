# MVP 意思決定スキル

`docs/` ディレクトリ内のアイデア・MVP 定義を読み込み、対象プラットフォームのリファレンス技術スタックを踏まえた上で、ユーザーへの質問を通じて技術的意思決定や仕様の確定を漏れなく行い、`docs/` を更新する。

このスキルは **Web 版** と **モバイル版** の両方に対応している。

| プラットフォーム | リファレンスファイル | 元リポジトリ |
|----------------|-------------------|------------|
| Web（フルスタック TypeScript） | `references/web-tech-stack.md` | [SugiKent/full-ts-template](https://github.com/SugiKent/full-ts-template) |
| モバイル（Expo + React Native） | `references/mobile-tech-stack.md` | [SugiKent/my-wish-list](https://github.com/SugiKent/my-wish-list) の `apps/mobile/` |

> いずれもバージョン固定はしていない。GitHub への直接アクセスは不要（参照ファイルだけを読めばよい）。両方のプラットフォームを含むハイブリッドなプロジェクト（Web 管理画面 + モバイルアプリなど）の場合は両方のリファレンスを読み込む。

このスキルはある程度の対話を用いてかなり MVP などのサービス構造が確定された時に実行される。しかしその時点ではコア技術以外は確定していなかったり、余分な情報が含まれている可能性がある。それを最後のブラッシュアップを行うのがこのスキル。

## 目的

個人開発キックオフ時の「なんとなくで後回しにしがちな技術的意思決定」を、実装前に体系的に整理する。

- **実装はしない** — 意思決定を `docs/` に書き込むことに集中する
- **一問一答** — ユーザーが考えやすいよう、一度に問う質問を絞る
- **リファレンスと相違がある箇所も確認の質問を必ずする** — docs の記載とリファレンスのデフォルトが異なる場合は必ず確認する

---

## 実行フロー

### Phase 1: 現状把握（コンテキスト収集）

#### Step 1-1: プロジェクトの docs を読み込む

`docs/` ディレクトリ内のファイルはすべて読む。さらに ROOT の `README.md` も読む。

#### Step 1-2: 対象プラットフォームを判定する

以下の手順で、このプロジェクトが **Web** / **モバイル** / **両方** のどれを対象にしているかを判定する。

1. プロジェクトのファイル構成を確認する
   - `apps/mobile/` または `app.config.js` / `app.json`（Expo 設定）の存在 → **モバイル要素あり**
   - `apps/client/` + `apps/server/` または `prisma/schema.prisma` の存在 → **Web 要素あり**
2. docs / README に「モバイルアプリ」「iOS」「Android」「Expo」「React Native」などの語があるか確認
3. docs / README に「Web」「管理画面」「Fastify」「React + Vite」などの語があるか確認

判定結果に応じて次のリファレンスを読み込む:

| 判定 | 読み込むリファレンス |
|-----|------------------|
| Web のみ | `references/web-tech-stack.md` |
| モバイルのみ | `references/mobile-tech-stack.md` |
| 両方（ハイブリッド） | 両方 |
| 不明 | ユーザーに 1 問だけ確認（「このプロジェクトは Web / モバイル / 両方のどれが対象ですか？」） |

各リファレンスの **最後にある「このリファレンスとの差分が出やすいポイント」セクション** を必ず頭に入れておく — そこに挙げられた項目で docs の記述と食い違いがあれば、Phase 2 で必ず確認の質問をする。

#### Step 1-3: カテゴリ別の未決定事項を整理

`references/decision-categories.md` を読み込み、各カテゴリの質問リストを取得する。decision-categories.md ではカテゴリごとに「共通 / Web 専用 / モバイル専用」のタグが付いている。Step 1-2 で判定したプラットフォームに該当するカテゴリだけを採用する。

Step 1-1 で読んだ docs と照合して、以下に分類する。

| ステータス | 条件 |
|-----------|------|
| ✅ 決定済み | docs にすでに明記されていて、リファレンスと相違がない箇所 |
| ❓ 未決定 | `{TODO}` プレースホルダー / 記述なし / 曖昧な記述 |
| ⏭ 後回し確認 | docs に「後回し」と明記されているが念のため確認 |

ユーザーに現状サマリーを見せる:

```
## 現状サマリー

対象プラットフォーム: [Web / モバイル / 両方]
読み込んだリファレンス: [web-tech-stack.md / mobile-tech-stack.md / 両方]

✅ 決定済み:
- サービス名: [抽出した内容]
- ...（ある場合のみ表示）

❓ 未決定（今回確認したい事項）:
- カテゴリ 1: プロジェクト基本情報 → N 件
- カテゴリ 2: 認証・認可 → N 件
- ...
```

サマリーを表示したら、即座に Phase 2 の質問を開始する。

---

### Phase 2: 意思決定セッション（カテゴリ別質問）

**重要**: 一度に大量の質問を投げない。1カテゴリずつ進め、ユーザーの回答を受けてから次へ。

各カテゴリの進め方:

1. カテゴリ名と「P0（必須）」の質問のみを最初に提示する
2. ユーザーの回答を受けた後、「P1（重要）」の質問に進む
3. P2（任意）の質問は「追加でお聞きしてもよいですか？」と確認してから進む
4. ユーザーが「スキップ」「後で」「わからない」と答えた場合は、その項目を「保留」としてマークし次へ

各カテゴリの質問内容は `references/decision-categories.md` を参照し、P0 → P1 → P2 の優先度順に進める。リファレンスとのデフォルト差分確認には対象プラットフォームの `web-tech-stack.md` / `mobile-tech-stack.md` を併用する。

---

### Phase 3: 意思決定のドキュメント更新

全カテゴリの質問が終わったら（または途中でユーザーが「完了」と言ったら）、収集した回答を docs/ に書き込む。問題が 20 問以上続いた場合は一旦質問を中断して docs を書き換える。

#### Step 3-1: `docs/PROJECT.md` のプレースホルダーを埋める

`docs/PROJECT.md` が存在し、`{TODO:` プレースホルダーがある場合は、Phase 2 で収集した情報で埋める。

埋め方の原則:
- `{TODO: サービス名}` → 決定したサービス名に置換
- `{TODO: 目的の説明}` → 決定した内容に置換
- まだ未定の場合は `{PENDING: 未定 - [理由]}` に変更してスキップ

#### Step 3-2: 結果を報告する

```
## 意思決定セッション完了

### 更新したドキュメント
- docs/PROJECT.md: N 箇所のプレースホルダーを更新しました

### 決定済み事項のサマリー
[カテゴリ別サマリー]

### 保留事項（N 件）
[保留にした項目のリスト]

### 次のアクション
1. 保留事項を決定後、このスキルを再実行して docs を更新してください
2. データモデルが固まっていればスキーマに反映してください（Web の場合: prisma/schema.prisma）
3. 環境変数 / プラットフォーム設定を確認してください（Web: .env.example、モバイル: app.config.js / .env.example）
```

---

## 注意事項

- **質問をまとめて投げない**: ユーザーが回答しやすいように、一度に問う質問は 3〜4 件以内に絞る
- **「わからない」を受け入れる**: 個人開発初期は「まだわからない」が正常。保留として記録し先へ進む
- **実装の提案はしない**: このスキルの役割は意思決定の記録のみ。実装方法の議論が始まったら「実装は後で行います。今は決定事項をまとめています」と伝える
- **プラットフォームに該当しないカテゴリは飛ばす**: モバイル単体のプロジェクトに Web 管理画面の質問を投げない、など対象外の質問でユーザーを混乱させない
