# MVP 実装順序プランニングスキル

MVP の仕様が固まった後、プロジェクト全体を一気に実装するのではなく、 **正しい順序** で段階的に実装するためのタスクリストを作る。

## なぜ「順序」が重要か

プロジェクトの構想が出来上がると、ついつい一気に実装してしまいがちだが、そうすると以下の問題が発生する。

- 自分（および AI）がコード全体を把握できなくなる
- 動作確認のフィードバックループが長くなり、デバッグコストが跳ね上がる
- 認証・データ層・UI が同時に未完成だと、どの層に問題があるか切り分けられない

このため、実装は必ず **下から積み上げる** 順序で行う。

## 実装順序の原則

抽象化すると **基盤 → 実装のための仕組み → ユーザー体験順** という順序が好ましい。具体的には以下の 4 ステップ。

1. **基盤**: `package.json` / lint / `tsconfig` / Turborepo（モノレポなら）など、必須ライブラリの初期化。client/server で簡単な hello world が動く程度の `dev` / `build` を通すところまで
2. **仕組み**: 必要に応じて seed / testData の投入の仕組み（DB がある場合）、 **動作確認用 CLI (`pnpm cli help`) の導入** （[[85-create-cli]] 参照、必須）
3. **認証周り**: 認証・認可の実装（必要な場合のみ）
4. **ユーザー体験順**: 実際にユーザーが体験する順序で画面・機能を実装していく

しかしコア機能になるべく早くたどり着く順序にする。コア機能が依存していない機能は例えユーザーの体験順でも後回しで良い

## 実行フロー

### Step 1: コンテキスト収集

- `docs/` 配下のドキュメント（特に `PROJECT.md` や MVP 定義） をすべて読む
- ROOT の `README.md` も読む
- 既に実装済みのファイル（`package.json` / `apps/` / `packages/` など）があれば軽くスキャンし、どこまで進んでいるかを把握する

### Step 2: ユーザー体験フローの確認

`AskUserQuestion` を用いて、 **MVP におけるユーザー体験の主要フロー** を 1〜3 個に絞り込む。すでに docs に書かれていれば確認のみ。

例:
- ログイン → ダッシュボード → 新規作成 → 一覧
- 公開ページ閲覧 → サインアップ → 投稿

### Step 3: 実装順序の起案

上記 4 ステップ（基盤 → 仕組み → 認証 → ユーザー体験順）に沿って、タスクを並べる。

**重要**: 詳細までは書かない。各タスクは 1〜3 行程度の見出しレベルで十分。詳細実装 spec は後続の OpenSpec 系スキルで設計するため、ここで書きすぎると二重管理になる。

### Step 4: `docs/MVP_IMPL_TASKS.md` への書き出し

以下のフォーマットで `docs/MVP_IMPL_TASKS.md` を作成（既存があれば更新）する。

```markdown
# MVP 実装タスク（順序付き）

> このドキュメントは「実装の順序」を定義する。各タスクの詳細実装 spec は OpenSpec で別途設計する。
> 上から順番に着手し、前段が完了してから次へ進むこと。

## 1. 基盤セットアップ

- [ ] package.json / Turborepo（必要なら） の初期化
- [ ] tsconfig / lint / formatter のセットアップ
- [ ] client / server の hello world で dev / build が通る状態にする
- [ ] CI（最低限の typecheck / lint） を通す

## 2. 実装のための仕組み

- [ ] DB スキーマ初期定義（Prisma 等）
- [ ] seed / testData 投入スクリプト
- [ ] 動作確認用 CLI の導入（`pnpm cli help` で起動。データ作成・worker キック用。詳細は 85-create-cli を参照）
- [ ] （必要なら）共通 UI コンポーネント / レイアウト基盤

## 3. 認証・認可

- [ ] 認証プロバイダの組み込み
- [ ] 認可境界（middleware / API guard）

（認証が不要な MVP の場合はこのセクションごとスキップ）

## 4. ユーザー体験順の実装

ユーザーが触る順番で並べる。各項目は OpenSpec で別途詳細 spec を作る単位。

- [ ] [体験 1] 例: トップページ閲覧
- [ ] [体験 2] 例: サインアップ
- [ ] [体験 3] 例: 新規作成
- [ ] [体験 4] 例: 一覧表示
```

### Step 5: 報告

ユーザーに以下を伝える。

- `docs/MVP_IMPL_TASKS.md` を作成/更新した旨
- 並べたタスクの section 構造（章タイトルのみ）
- 次のアクション: 「1 つ目のタスクから OpenSpec で詳細 spec を作る」

## 注意事項

- **詳細を書きすぎない**: 詳細は OpenSpec の責務。ここでは順序と粒度のみ
- **章を勝手に増やさない**: 「基盤 → 仕組み → 認証 → 体験」の 4 章構造を崩さない。認証不要なら章ごと省略
- **既存ファイルは尊重**: すでに `docs/MVP_IMPL_TASKS.md` がある場合は上書きせず、追記/更新の差分を確認する
- **実装はしない**: このスキルの役割は順序の計画のみ
