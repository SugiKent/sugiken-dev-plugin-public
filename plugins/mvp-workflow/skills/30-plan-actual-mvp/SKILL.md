# 何を作るべきかを考える

1. docs 内のドキュメントを読み、まず真っ先に検証するべき点を見つけます
2. 検証のスコープについて AskUserQuestion を用いて認識齟齬がないかを確認します
3. 検証内容と実現方法を

## 基準

- 技術的に複雑性の高い箇所(枯れていないライブラリ, AIなど不確定要素の多いなどの箇所)
- 良い体験を得られるかをまずは作って触るべき箇所

## 実装プランニング方法

技術的に複雑な点もミニマムなプロジェクト構成で実現します。例えば Mastra を使った Express プロジェクトであっても検証するべき点が AI の振る舞いなのであれば Express は無視できる可能性があります。また認証やセキュリティなども基本的に無視して良いはずです。

## actual mvp の実装場所

actual-mvp ディレクトリを利用します。
プランニング結果は actual-mvp/{スコープ名}/PLAN.md　に作成します。
